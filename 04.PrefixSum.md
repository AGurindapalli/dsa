# Prefix Sum Techniques

> **Tier 0 — Foundations** | Prerequisites: Time & Space Complexity | Required by: Sliding Window (Tier 1), Binary Search on Answer (Tier 1), Difference Array (Tier 6)

---

## 1. Intuition First (No Jargon)

**The core idea:** Precompute cumulative sums so that any range sum becomes a single subtraction: `sum(l, r) = prefix[r+1] - prefix[l]`. You're trading O(n) preprocessing space for O(1) range queries.

**Real-world analogy:** You're monitoring cumulative revenue on a Grafana dashboard. The Y-axis shows total revenue from day 1. If you want revenue for Q3 (July 1 to Sept 30), you don't re-sum every daily transaction. You subtract: `cumulative[Sept 30] - cumulative[June 30]`. That's a prefix sum. The dashboard already stores the running total — every range query is instant.

**WHY this pattern exists:** Many problems ask for sums (or counts, or XORs) over subarrays. Naive approach: recompute from scratch each time → O(n) per query, O(n×q) for q queries. Prefix sum: O(n) preprocessing, then O(1) per query. For problems with many range queries or when you need to check every subarray sum, this transforms O(n²) into O(n).

**The mental shift:** Stop thinking about individual subarrays. Think about the *cumulative function*. A subarray sum is just the difference between two points on the cumulative curve.

## 2. Pattern Recognition

### When to use

| Signal | Example |
|--------|---------|
| "Sum of subarray from i to j" | Range sum query |
| "Count of subarrays with sum = k" | Subarray Sum Equals K (LC 560) |
| "Is there a subarray with sum = 0?" | Check if any prefix sum repeats |
| "Sum in a 2D region" | 2D prefix sum for matrix region queries |
| "Number of elements ≤ x in range [l, r]" | Prefix count (variant of prefix sum) |
| Multiple range queries on static array | Preprocessing pays off when q is large |

### Common keywords in problem statements

- "subarray sum", "range sum", "cumulative"
- "sum equals k", "sum divisible by k"
- "number of subarrays where..."
- "query", "range", "between indices"
- "contiguous", "prefix", "running total"

### 10-Second Identification Rule

> **Are you being asked about sums (or counts, or XORs) over contiguous subarrays, and either:**
> 1. **Multiple range queries on the same array?** → Prefix sum for O(1) per query.
> 2. **Counting/finding subarrays with a specific sum property?** → Prefix sum + hash map.
>
> If the array is being modified between queries → Prefix sum won't work (use Fenwick/Segment Tree, Tier 6).
> If you need max/min subarray (not sum) → Different technique (Kadane's, sliding window, monotonic structures).

### Prefix Sum vs Sliding Window

| | Prefix Sum | Sliding Window |
|---|---|---|
| **Works with negatives** | Yes | No (for sum-based problems) |
| **Requires contiguity** | Yes | Yes |
| **Query pattern** | Any range sum in O(1) | Optimal contiguous subarray |
| **Space** | O(n) | O(1) or O(k) |
| **Use when** | Need range sums or subarray sum counting with negatives | Optimizing over contiguous window without negatives |

## 3. Core Algorithm / Template

### Template A: Basic 1D Prefix Sum

```csharp
// Build prefix sum array
// prefix[i] = sum of nums[0..i-1]
// prefix[0] = 0 (identity — empty prefix)
// Range sum [l, r] = prefix[r+1] - prefix[l]
public int[] BuildPrefixSum(int[] nums)
{
    int n = nums.Length;
    var prefix = new int[n + 1];  // n+1 to handle prefix[0] = 0

    for (int i = 0; i < n; i++)
        prefix[i + 1] = prefix[i] + nums[i];

    return prefix;
}

public int RangeSum(int[] prefix, int left, int right)
{
    return prefix[right + 1] - prefix[left];
}

// Example: nums = [1, 2, 3, 4, 5]
// prefix =        [0, 1, 3, 6, 10, 15]
// RangeSum(1, 3) = prefix[4] - prefix[1] = 10 - 1 = 9  (2+3+4)
```

**Why size n+1?** The `prefix[0] = 0` sentinel eliminates special-casing when `left = 0`. Without it, `RangeSum(0, r)` needs a separate branch. This is a clean trick — use it always.

### Template B: Prefix Sum + HashMap (Subarray Sum Equals K)

```csharp
// Count subarrays with sum exactly equal to k
// Key insight: sum(l, r) = k  ⟺  prefix[r+1] - prefix[l] = k
//              ⟺  prefix[l] = prefix[r+1] - k
// So for each prefix sum, check if (current_prefix - k) has been seen before
public int SubarraySum(int[] nums, int k)
{
    var prefixCount = new Dictionary<int, int>();
    prefixCount[0] = 1;  // empty prefix has sum 0 — handles subarray starting at index 0

    int currentSum = 0;
    int count = 0;

    for (int i = 0; i < nums.Length; i++)
    {
        currentSum += nums[i];

        // How many previous prefix sums equal (currentSum - k)?
        if (prefixCount.ContainsKey(currentSum - k))
            count += prefixCount[currentSum - k];

        prefixCount[currentSum] = prefixCount.GetValueOrDefault(currentSum, 0) + 1;
    }

    return count;
}
```

**Why this works:** If `prefix[r+1] - prefix[l] = k`, then `prefix[l] = prefix[r+1] - k`. For each position r, we ask: "How many earlier prefix sums equal `currentSum - k`?" The hash map stores prefix sum frequencies as we go → O(n) time, O(n) space.

**Critical: Initialize `prefixCount[0] = 1`.** This handles subarrays that start at index 0 (where `prefix[l] = prefix[0] = 0`). Missing this initialization is the #1 bug.

### Template C: 2D Prefix Sum

```csharp
// Build 2D prefix sum for matrix region queries
// prefix[i][j] = sum of all elements in rectangle (0,0) to (i-1, j-1)
public int[,] Build2DPrefixSum(int[][] matrix)
{
    int m = matrix.Length, n = matrix[0].Length;
    var prefix = new int[m + 1, n + 1];

    for (int i = 1; i <= m; i++)
    {
        for (int j = 1; j <= n; j++)
        {
            prefix[i, j] = matrix[i - 1][j - 1]
                          + prefix[i - 1, j]
                          + prefix[i, j - 1]
                          - prefix[i - 1, j - 1];  // inclusion-exclusion
        }
    }
    return prefix;
}

// Sum of region from (r1, c1) to (r2, c2) inclusive
public int RegionSum(int[,] prefix, int r1, int c1, int r2, int c2)
{
    return prefix[r2 + 1, c2 + 1]
         - prefix[r1, c2 + 1]
         - prefix[r2 + 1, c1]
         + prefix[r1, c1];  // add back the doubly-subtracted corner
}
```

**Inclusion-exclusion principle:** To get the rectangle sum, take the full rectangle, subtract the top strip, subtract the left strip, add back the corner that was subtracted twice. Draw it — it clicks immediately.

### Template D: Prefix XOR (for XOR range queries)

```csharp
// XOR is its own inverse: a ^ a = 0, a ^ 0 = a
// So prefix XOR works identically to prefix sum
// XOR(l, r) = prefixXor[r+1] ^ prefixXor[l]
public int[] BuildPrefixXor(int[] nums)
{
    int n = nums.Length;
    var prefix = new int[n + 1];
    for (int i = 0; i < n; i++)
        prefix[i + 1] = prefix[i] ^ nums[i];
    return prefix;
}
```

## 4. Complexity Deep Dive

### Time Complexity

| Operation | Complexity | Notes |
|-----------|-----------|-------|
| Build prefix sum | O(n) | Single pass |
| Range sum query | O(1) | Single subtraction |
| Subarray sum = k (with HashMap) | O(n) | Single pass, O(1) per element |
| 2D prefix sum build | O(m × n) | Single pass over matrix |
| 2D region query | O(1) | Constant arithmetic |

### Space Complexity

| Variant | Space | Notes |
|---------|-------|-------|
| 1D prefix array | O(n) | Stored explicitly |
| 1D with HashMap | O(n) | Map stores up to n distinct prefix sums |
| 2D prefix matrix | O(m × n) | Same size as input |

### Trade-offs

| Approach | Time per query | Preprocessing | Space | When to use |
|----------|---------------|--------------|-------|-------------|
| Naive sum | O(n) | None | O(1) | Single query, small n |
| Prefix sum | O(1) | O(n) | O(n) | Multiple queries, static array |
| Fenwick Tree | O(log n) | O(n) | O(n) | Queries + updates |
| Segment Tree | O(log n) | O(n) | O(4n) | Queries + updates + complex operations |

### When this approach breaks

- **Array is modified between queries.** Prefix sum is static — rebuilding after each update is O(n). Use Fenwick/Segment Tree for dynamic updates + queries.
- **You need max/min over a range, not sum.** Prefix sum doesn't help (max isn't invertible — you can't "subtract" it). Use sparse table or segment tree.
- **Floating point precision.** Prefix sums of floats accumulate rounding errors. For financial calculations, use decimal types or compensated summation (Kahan).

## 5. Edge Cases & Pitfalls

### Common senior-level mistakes

1. **Forgetting `prefixCount[0] = 1` in the HashMap variant.** Without this, subarrays starting at index 0 are missed. This bug passes many test cases but fails on inputs like `[k]` with target `k`.

2. **Off-by-one in prefix array indexing.** Convention: `prefix[i]` = sum of first `i` elements. So `prefix[0] = 0`, `prefix[n] = total sum`. Range `[l, r]` = `prefix[r+1] - prefix[l]`. If you use `prefix[r] - prefix[l-1]`, handle `l=0` carefully.

3. **Integer overflow.** Prefix sums can grow large. For an array of 10⁵ elements each up to 10⁴, prefix sum can reach 10⁹ — fits in `int`. But for 10⁵ elements each up to 10⁹, you need `long`. Always check.

4. **Applying prefix sum when sliding window works.** If the array has only positive integers and you want min/max length subarray with sum constraint → sliding window is O(n) time, O(1) space. Prefix sum works but wastes space.

5. **2D prefix sum off-by-one.** The inclusion-exclusion formula is error-prone. Draw the grid. Label the corners. The formula is: `prefix[r2+1][c2+1] - prefix[r1][c2+1] - prefix[r2+1][c1] + prefix[r1][c1]`. Practice it until it's muscle memory.

### Boundary conditions

- Empty array → prefix sum is `[0]`, no valid range queries
- Single element → `prefix = [0, nums[0]]`, range `[0,0]` = `prefix[1] - prefix[0]`
- All zeros → prefix sum is all zeros, every subarray sums to zero
- Negative numbers → prefix sum can decrease — this is FINE and is exactly why prefix sum works where sliding window doesn't

### Debugging heuristics

1. Print the prefix array and verify manually for a small input
2. Check that `prefix[0] = 0` (not `prefix[0] = nums[0]`)
3. For HashMap variant: check `prefixCount[0] = 1` is initialized before the loop
4. For 2D: verify the formula with a 2×2 matrix by hand

## 6. Variations (Pattern Stretching)

### Easy: Static range sum queries

**Range Sum Query - Immutable (LC 303).** Pure template A. Build prefix sum once, answer each query in O(1).

### Medium: Subarray sum equals K

**Subarray Sum Equals K (LC 560).** Template B. Prefix sum + HashMap. Count subarrays with exact sum. Works with negatives — this is what sliding window cannot do.

### Medium: Subarray sum divisible by K

**Subarray Sums Divisible by K (LC 974).** Variant of Template B: instead of tracking exact prefix sums, track `prefix[i] % k`. Two prefix sums with the same remainder → their difference is divisible by k. Handle negative remainders: `((prefix % k) + k) % k`.

### Medium-Hard: 2D region sum queries

**Range Sum Query 2D - Immutable (LC 304).** Template C. Inclusion-exclusion for 2D prefix sum.

### Hard: Maximum subarray sum with constraint

**Max Sum of Rectangle No Larger Than K (LC 363).** 2D prefix sum + for each pair of rows, reduce to 1D problem: "largest subarray sum ≤ k." Use prefix sum + sorted set (TreeSet / SortedList) with binary search.

### Evolution map

```
1D Range Sum → 1D Sum = K (HashMap) → 1D Divisible by K (modular arithmetic)
       ↓                                        ↓
   2D Region Sum                    Count subarrays with condition
       ↓                                        ↓
   2D + constraint                  Prefix XOR (same idea, different operation)
```

## 7. Interview-Level Problems

### Problem 1 (Medium): Subarray Sum Equals K

**LC 560**

**Summary:** Given an integer array `nums` and integer `k`, return the total number of subarrays whose sum equals `k`.

**Expected approach:** Prefix sum + HashMap. Track running sum. For each position, check how many previous prefix sums equal `currentSum - k`. Those correspond to subarrays ending here with sum exactly k.

**Key insight:** This problem has negative numbers, so sliding window doesn't work. The prefix sum + HashMap approach handles negatives naturally. The initialization `map[0] = 1` is critical — it accounts for subarrays starting at index 0. This is the foundational "prefix sum + HashMap" pattern that appears in dozens of variations.

---

### Problem 2 (Medium-Hard): Contiguous Array (Equal 0s and 1s)

**LC 525**

**Summary:** Given a binary array `nums`, find the maximum length of a contiguous subarray with equal number of 0 and 1.

**Expected approach:** Transform the problem: replace 0 with -1. Now "equal 0s and 1s" becomes "subarray sum = 0." Apply prefix sum + HashMap: for each prefix sum, store the earliest index where that sum occurred. If the same prefix sum appears again, the subarray between them has sum 0. Track maximum length.

**Key insight:** The transformation (`0 → -1`) converts a counting problem into a sum problem, which unlocks prefix sum. This "transform then apply standard technique" pattern is a staff-level move. The HashMap stores first occurrence (not count) because we want maximum length, not count.

---

### Problem 3 (Hard): Max Sum of Rectangle No Larger Than K

**LC 363**

**Summary:** Given an m × n matrix and integer k, find the max sum of a rectangle in the matrix such that its sum is no larger than k.

**Expected approach:** Fix two column boundaries (O(n²)). For each pair, compute row sums to reduce to a 1D problem: "largest prefix sum difference ≤ k." Use a sorted set (SortedList/TreeSet) to binary search for the closest prefix sum ≥ `currentPrefix - k`. Total: O(m × n² × log m) or O(n × m² × log n) depending on which dimension you fix.

**Key insight:** The 2D → 1D reduction via fixing column boundaries is a fundamental technique. The 1D subproblem uses prefix sum + sorted set (not just HashMap) because you need "sum ≤ k" (inequality), not "sum = k" (equality). The sorted set lets you binary search for the optimal previous prefix sum. This combines prefix sum, binary search, and 2D reduction.

## 8. Staff-Level Thinking

### Trade-offs between approaches

| Approach | Best for | Limitation |
|----------|---------|-----------|
| Prefix Sum + HashMap | Exact sum / divisibility conditions, works with negatives | O(n) space, static array only |
| Sliding Window | Min/max length with sum constraint, positives only | Breaks with negatives |
| Kadane's | Max subarray sum (no target, no constraint) | Only solves one specific problem |
| Fenwick / Segment Tree | Range queries with updates | O(log n) per query — overkill for static |

### Scalability considerations

- **Prefix sum is the O(n) preprocessing, O(1) query paradigm.** In production, this maps to precomputed aggregation tables (materialized views in databases, precomputed metrics in time-series stores). The trade-off is identical: pay upfront for fast reads.
- **Space is the cost.** For a 10⁶-element array, the prefix sum array is 4MB (int) or 8MB (long). Usually fine. For 2D with 10⁴ × 10⁴ matrix, the prefix sum is 400MB — now space matters.
- **For streaming data** where the array grows over time, prefix sum requires extending the array. This is O(1) amortized per new element (just append `prefix[n+1] = prefix[n] + newElement`). But if you need to modify historical data, prefix sum breaks — you need a Fenwick tree.

### When NOT to use prefix sum

- **Only positive numbers + length optimization** → Sliding window is O(1) space.
- **Max/min range queries** → Sparse table (O(1) query, O(n log n) space) or segment tree.
- **Dynamic array with updates** → Fenwick or segment tree.
- **When n is small and queries are few** → Brute force is simpler and the constant factor of prefix sum isn't worth it.

### How to explain this in interviews

**Opening (5 seconds):** "Range sum queries suggest prefix sum. I'll precompute cumulative sums so each query is O(1)."

**For the HashMap variant (10 seconds):** "I need `prefix[r] - prefix[l] = k`, which means I need a prefix sum value of `currentSum - k` earlier in the array. A HashMap tracks how many times each prefix sum has occurred → O(n) total."

**Why not sliding window (when asked):** "The array contains negative numbers, so shrinking the window from the left doesn't monotonically decrease the sum. Sliding window invariant breaks. Prefix sum + HashMap handles negatives naturally."

## 9. Real-World Mapping

| Production scenario | Prefix sum analog |
|---|---|
| **Cumulative metrics dashboards** | Grafana's cumulative counters — range query = difference of two points |
| **Database running totals** | `SUM() OVER (ORDER BY date)` window function — it's a prefix sum |
| **CDN bandwidth accounting** | Monthly bandwidth = `cumulative[end] - cumulative[start]` |
| **Financial P&L statements** | Year-to-date revenue/expense is a prefix sum; quarterly is a range query |
| **Image processing** | Integral image (summed area table) = 2D prefix sum, used for fast blurring, feature detection |
| **Distributed counters** | Kafka consumer offsets — lag = `latest offset - committed offset` — conceptually a prefix sum difference |
| **Rate limiting (sliding window counter)** | Approximate rate = `count[current_window] + fraction × count[previous_window]` — interpolation between prefix counts |

**System design relevance:** When designing a metrics or analytics system and someone asks "how do you compute aggregate X over any time range in O(1)?", the answer is a precomputed prefix sum (materialized as a cumulative table or running counter). This is used in time-series databases, OLAP cubes, and monitoring infrastructure universally.

## 10. 10-Second Recall Cheat Sheet

```
CORE FORMULA:
  prefix[i] = sum(nums[0..i-1])
  prefix[0] = 0  (always!)
  RangeSum(l, r) = prefix[r+1] - prefix[l]

PATTERNS:
  "range sum query"            → Build prefix array, O(1) per query
  "subarray sum = k"          → Prefix + HashMap: look for (currentSum - k) in map
  "subarray sum divisible by k" → Track prefix % k in map (handle negative mod!)
  "equal 0s and 1s"           → Transform 0→-1, then "subarray sum = 0"
  "2D region sum"             → 2D prefix sum with inclusion-exclusion

CRITICAL INIT:
  HashMap variant: prefixCount[0] = 1  (handles subarrays starting at index 0)

PREFIX SUM vs SLIDING WINDOW:
  Has negatives? → Prefix Sum (sliding window breaks)
  All positive + length optimization? → Sliding Window (O(1) space)

COMPLEXITY:
  Build: O(n)
  Query: O(1)
  Space: O(n)
  With HashMap: O(n) time, O(n) space

PITFALLS:
  - Forgetting prefix[0] = 0
  - Forgetting map[0] = 1 in HashMap variant
  - Integer overflow on large cumulative sums
  - Using prefix sum when sliding window suffices (wastes space)
```
