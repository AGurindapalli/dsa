## 1. Intuition First (No Jargon)

**The core idea:** A function that calls itself to solve smaller versions of the same problem, until it hits a base case it can solve directly. Every recursive solution has exactly two parts: (1) the base case — when to stop, and (2) the recursive case — how to reduce the problem.

**Real-world analogy:** You're an incident commander during a production outage affecting 50 microservices. You don't debug all 50 simultaneously. You identify the root service, check if IT is the source (base case). If not, you trace to its upstream dependency and repeat the same diagnostic process (recursive case). The chain terminates when you find the service that actually failed. Recursion IS this — delegating a smaller version of the same investigation to yourself.

**WHY this pattern exists:** Many data structures and problems are inherently self-similar — a tree is a node connected to smaller trees, a sorted array can be split into sorted halves, a combinatorial search space is a choice plus a smaller search space. Recursion lets your code mirror the structure of the problem directly. Trying to solve these iteratively often means manually managing a stack — you're just simulating recursion with more bookkeeping.

**The mental shift:** Stop thinking about what happens at every level. Trust the recursive call to handle its subproblem correctly. Your job is: (1) handle the base case, (2) break the problem into subproblems, (3) combine the results. That's it. If you try to mentally trace every recursive call, you'll lose the forest for the trees.

## 2. Pattern Recognition

### When to use

| Signal | Example |
|--------|---------|
| Problem has self-similar structure | Tree problems, nested structures, fractals |
| "Divide and conquer" | Merge sort, binary search, quicksort |
| "Generate all..." | Permutations, subsets, combinations (backtracking) |
| Recursive data structure | Trees, graphs (DFS), linked lists |
| Problem can be defined in terms of smaller subproblems | Fibonacci, factorial, tower of Hanoi |
| Mathematical recurrence | T(n) = f(T(n-1), T(n-2), ...) |

### Common keywords in problem statements

- "all possible", "generate", "enumerate"
- "tree", "subtree", "node"
- "divide", "split", "partition"
- "depth", "height", "level"
- "nested", "recursive structure"

### 10-Second Identification Rule

> **Can you define the solution to the problem in terms of solutions to smaller instances of the exact same problem?**
>
> If YES → Recursion is natural. Then ask: do subproblems overlap? If yes → add memoization (DP). If no → pure recursion or divide-and-conquer.
> If NO → The problem likely needs iteration, greedy, or a different paradigm.

## 3. Core Algorithm / Template

### The Recursion Template (3 parts, always)

```csharp
public ReturnType Solve(ProblemInput input)
{
    // 1. BASE CASE — the smallest problem you can solve directly
    if (IsBaseCase(input))
        return baseCaseAnswer;

    // 2. RECURSIVE CASE — reduce and delegate
    var subResult = Solve(SmallerInput(input));

    // 3. COMBINE — build the answer from subproblem result
    return Combine(input, subResult);
}
```

### Example 1: Factorial (single branch)

```csharp
// n! = n × (n-1)!
// Base case: 0! = 1
public long Factorial(int n)
{
    if (n <= 1) return 1;          // base case
    return n * Factorial(n - 1);    // reduce by 1, combine by multiplying
}
// Time: O(n) — n recursive calls
// Space: O(n) — n stack frames
```

### Example 2: Binary Search (single branch, halving)

```csharp
// Search for target in sorted array
public int BinarySearch(int[] nums, int target, int left, int right)
{
    if (left > right) return -1;                       // base case: not found

    int mid = left + (right - left) / 2;               // avoid overflow

    if (nums[mid] == target) return mid;                // base case: found
    if (nums[mid] < target)
        return BinarySearch(nums, target, mid + 1, right);  // right half
    else
        return BinarySearch(nums, target, left, mid - 1);   // left half
}
// Time: O(log n) — halving each time
// Space: O(log n) — log n stack frames (can be O(1) iteratively)
```

### Example 3: Merge Sort (two branches, full divide-and-conquer)

```csharp
public void MergeSort(int[] arr, int left, int right)
{
    if (left >= right) return;                    // base case: single element

    int mid = left + (right - left) / 2;
    MergeSort(arr, left, mid);                    // sort left half
    MergeSort(arr, mid + 1, right);               // sort right half
    Merge(arr, left, mid, right);                 // combine sorted halves
}

private void Merge(int[] arr, int left, int mid, int right)
{
    var temp = new int[right - left + 1];
    int i = left, j = mid + 1, k = 0;

    while (i <= mid && j <= right)
    {
        if (arr[i] <= arr[j])
            temp[k++] = arr[i++];
        else
            temp[k++] = arr[j++];
    }

    while (i <= mid) temp[k++] = arr[i++];
    while (j <= right) temp[k++] = arr[j++];

    Array.Copy(temp, 0, arr, left, temp.Length);
}
// Time: O(n log n) — log n levels, O(n) merge work per level
// Space: O(n) for temp array + O(log n) stack = O(n)
```

### Example 4: Tree traversal (the canonical recursive structure)

```csharp
// A binary tree IS recursion — left and right subtrees are smaller trees
public int MaxDepth(TreeNode root)
{
    if (root == null) return 0;                                  // base case
    int leftDepth = MaxDepth(root.Left);                         // recurse left
    int rightDepth = MaxDepth(root.Right);                       // recurse right
    return 1 + Math.Max(leftDepth, rightDepth);                  // combine
}
// Time: O(n) — visit every node once
// Space: O(h) — h = tree height (log n balanced, n skewed)
```

### The "Leap of Faith"

The most important concept in recursion:

```
When writing MaxDepth:
  - ASSUME MaxDepth(root.Left) correctly returns the depth of the left subtree
  - ASSUME MaxDepth(root.Right) correctly returns the depth of the right subtree
  - YOUR JOB: combine them correctly (add 1 for current node, take max)
  - DON'T trace what happens inside the recursive calls
```

This "assume the recursive call works" approach is called the **recursive leap of faith**. It's how you write correct recursive code without your brain exploding.

## 4. Complexity Deep Dive

### Time Complexity of Recursive Functions

Use the **recursion tree method** or **Master Theorem**:

| Pattern | Recurrence | Complexity | Example |
|---------|-----------|------------|---------|
| Linear recursion | T(n) = T(n-1) + O(1) | O(n) | Factorial, linked list traversal |
| Linear with work | T(n) = T(n-1) + O(n) | O(n²) | Selection sort, bad partition in quicksort |
| Binary split | T(n) = T(n/2) + O(1) | O(log n) | Binary search |
| Divide & conquer | T(n) = 2T(n/2) + O(n) | O(n log n) | Merge sort |
| Binary tree | T(n) = 2T(n/2) + O(1) | O(n) | Tree traversal |
| Double recursion | T(n) = T(n-1) + T(n-2) | O(2ⁿ) | Naive Fibonacci |
| K-way branching | T(n) = kT(n/k) + O(n) | O(n log n) | K-way merge sort |

### Space Complexity

**Recursion space = O(maximum stack depth)**

The stack depth is the longest chain of recursive calls alive simultaneously:
- Linear recursion (factorial): O(n)
- Binary search: O(log n)
- Balanced tree traversal: O(log n)
- Skewed tree traversal: O(n)
- Merge sort: O(log n) for stack + O(n) for merge array

### Trade-offs

| | Recursive | Iterative |
|---|---|---|
| **Clarity** | Mirrors problem structure naturally | Can be harder to read for tree/graph problems |
| **Stack space** | O(depth) — can overflow on large inputs | O(1) or explicit stack under your control |
| **Performance** | Function call overhead per frame | No call overhead |
| **When to prefer** | Trees, graphs, backtracking, D&C | Tail-recursive patterns, tight performance constraints |

### When recursion breaks

- **Deep recursion:** C# default stack size is 1MB → ~10K-50K frames depending on frame size. For n = 10⁶, recursion will stack overflow. Convert to iteration.
- **Overlapping subproblems without memoization:** Naive Fibonacci is O(2ⁿ). Add memoization → O(n). This is literally the transition from recursion to DP.
- **Non-tail recursion that you need to optimize:** Tail call optimization (TCO) is NOT guaranteed in C# / Java. If you need O(1) stack space, convert to a loop.

## 5. Edge Cases & Pitfalls

### Common senior-level mistakes

1. **Missing or wrong base case.** This causes infinite recursion → stack overflow. Every recursive function needs at least one base case that doesn't recurse. Common bug: base case handles `n == 0` but input can be negative.

2. **Not reducing toward the base case.** If your recursive call doesn't make the problem strictly smaller, you'll recurse forever. Classic: calling `Solve(n)` instead of `Solve(n - 1)`.

3. **Mutating shared state between recursive calls.** If you modify an array/list during one branch and it affects the other branch, you get subtle bugs. This is the #1 backtracking pitfall — forgetting to "undo" the choice.

    ```csharp
    // WRONG: shared list mutation leaks between branches
    path.Add(node.Val);
    DFS(node.Left, path);
    DFS(node.Right, path);  // path still has Left's additions!

    // CORRECT: undo (backtrack) after each branch
    path.Add(node.Val);
    DFS(node.Left, path);
    DFS(node.Right, path);
    path.RemoveAt(path.Count - 1);  // backtrack
    ```

4. **Ignoring stack overflow risk.** For n > ~10K, test whether your recursion depth is safe. In interviews, mention it: "This recursion goes O(n) deep. For n = 10⁵, I'd convert to iterative with an explicit stack."

5. **Redundant recomputation.** If you see the same function called with the same arguments multiple times, you need memoization. Draw the recursion tree — if it has overlapping nodes, it's crying out for DP.

### Boundary conditions

- `null` / empty input → base case, return identity value (0, empty list, etc.)
- Single element → often a base case
- Two elements → test that reduction works correctly (off-by-one here is common)
- Maximum recursion depth → mention stack overflow risk

### Debugging heuristics

1. **Add a depth parameter and print it.** `Solve(n, depth=0)` → print `depth` and `n` at each call. This reveals infinite loops immediately.
2. **Verify the base case fires.** If it never triggers, you're never reducing.
3. **Check the combine step.** After recursive calls return, are you combining correctly? Off-by-one errors in combining are common and hard to spot.

## 6. Variations (Pattern Stretching)

### Easy: Single-branch linear recursion

**Factorial, sum of list, reverse a string.** One recursive call per frame, problem reduces by 1 each time. O(n) time, O(n) space.

### Medium: Multi-branch recursion

**Tree traversals, Fibonacci.** Two or more recursive calls per frame. Time depends on branching factor and depth. Space is O(depth), not O(nodes), because only one path is on the stack at a time.

### Medium-Hard: Divide and conquer

**Merge sort, quicksort, closest pair of points.** Split problem into independent subproblems, solve recursively, merge results. The "merge" step is where the real work happens.

### Hard: Recursion with state (backtracking)

**N-Queens, Sudoku solver, generate all permutations.** Recursion explores a decision tree. At each node, make a choice, recurse, undo the choice. The "undo" step is what makes it backtracking, not just recursion.

### Evolution map

```
Linear recursion → Multi-branch → Divide & Conquer
       ↓                ↓              ↓
  Tail recursion    Tree problems   Merge Sort / Quick Sort
       ↓                ↓
  Convert to loop   Add memoization → DP (Tier 4)
                        ↓
                    Backtracking (Tier 3)
```

## 7. Interview-Level Problems

### Problem 1 (Medium): Pow(x, n) — Fast Exponentiation

**LC 50**

**Summary:** Implement `pow(x, n)` — calculate x raised to the power n.

**Expected approach:** Recursion with halving. `x^n = (x^(n/2))² if n is even, x × (x^(n/2))² if n is odd`. This reduces the problem by half each step → O(log n) time.

**Key insight:** Naive recursion `x × pow(x, n-1)` is O(n). The halving approach is O(log n). This is the foundational "divide and conquer saves you from linear" pattern. Handle negative n by computing `1.0 / pow(x, -n)`. Watch for integer overflow when negating `int.MinValue`.

---

### Problem 2 (Medium-Hard): Generate All Valid Parentheses

**LC 22**

**Summary:** Given n pairs of parentheses, generate all combinations of well-formed parentheses.

**Expected approach:** Recursion that tracks `open` and `close` counts. At each step, you can add `(` if `open < n`, and `)` if `close < open`. Base case: string length = 2n.

**Key insight:** The constraint `close < open` is what ensures validity. This is recursion as *constrained enumeration* — you're traversing a decision tree where invalid branches are pruned. It bridges recursion → backtracking. The number of valid strings is the Catalan number C(n) = O(4ⁿ/n√n).

---

### Problem 3 (Hard): K-th Symbol in Grammar

**LC 779**

**Summary:** Row 1 is `0`. Each row is formed by replacing `0` with `01` and `1` with `10`. Find the k-th symbol in row n.

**Expected approach:** Don't generate the row (it's 2ⁿ⁻¹ long). Instead, recurse: the k-th symbol in row n depends on the ⌈k/2⌉-th symbol in row n-1. If k is odd, it's the same as the parent. If k is even, it's the complement.

**Key insight:** This is recursion as *problem reduction without materialization*. You never build the actual data — you trace back through the recursive definition to compute a single value. The key observation is that each element's value is determined by its parent in the previous row. O(n) time, O(n) stack space.

## 8. Staff-Level Thinking

### Trade-offs between approaches

| Decision | Recursion | Iteration | Memoized Recursion (DP) |
|----------|-----------|-----------|------------------------|
| Readability for tree/graph problems | Best | Worse (manual stack) | Good |
| Stack safety for large n | Risky (overflow) | Safe | Risky (overflow) |
| Performance overhead | Function call overhead | Minimal | Overhead + hash map cost |
| When overlapping subproblems exist | Exponential blowup | N/A | Polynomial collapse |

### Scalability considerations

- **Stack depth is the real limit.** In production services (EKS pods with default stack sizes), a recursive DFS on a graph with 10⁶ nodes will crash. Always convert deep recursion to iterative with an explicit stack for production code.
- **Tail call optimization (TCO)** is NOT supported in C# or Java. Languages like Scheme and some Kotlin configurations support it. In interviews, mentioning TCO shows depth. In production C#, convert tail-recursive functions to loops.
- **Recursion overhead** is real but usually insignificant compared to I/O. For CPU-bound hot paths (e.g., game loop, real-time pricing), prefer iteration. For I/O-bound code, recursion readability wins.

### When NOT to use recursion

- **Linear problems with simple state.** Iterating over an array to find a max doesn't benefit from recursion. Use a loop.
- **Deep recursion on large inputs.** n = 10⁵ or more? Use iteration. Stack overflow is not a graceful failure.
- **Overlapping subproblems without memoization.** If your recursion tree has repeated computations, you're wasting exponential time. Add memoization or convert to bottom-up DP.
- **When the iterative version is equally clear.** Binary search is easy to write iteratively. Fibonacci is trivial iteratively. Don't recurse for recursion's sake.

### How to explain this in interviews

**Opening (5 seconds):** "This has recursive structure — each [node/subproblem] reduces to smaller [nodes/subproblems] of the same form."

**Base case articulation (5 seconds):** "My base case is [empty/null/size 1], which returns [identity value]. This guarantees termination."

**Complexity defense (10 seconds):** "Each call does O(1) work and makes [1/2/k] recursive calls, reducing the problem by [1/half]. So the recurrence is T(n) = [relation], giving O([complexity])."

**Stack overflow awareness (5 seconds):** "For very large n, I'd convert this to an iterative approach with an explicit stack to avoid stack overflow."

Understanding recursion isn't academic — it's how you reason about any hierarchical or self-similar system.

## 10. 10-Second Recall Cheat Sheet

```
RECURSION = BASE CASE + RECURSIVE CASE + COMBINE

TEMPLATE:
  if (base case) → return directly
  subResult = Solve(smaller problem)
  return Combine(current, subResult)

LEAP OF FAITH:
  Trust the recursive call handles its subproblem correctly.
  Your job: handle THIS level only.

COMPLEXITY:
  Linear (T(n) = T(n-1) + O(1))        → O(n)
  Halving (T(n) = T(n/2) + O(1))       → O(log n)
  Divide & Conquer (T(n) = 2T(n/2) + O(n)) → O(n log n)
  Double branch (T(n) = T(n-1) + T(n-2))   → O(2^n) — NEEDS MEMOIZATION

SPACE = O(max stack depth):
  Linear recursion → O(n)
  Halving / balanced tree → O(log n)
  Skewed tree → O(n)

PITFALLS:
  - Missing base case → infinite recursion
  - Not reducing toward base case → infinite recursion
  - Shared mutable state between branches → backtracking bug
  - Deep recursion (n > 10K) → stack overflow, convert to iterative
  - Overlapping subproblems → add memoization or use DP

WHEN TO CONVERT TO ITERATION:
  - Stack depth > 10K
  - Performance-critical hot path
  - Tail recursion (C#/Java won't optimize it)
```
