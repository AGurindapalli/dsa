## 1. 🧠 Intuition First (No Jargon)

### First Principles

A heap is an **ordered access structure**: it guarantees O(log n) insertion and O(1) peek / O(log n) removal of the extreme element (max or min). Nothing else — no O(1) search, no sorted order, no random access.

The underlying structure is a **complete binary tree stored in an array**, where every parent is greater (max-heap) or smaller (min-heap) than its children. This invariant is maintained by `sift-up` on insert and `sift-down` on remove.

The real value: **you don't need full sort order**. When a problem only ever asks "give me the current best element", sorting is overkill — it computes total order when you only need the top. A heap gives you the top in O(1), updates in O(log n), and never does unnecessary work for the elements you'll never look at.

### Real-World Analogy

**Emergency room triage**. Patients don't get seen in arrival order (queue). They get seen by severity — the most critical patient first. When a new critical patient arrives, they jump ahead. When the most critical patient is treated, the next most critical moves to front. This is a max-heap on severity score.

The ER never needs a *sorted list of all patients by severity* — it only ever needs to answer "who's most critical right now?" A heap is that data structure: O(1) to check who's at the top, O(log n) to add a new patient or discharge the current one.

### Why This Pattern Exists

Sorting is O(n log n) and gives you all n elements in order. But if you only need the top k elements (k << n), sorting wastes work. A heap of size k maintains the k best elements at O(log k) per insertion — total O(n log k) instead of O(n log n). For k = 10 and n = 10M, this is a 20x speedup.

The second driver: **online/streaming algorithms**. You can't sort a stream — you don't have all elements upfront. But a heap handles insertions dynamically, always maintaining access to the current extreme. This makes it the go-to for "maintain the top k from a continuous stream" problems.

## 2. 🔍 Pattern Recognition

### When to Use a Heap

| Situation | Heap type |
|---|---|
| Find top k largest elements | Min-heap of size k |
| Find top k smallest elements | Max-heap of size k |
| Median from a data stream | Two heaps (max-heap + min-heap) |
| Merge k sorted lists/arrays | Min-heap of (value, list_index, element_index) |
| Shortest path (Dijkstra) | Min-heap of (cost, node) |
| Task scheduling by priority | Max-heap of (priority, task) |
| K closest points to origin | Max-heap of size k on distance |
| Running maximum / minimum | Max/min heap |
| "Process the cheapest/fastest first" | Min-heap |

### Input/Output Signals

**Problem likely needs a heap when:**
- "Find the k-th largest/smallest element"
- "Return top k most frequent"
- "Find median of a stream"
- "Merge k sorted lists"
- "Minimum cost to do X" (greedy + priority)
- "Schedule tasks with cooldown"
- "Find k closest points"
- "Connect ropes with minimum cost"
- Graph problem asking for "shortest path" with weights

### Common Keywords

`top k`, `k largest`, `k smallest`, `k most frequent`, `k closest`, `median`, `merge k`, `stream`, `online algorithm`, `priority`, `schedule`, `minimum cost`, `shortest path`, `greedy selection`, `running minimum/maximum`

### 10-Second Identification Rule

> **If the problem asks for the k-th something, or repeatedly needs the current best element from a dynamically-changing collection → heap.**

The signal is: you need ordering, but not full sort order. If k is fixed and small relative to n, heap beats sort. If the data arrives in a stream, heap is the only option.

## 3. ⚙️ Core Algorithm / Template

### Pattern 1: Top K Largest Elements (Min-Heap of size k)

```rust
use std::collections::BinaryHeap;
use std::cmp::Reverse;

fn top_k_largest(nums: &[i32], k: usize) -> Vec<i32> {
    // Min-heap of size k: smallest of the "top k" sits at top
    // When a new element beats the current minimum, swap it in
    let mut min_heap: BinaryHeap<Reverse<i32>> = BinaryHeap::new();

    for &n in nums {
        min_heap.push(Reverse(n));
        if min_heap.len() > k {
            min_heap.pop(); // evict the smallest — keep the k largest
        }
    }

    // Collect — descending order
    let mut result: Vec<i32> = min_heap.into_iter().map(|Reverse(v)| v).collect();
    result.sort_unstable_by(|a, b| b.cmp(a));
    result
}

// K-th largest only (no need to collect all k)
fn kth_largest(nums: &[i32], k: usize) -> i32 {
    let mut min_heap: BinaryHeap<Reverse<i32>> = BinaryHeap::new();
    for &n in nums {
        min_heap.push(Reverse(n));
        if min_heap.len() > k {
            min_heap.pop();
        }
    }
    let Reverse(val) = min_heap.peek().unwrap();
    *val
}
```

**Mental model:** "Keep a bouncer at the door of a VIP club with exactly k spots. Bouncer = min-heap top (weakest VIP). When someone new arrives, if they outrank the weakest VIP, the weakest leaves and the new person enters." The heap top is always the current threshold — "you must be at least this good to stay."

**Counterintuitive:** To find the k *largest*, use a *min*-heap. To find the k *smallest*, use a *max*-heap. The heap holds the answer set; its top is the eviction candidate.

### Pattern 2: Top K Most Frequent Elements

```rust
use std::collections::{BinaryHeap, HashMap};
use std::cmp::Reverse;

fn top_k_frequent(nums: &[i32], k: usize) -> Vec<i32> {
    // Step 1: frequency map
    let mut freq: HashMap<i32, usize> = HashMap::new();
    for &n in nums { *freq.entry(n).or_insert(0) += 1; }

    // Step 2: min-heap of (freq, value) — size k
    let mut min_heap: BinaryHeap<Reverse<(usize, i32)>> = BinaryHeap::new();
    for (&val, &count) in &freq {
        min_heap.push(Reverse((count, val)));
        if min_heap.len() > k {
            min_heap.pop();
        }
    }

    min_heap.into_iter().map(|Reverse((_, v))| v).collect()
}
```

**Mental model:** Same "size-k bouncer" pattern, but the "VIP score" is frequency instead of value. First build the frequency map (O(n)), then run the k-size heap over the distinct elements (O(m log k) where m = distinct count).

**Alternative — bucket sort for O(n):**
```rust
fn top_k_frequent_linear(nums: &[i32], k: usize) -> Vec<i32> {
    let mut freq: HashMap<i32, usize> = HashMap::new();
    for &n in nums { *freq.entry(n).or_insert(0) += 1; }

    // Bucket: index = frequency, value = list of nums with that frequency
    let mut buckets: Vec<Vec<i32>> = vec![vec![]; nums.len() + 1];
    for (&val, &count) in &freq { buckets[count].push(val); }

    // Collect from highest frequency down
    let mut result = Vec::new();
    for bucket in buckets.iter().rev() {
        for &val in bucket {
            result.push(val);
            if result.len() == k { return result; }
        }
    }
    result
}
```
Mention bucket sort in interviews — O(n) vs O(n log k) signals you know the full solution space.

### Pattern 3: Median from Data Stream (Two Heaps)

```rust
use std::collections::BinaryHeap;
use std::cmp::Reverse;

struct MedianFinder {
    lower: BinaryHeap<i32>,         // max-heap: left half (smaller elements)
    upper: BinaryHeap<Reverse<i32>>, // min-heap: right half (larger elements)
}

impl MedianFinder {
    fn new() -> Self {
        Self { lower: BinaryHeap::new(), upper: BinaryHeap::new() }
    }

    fn add_num(&mut self, num: i32) {
        // Always push to lower first
        self.lower.push(num);

        // Balance: lower's max must be <= upper's min
        if let (Some(&lo_max), Some(&Reverse(up_min))) =
            (self.lower.peek(), self.upper.peek())
        {
            if lo_max > up_min {
                let val = self.lower.pop().unwrap();
                self.upper.push(Reverse(val));
            }
        }

        // Size invariant: lower can have at most 1 more element than upper
        if self.lower.len() > self.upper.len() + 1 {
            let val = self.lower.pop().unwrap();
            self.upper.push(Reverse(val));
        } else if self.upper.len() > self.lower.len() {
            let Reverse(val) = self.upper.pop().unwrap();
            self.lower.push(val);
        }
    }

    fn find_median(&self) -> f64 {
        if self.lower.len() > self.upper.len() {
            *self.lower.peek().unwrap() as f64
        } else {
            let lo = *self.lower.peek().unwrap() as f64;
            let Reverse(hi) = *self.upper.peek().unwrap();
            (lo + hi as f64) / 2.0
        }
    }
}
```

**Mental model:** Partition the stream into two halves — lower half in a max-heap (top = largest of the small), upper half in a min-heap (top = smallest of the large). Median = top of lower (odd total) or average of both tops (even total). Maintain two invariants:
1. `lower.max <= upper.min` (all left < all right)
2. `|lower.len() - upper.len()| <= 1` (balanced halves)

### Pattern 4: Merge K Sorted Lists

```rust
use std::collections::BinaryHeap;
use std::cmp::Reverse;

fn merge_k_sorted(lists: Vec<Vec<i32>>) -> Vec<i32> {
    // Heap entry: (value, list_index, element_index)
    let mut min_heap: BinaryHeap<Reverse<(i32, usize, usize)>> = BinaryHeap::new();

    // Seed with first element of each list
    for (i, list) in lists.iter().enumerate() {
        if !list.is_empty() {
            min_heap.push(Reverse((list[0], i, 0)));
        }
    }

    let mut result = Vec::new();
    while let Some(Reverse((val, list_idx, elem_idx))) = min_heap.pop() {
        result.push(val);
        let next_idx = elem_idx + 1;
        if next_idx < lists[list_idx].len() {
            min_heap.push(Reverse((lists[list_idx][next_idx], list_idx, next_idx)));
        }
    }
    result
}
```

**Mental model:** The heap always holds exactly one "candidate" from each list — the current minimum candidate. Pop the global minimum, push the next element from the same list. The heap never holds more than k elements. Total work: O(n log k) where n = total elements.

### Pattern 5: Dijkstra's Algorithm

```rust
use std::collections::BinaryHeap;
use std::cmp::Reverse;

fn dijkstra(graph: &Vec<Vec<(usize, i32)>>, start: usize) -> Vec<i32> {
    let n = graph.len();
    let mut dist = vec![i32::MAX; n];
    let mut heap: BinaryHeap<Reverse<(i32, usize)>> = BinaryHeap::new();

    dist[start] = 0;
    heap.push(Reverse((0, start)));

    while let Some(Reverse((cost, node))) = heap.pop() {
        // Stale entry — we already found a shorter path to this node
        if cost > dist[node] { continue; }

        for &(next, weight) in &graph[node] {
            let new_cost = dist[node] + weight;
            if new_cost < dist[next] {
                dist[next] = new_cost;
                heap.push(Reverse((new_cost, next)));
            }
        }
    }
    dist
}
```

**Mental model:** Greedy expansion — always process the unvisited node with the smallest known distance. The heap maintains the frontier ordered by cost. "Stale entry" check (`cost > dist[node]`) replaces a traditional visited array — a node can be in the heap multiple times with different costs; skip the outdated ones.

### Pattern 6: Custom Comparator Heap

```rust
use std::collections::BinaryHeap;
use std::cmp::Ordering;

// When you need multi-field ordering or non-standard comparison
#[derive(Eq, PartialEq)]
struct Task {
    priority: i32,
    deadline: i32,
    id: usize,
}

impl Ord for Task {
    fn cmp(&self, other: &Self) -> Ordering {
        // Primary: higher priority first
        // Tiebreak: earlier deadline first (lower deadline = higher urgency)
        self.priority.cmp(&other.priority)
            .then(other.deadline.cmp(&self.deadline))
    }
}

impl PartialOrd for Task {
    fn partial_cmp(&self, other: &Self) -> Option<Ordering> {
        Some(self.cmp(other))
    }
}

// BinaryHeap<Task> is now a max-heap ordered by (priority desc, deadline asc)
```

**Mental model:** Implement `Ord` on your struct. `BinaryHeap` is always a max-heap — "larger" according to your `Ord` is at the top. For min-heap behavior, either wrap with `Reverse<T>` or invert the comparison in `cmp`.

## 4. 📊 Complexity Deep Dive

| Operation | BinaryHeap | Sorted Array | BST (BTreeMap) |
|---|---|---|---|
| Insert | O(log n) | O(n) | O(log n) |
| Get max/min | O(1) | O(1) | O(log n) |
| Remove max/min | O(log n) | O(1) | O(log n) |
| Remove arbitrary | O(n) | O(n) | O(log n) |
| Search | O(n) | O(log n) | O(log n) |
| Build from array | O(n) | O(n log n) | O(n log n) |

### Top K Complexity Comparison

| Approach | Time | Space |
|---|---|---|
| Sort + slice | O(n log n) | O(1) extra |
| Min-heap of size k | O(n log k) | O(k) |
| QuickSelect | O(n) avg, O(n²) worst | O(1) |
| Counting sort (bounded int) | O(n + range) | O(range) |

For k = n, sort wins (lower constant). For k << n, heap wins. For k = 1 (maximum), a single pass is O(n).

### Build Heap: O(n) vs O(n log n)

Building a heap by inserting n elements one-by-one: O(n log n).
Building via `BinaryHeap::from(vec)` (heapify): **O(n)**.

```rust
// O(n log n) — don't do this for large n
let mut heap = BinaryHeap::new();
for x in vec { heap.push(x); }

// O(n) — use this
let heap: BinaryHeap<i32> = vec.into_iter().collect();
// or
let heap = BinaryHeap::from(vec);
```

The O(n) heapify works because most nodes are near the bottom of the tree (leaf level) and require O(1) sift-down, not O(log n).

### When Heap Breaks

1. **Need arbitrary removal** — BinaryHeap has no O(log n) arbitrary removal. If you need "decrease-key" (update priority of an existing element), Rust's BinaryHeap doesn't support it. Common workaround: lazy deletion (push a new entry, ignore stale ones via a visited/version check — same as Dijkstra's stale entry pattern).
2. **Need search by value** — O(n) in a heap. Use a HashMap alongside the heap if you need O(1) lookup.
3. **Sorted iteration needed** — heap destruction order gives sorted output, but in O(n log n). Just sort instead.
4. **Range queries** — "k-th element in range [l, r]" — segment tree or wavelet tree, not heap.

## 5. ⚠️ Edge Cases & Pitfalls

### Senior-Level Mistakes

**1. Wrong heap type for "top k largest"**
```rust
// WRONG mental model: "I want largest, so use max-heap"
// A max-heap of ALL elements doesn't help — you still iterate everything

// CORRECT: min-heap of size k
// The min-heap top = current k-th largest = the eviction threshold
let mut min_heap: BinaryHeap<Reverse<i32>> = BinaryHeap::new();
// If new element > heap.top() → pop min, push new (heap handles this via size check)
```

**2. Forgetting stale entry check in Dijkstra**
```rust
// Without this, you re-process nodes that already have optimal distance
// Cost: O(E log E) total heap ops instead of O((V+E) log V)
if cost > dist[node] { continue; } // MANDATORY stale check
```

**3. Tuple ordering in Rust's BinaryHeap**

Rust tuples compare lexicographically. `(cost, node)` sorts by cost first, then node. This matters for Dijkstra — ties broken by node ID, which is usually fine. But for custom problems, be intentional:
```rust
// (i32, usize) in BinaryHeap — larger cost = higher priority (max-heap default)
// Use Reverse<(i32, usize)> for min-cost-first behavior
heap.push(Reverse((cost, node)));
```

**4. Two-heap median — invariant violation order**

When adding to MedianFinder, the order of operations matters:
1. Push to lower (always)
2. Fix value invariant (lower.max <= upper.min)
3. Fix size invariant

Doing step 3 before step 2 can leave an incorrect element in the wrong heap.

**5. k = 0 or k > n in Top K**
```rust
// Guard at function entry
assert!(k > 0 && k <= nums.len(), "k out of bounds");
// Or handle gracefully with min(k, nums.len())
let k = k.min(nums.len());
```

**6. Integer overflow in Dijkstra**
```rust
// dist[node] + weight can overflow if dist[node] = i32::MAX
// Always initialize distances to i32::MAX and guard before adding
if dist[node] != i32::MAX {
    let new_cost = dist[node] + weight;
    // ...
}
// Or use i64 for distances when weights are large
```

**7. Rust's `Reverse` unwrapping pattern**
```rust
// Correct destructuring
let Reverse(val) = min_heap.pop().unwrap();
// Not:
let val = min_heap.pop().unwrap().0; // this works too but less idiomatic
```

### Debugging Heuristics

- Wrong k-th element? → Check if you're using min-heap for largest or max-heap for smallest (counterintuitive)
- Dijkstra giving wrong distances? → Check stale entry guard and initialization (dist[start] = 0, rest = MAX)
- Median wrong on even count? → Verify both heaps are same size and you're averaging both tops
- Merge k lists skipping elements? → Check that you're pushing `next_idx` from the same `list_idx`
- Heap producing unexpected order? → Verify `Ord` implementation; add debug print of comparison results

## 6. 🔁 Variations (Pattern Stretching)

### Easy
- **Last Stone Weight** — max-heap, pop two heaviest, push difference if non-zero
- **Kth Largest Element in a Stream** — maintain min-heap of size k; top = k-th largest
- **Relative Ranks** — sort with index tracking; assign medal strings

### Medium
- **Top K Frequent Elements** — frequency map + min-heap of size k
- **K Closest Points to Origin** — max-heap of size k on Euclidean distance
- **Kth Largest Element in an Array** — min-heap of size k, or QuickSelect
- **Task Scheduler** — frequency map + max-heap; simulate cooldown cycles
- **Reorganize String** — max-heap on char frequency; greedily place most frequent
- **Find Median from Data Stream** — two heaps (max + min)

### Medium-Hard
- **Minimum Cost to Connect Sticks (Ropes)** — min-heap; always merge two cheapest (greedy)
- **K Pairs with Smallest Sums** — min-heap seeded with (nums1[i] + nums2[0], i, 0)
- **Ugly Number II** — three pointers or min-heap tracking next ugly numbers
- **IPO (Maximize Capital)** — two heaps: future projects sorted by capital; available projects by profit

### Hard
- **Merge K Sorted Lists** — min-heap of (value, list_idx, elem_idx)
- **Sliding Window Median** — two heaps + lazy deletion with a "to-remove" hashmap
- **Find Median from Data Stream** — two heaps (medium, but full implementation is hard)
- **Smallest Range Covering Elements from K Lists** — min-heap tracking current element per list, window = [min, max in heap]
- **Maximum Performance of a Team** — sort engineers by speed desc; min-heap of efficiency of size k

### How the Pattern Evolves

```
Base: push/pop for priority-ordered access
↓
Top K: size-k heap acts as a filter — evict weak, keep strong
↓
Streaming Top K: same, but over an online stream (can't sort upfront)
↓
Two heaps: maintain partition of data for median / balanced split
↓
Heap + HashMap: lazy deletion for arbitrary removal
↓
Multi-source heap: merge k sorted inputs (one pointer per input)
↓
Heap in graph algorithms: Dijkstra, Prim's MST, A*
↓
Heap-optimized DP: priority-queue driven DP state transitions
```

## 7. 🧪 Interview-Level Problems

### Problem 1 — Medium: Task Scheduler
**LeetCode 621**

**Summary:** Given a list of tasks (letters A-Z) and a cooldown period `n`, find the minimum intervals to execute all tasks. Same task must have `n` intervals between executions. Can idle.

**Expected approach:**
- Frequency map of tasks → max-heap on frequency
- Simulate in rounds of `n+1` slots: pop up to `n+1` tasks, execute them, decrement frequency, re-push if > 0
- Idle slots = max(0, (n+1) - tasks executed this round)
- Continue until heap is empty

**Key insight:** You always want to greedily execute the highest-frequency task to minimize idle time. The max-heap ensures this. The mathematical formula shortcut (`(max_freq - 1) * (n + 1) + count_of_max_freq_tasks`) gives O(n) time, but the heap simulation is what interviewers want to see — it generalizes to harder variants with different cooldown rules.

---

### Problem 2 — Medium-Hard: K Pairs with Smallest Sums
**LeetCode 373**

**Summary:** Given two sorted arrays `nums1` and `nums2`, find the k pairs `(nums1[i], nums2[j])` with the smallest sums.

**Expected approach:**
- Min-heap seeded with `(nums1[i] + nums2[0], i, 0)` for all valid `i` (up to k entries)
- On each pop of `(sum, i, j)`: add pair to result, push `(nums1[i] + nums2[j+1], i, j+1)` if `j+1 < nums2.len()`
- Stop after k pairs

**Key insight:** The array is sorted — `nums1[i] + nums2[j]` is smallest when both indices are small. The key observation is you don't need to seed all `(i, 0)` combinations — only the first `min(k, nums1.len())` since pairs beyond that index can't be in the top k. On each pop, advance only the `j` index (the `i` index is fixed per "lane"). This is the exact same structure as merge k sorted lists — each "lane" `i` generates `(nums1[i] + nums2[0])`, `(nums1[i] + nums2[1])`, ... in sorted order.

---

### Problem 3 — Hard: Smallest Range Covering Elements from K Lists
**LeetCode 632**

**Summary:** Given k sorted lists of integers, find the smallest range `[a, b]` such that at least one number from each list lies within `[a, b]`.

**Expected approach:**
- Min-heap of `(value, list_idx, elem_idx)` — seeded with first element of each list
- Track current range max separately (updated on each push)
- On each step: range = `[heap.min, current_max]`; try to shrink by popping min and advancing to next element in that list
- Update answer when current range is smaller than best range
- Stop when any list is exhausted (popping its last element means a list is no longer represented)

**Key insight:** The window `[min_in_heap, tracked_max]` always covers at least one element from each list (invariant: exactly one element per list is in the heap). Shrinking the window = advancing the list that contributes the current minimum (move its pointer forward). The window can only shrink from the left (move min up); the max is tracked separately and can only grow or stay. This is the "sliding window over k lists simultaneously" pattern — the heap enforces that all k lists are always represented.

## 8. 🧠 Staff-Level Thinking

### Trade-offs Between Approaches

| Problem | Heap | Alternative | When to switch |
|---|---|---|---|
| Top K from static array | O(n log k) | QuickSelect O(n) avg | k << n and you need optimal; heap simpler to code |
| Top K frequent | O(n log k) | Bucket sort O(n) | n is bounded; bucket sort is O(n) but O(n) space |
| Median stream | O(log n) per insert | Order statistics tree O(log n) | Need more than just median (percentiles) → BST |
| Dijkstra | O((V+E) log V) | Bellman-Ford O(VE) | Negative weights → Bellman-Ford |
| Dijkstra | O((V+E) log V) | Dial's algorithm O(V+E+W) | Integer weights in range [0,W] → Dial's |
| Priority queue | BinaryHeap | BTreeMap | Need arbitrary removal or decrease-key → BTreeMap |

### Lazy Deletion Pattern (No Decrease-Key)

Rust's `BinaryHeap` has no `decrease_key`. The standard workaround:

```rust
// Instead of updating an existing entry, push a new one with updated priority
heap.push(Reverse((new_cost, node)));

// On pop, check if this entry is stale
if popped_cost > dist[node] { continue; } // stale — skip it
```

This is what Dijkstra does. Generalizes to any heap problem where elements need priority updates. Downside: heap can hold O(E) entries instead of O(V). For dense graphs (E ~ V²), this matters — use an indexed priority queue or a `BTreeSet` instead.

**BTreeSet as indexed priority queue:**
```rust
use std::collections::BTreeSet;
// BTreeSet<(cost, node)> supports O(log n) insert, delete, and min-access
// Unlike BinaryHeap, supports arbitrary removal in O(log n)
let mut pq: BTreeSet<(i32, usize)> = BTreeSet::new();
pq.insert((0, start));
// To decrease key: remove old entry, insert new
pq.remove(&(old_cost, node));
pq.insert((new_cost, node));
// Get minimum:
let &(cost, node) = pq.iter().next().unwrap();
```

### Scalability Considerations

- **Heap of size k on n items**: O(n log k) — for n = 1B and k = 100, log₂(100) ≈ 7 operations per element. Very efficient even at massive scale.
- **Two-heap median**: each insert is O(log n). For a stream of 10M events, this is 10M × ~23 operations = ~230M ops. Acceptable, but if you need percentiles across the entire distribution, a histogram or t-digest is more space-efficient.
- **Distributed top-k**: each shard maintains its own top-k heap; a coordinator merges k heaps, each of size k → O(k log k) merge. MapReduce top-k pattern.
- **Memory**: a heap of size k uses O(k) memory. For k = 10 and n = 1B items, you never hold more than 10 items in memory at once — the heap is the streaming algorithm.

### When NOT to Use a Heap

1. **You need the exact k-th element from a static array** — QuickSelect is O(n) average, beats O(n log k).
2. **k = n (you want all elements sorted)** — just sort. O(n log n) sort is faster than O(n log n) heap-sort due to cache behavior.
3. **You need range queries** — "k-th largest in subarray [l, r]" → segment tree with merge sort tree, not a heap.
4. **Decrease-key is frequent** — BinaryHeap has no O(log n) decrease-key. Use `BTreeSet` or a Fibonacci heap (theoretical only — don't implement in interviews).
5. **You need order statistics beyond just top/bottom** — "find the 37th percentile" — ordered statistics tree (e.g., augmented BST) handles arbitrary rank queries; a heap only handles rank 1 (top/bottom).

### How to Explain in Interviews

**Top K:**
> "I use a min-heap of size k to maintain the k largest elements seen so far. The heap top is the current k-th largest — the weakest element that made the cut. For each new element, if it beats the current threshold, I pop the weakest and push the new one. This runs in O(n log k) vs O(n log n) for sort — when k << n, the savings are significant."

**Two Heaps for Median:**
> "I partition elements into two halves using a max-heap for the lower half and a min-heap for the upper half. I maintain two invariants: the halves are balanced in size (differ by at most 1), and every element in the lower half is ≤ every element in the upper half. Median is either the top of the larger heap, or the average of both tops. Insert is O(log n), median is O(1)."

**Dijkstra:**
> "Dijkstra is BFS generalized to weighted edges. Instead of a FIFO queue (BFS explores level-by-level), I use a min-heap ordered by current known cost. The greedy invariant: when a node is popped from the heap, its distance is final — no future path can be shorter, assuming non-negative weights. Stale entries are skipped with a cost-vs-dist check."

## 9. 🔗 Real-World Mapping

### Where Heaps Live in Production

**Job / Task Schedulers (OS, Kubernetes, Cloud)**
The Linux kernel's completely fair scheduler (CFS) uses a red-black tree (ordered by virtual runtime) — conceptually equivalent to a priority queue. Kubernetes scheduler assigns pods to nodes using priority classes — a max-heap on pod priority. AWS SQS with message attributes supports priority-ordered processing.

**Dijkstra in Network Routing**
Every routing protocol that computes shortest paths (OSPF, IS-IS) runs Dijkstra on the network topology — a priority queue on (cost, router). CDN edge selection ("route to nearest POP") is Dijkstra at the infrastructure level.

**Streaming Top-K (Analytics Pipelines)**
Kafka Streams / Flink applications commonly maintain a top-k heap per window. "Top 100 trending hashtags in the last 5 minutes" = sliding window + min-heap of size 100 per partition + merge across partitions. Apache Flink's `TopN` operator is implemented with an exactly-this heap pattern.

**Timer Wheels / Event Schedulers**
Delayed job queues (Sidekiq, BullMQ scheduled jobs, AWS EventBridge) use a min-heap on execution timestamp. "Run this job at T+5 minutes" → push `(T+5, job_id)` to heap. The scheduler thread peeks the heap top and sleeps until that timestamp.

**Database Query Execution — Top-N Sort**
PostgreSQL's executor for `SELECT ... ORDER BY x LIMIT k` uses a heap of size k — not a full sort. This is the exact "top k from n" pattern. For k = 10 from a table with 10M rows, PostgreSQL maintains a 10-element heap, never fully sorting.

**A* Pathfinding (Games, Maps)**
Google Maps, game pathfinding (Unity NavMesh), GPS routing all use A* — Dijkstra + a heuristic. The priority queue orders nodes by `cost_so_far + heuristic_to_goal`. Same BinaryHeap pattern as Dijkstra, with an admissible heuristic (usually Manhattan or Euclidean distance) added to the priority.

**Median / Percentile Tracking (Observability)**
p99 latency tracking at high throughput. Two-heap median or a histogram-based approximation (HDRHistogram, t-digest) is what Grafana/Prometheus use for percentile computation. HDRHistogram is a bucket-based approximation — the exact two-heap approach doesn't scale to millions of data points per second, but the insight is identical.

**Event-Driven Simulations**
Discrete event simulation (network simulation, game tick systems, financial market simulation): events are processed in timestamp order. A min-heap on event timestamp is the simulation engine. Generating a new event = push to heap. Processing the next event = pop from heap.

## 10. ⚡ 10-Second Recall Cheat Sheet

```
TRIGGER → PATTERN
─────────────────────────────────────────────────────
"k largest elements"              → min-heap of size k (evict minimum)
"k smallest elements"             → max-heap of size k (evict maximum)
"k-th largest"                    → min-heap size k → peek top
"k most frequent"                 → freq map → min-heap size k on (freq, val)
"k closest points"                → max-heap size k on distance (evict farthest)
"Median from stream"              → two heaps: max-heap (lower) + min-heap (upper)
"Merge k sorted lists/arrays"     → min-heap of (value, list_idx, elem_idx)
"Shortest path weighted graph"    → Dijkstra: min-heap of (cost, node)
"Process cheapest/smallest first" → min-heap
"Process most urgent/largest first"→ max-heap
"Connect ropes minimum cost"      → min-heap, always merge two smallest
─────────────────────────────────────────────────────

CORE RUST IDIOMS
─────────────────────────────────────────────────────
Max-heap:       BinaryHeap<T>                    // default
Min-heap:       BinaryHeap<Reverse<T>>
Push:           heap.push(val) / heap.push(Reverse(val))
Pop max:        heap.pop()                       // Option<T>
Pop min:        let Reverse(v) = heap.pop().unwrap()
Peek:           heap.peek()                      // Option<&T>
Build O(n):     BinaryHeap::from(vec)            // NOT one-by-one push
Custom order:   impl Ord on struct, use BinaryHeap<MyStruct>
Tuple order:    tuples compare lexicographically — first field dominates
─────────────────────────────────────────────────────

TOP K DECISION TABLE
─────────────────────────────────────────────────────
k << n, streaming:    min-heap size k      O(n log k)    ← default choice
k << n, static:       QuickSelect          O(n) avg      ← optimal but complex
k ≈ n:                sort                 O(n log n)    ← simpler
bounded integer vals: counting/bucket sort O(n)          ← best but narrow use
─────────────────────────────────────────────────────

TWO-HEAP MEDIAN INVARIANTS
─────────────────────────────────────────────────────
lower (max-heap): holds smaller half   → top = max of lower half
upper (min-heap): holds larger half    → top = min of upper half
Invariant 1: lower.max <= upper.min    (value ordering)
Invariant 2: |lower.len - upper.len| <= 1 (size balance)
Median: lower.top() if lower is bigger, else avg(lower.top, upper.top)
─────────────────────────────────────────────────────

COMPLEXITY
─────────────────────────────────────────────────────
Insert:         O(log n)
Get min/max:    O(1)          ← peek only
Remove min/max: O(log n)
Remove arbitrary: O(n)        ← use lazy deletion instead
Build heap:     O(n)          ← use from(), not repeated push
Dijkstra:       O((V+E) log V)
─────────────────────────────────────────────────────

COMMON MISTAKES
─────────────────────────────────────────────────────
✗ Max-heap for "k largest" (need min-heap — top = eviction threshold)
✗ Forgetting stale entry check in Dijkstra (cost > dist[node] → continue)
✗ Building heap with repeated push instead of BinaryHeap::from() (O(n log n) vs O(n))
✗ Median: fixing size before value invariant (wrong element in wrong heap)
✗ Integer overflow: dist[i32::MAX] + weight overflows — use i64 or guard
✗ k=0 or k>n not handled at function boundary
✗ Tuple ordering surprise: (cost, node) — node breaks ties, may cause unexpected ordering
```

> **One-liner to burn in:**
> *A heap is a partial-order structure — O(1) access to the extreme, O(log n) insert/remove — use it whenever full sort order is unnecessary and you only need the current best.*
