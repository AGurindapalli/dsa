# Stack

> **Tier:** TIER 1 — CORE PATTERNS (Stack Basics) → TIER 3 (Monotonic Stack)
> **Prerequisites:** Arrays, Loops, Basic Recursion
> **Interview frequency:** Very High — appears directly in ~30% of problems; underpins DFS, expression evaluation, backtracking, and monotonic patterns

---

## 1. 🧠 Intuition First (No Jargon)

### First Principles

A stack is a **deferred decision machine**. When you encounter something you can't fully process yet — because you need more context from the future — you push it. When future context arrives and resolves a past element, you pop and act.

The core constraint: **Last In, First Out (LIFO)**. The thing you most recently deferred is the first thing resolved. This matches the structure of any problem where *later elements* answer questions about *earlier elements*.

### Real-World Analogy

A stack of plates in a cafeteria. You always take from the top (most recently placed). You never dig to the bottom mid-service.

More precise analogy: **undo history in a text editor**. You press Ctrl+Z — it undoes the most recent action, not the first. That's a stack. The most recently pushed change is the first one undone.

Call stack in every language runtime is a literal stack — function calls push frames, returns pop them. When you write recursive code, you're using the CPU's stack. When you convert recursion to iteration, you're managing your own explicit stack.

### Why This Pattern Exists

Many problems have a **nested or ordered dependency structure**: you need to resolve the innermost/most-recent thing before you can resolve the outer one. Parentheses matching, function call depth, "next greater element" — all share this shape. A stack is the minimal data structure that captures this "process in reverse order of arrival" semantics.

## 2. 🔍 Pattern Recognition

### When to Use a Stack

| Situation | Stack usage |
|---|---|
| Matching / balancing pairs | Push opens, pop+validate on close |
| "Next greater / smaller element" | Monotonic stack |
| Evaluate expressions | Operand stack + operator stack |
| Simulate recursion iteratively | Explicit DFS stack |
| Undo / redo operations | Two stacks |
| Nested structures (HTML, JSON) | Push on open, pop on close |
| "Previous greater / smaller element" | Monotonic stack (same, different direction) |
| Backtracking state management | Stack of states |

### Input/Output Signals

**Problem likely needs a stack when:**
- You see brackets: `()`, `[]`, `{}` — push/pop matching
- "Next greater element", "next smaller", "previous greater" → monotonic
- "Valid expression", "evaluate postfix/prefix" → operator stack
- The answer for element `i` depends on elements seen before `i` that haven't been resolved yet
- The problem has a nested / hierarchical structure
- You need to reverse processing order
- DFS on a graph/tree without recursion

### Common Keywords

`valid parentheses`, `balanced`, `next greater`, `next smaller`, `previous larger`, `stock span`, `largest rectangle`, `trapping rain water`, `evaluate expression`, `decode string`, `nested`, `depth`, `undo`, `backtrack`, `iterative DFS`, `postfix`, `infix`

### 10-Second Identification Rule

> **If processing element `i` requires knowing the most recently unresolved element — push it. When something resolves it (a closing bracket, a larger number) — pop and handle.**

The mental test: "Is this problem about matching/pairing things in reverse order of encounter?" → Stack.

## 3. ⚙️ Core Algorithm / Template

### Pattern 1: Bracket Matching

```rust
fn is_valid(s: &str) -> bool {
    let mut stack: Vec<char> = Vec::new();
    for c in s.chars() {
        match c {
            '(' | '[' | '{' => stack.push(c),
            ')' => if stack.pop() != Some('(') { return false; },
            ']' => if stack.pop() != Some('[') { return false; },
            '}' => if stack.pop() != Some('{') { return false; },
            _   => {}
        }
    }
    stack.is_empty() // unmatched opens left over = invalid
}
```

**Mental model:** Push openers. On each closer, the top of the stack must be the matching opener. If pop returns None (empty stack) or wrong opener → invalid. If stack non-empty at end → unclosed openers.

---

### Pattern 2: Monotonic Stack — Next Greater Element

```rust
/// Returns array where result[i] = next element to the right greater than nums[i].
/// -1 if none exists.
fn next_greater_element(nums: &[i32]) -> Vec<i32> {
    let n = nums.len();
    let mut result = vec![-1i32; n];
    let mut stack: Vec<usize> = Vec::new(); // indices of unresolved elements

    for i in 0..n {
        // nums[i] is the candidate "greater element" for everything on the stack smaller than it
        while let Some(&top) = stack.last() {
            if nums[i] > nums[top] {
                stack.pop();
                result[top] = nums[i]; // nums[i] is the answer for index `top`
            } else {
                break;
            }
        }
        stack.push(i);
    }
    // remaining indices in stack have no greater element → result stays -1
    result
}
```

**Mental model:** Stack holds indices of elements waiting for their "next greater". When you encounter a new element, pop everything smaller (they've found their answer). Push current index. At end, anything left in stack has no next greater.

The stack is always **monotonically decreasing** (from bottom to top) by value — that's the invariant.

### Pattern 3: Monotonic Stack — Previous Greater Element

```rust
fn previous_greater_element(nums: &[i32]) -> Vec<i32> {
    let n = nums.len();
    let mut result = vec![-1i32; n];
    let mut stack: Vec<usize> = Vec::new();

    for i in 0..n {
        // pop elements that are <= nums[i] (they can't be previous greater for i)
        while let Some(&top) = stack.last() {
            if nums[top] <= nums[i] {
                stack.pop();
            } else {
                break;
            }
        }
        // stack top is now the previous greater (if any)
        if let Some(&top) = stack.last() {
            result[i] = nums[top];
        }
        stack.push(i);
    }
    result
}
```

**Mental model:** Same stack, but you resolve the current element's answer from the stack top before pushing — not after. You're asking "what's already on the stack that's greater than me?"

### Pattern 4: Stack-Based Expression Evaluation (Postfix / RPN)

```rust
fn eval_rpn(tokens: &[&str]) -> i32 {
    let mut stack: Vec<i32> = Vec::new();
    for &token in tokens {
        match token {
            "+" | "-" | "*" | "/" => {
                let b = stack.pop().unwrap(); // second operand
                let a = stack.pop().unwrap(); // first operand
                let result = match token {
                    "+" => a + b,
                    "-" => a - b,
                    "*" => a * b,
                    "/" => a / b, // truncates toward zero in Rust (matches LeetCode spec)
                    _   => unreachable!(),
                };
                stack.push(result);
            }
            num => stack.push(num.parse::<i32>().unwrap()),
        }
    }
    stack.pop().unwrap() // single remaining value is the result
}
```

**Mental model:** Numbers go on the stack. Operators pop two, compute, push result. At the end, exactly one value remains.

---

### Pattern 5: Iterative DFS using Stack

```rust
fn dfs_iterative(graph: &Vec<Vec<usize>>, start: usize) -> Vec<usize> {
    let n = graph.len();
    let mut visited = vec![false; n];
    let mut stack = vec![start];
    let mut order = Vec::new();

    while let Some(node) = stack.pop() {
        if visited[node] { continue; }
        visited[node] = true;
        order.push(node);
        // push neighbors in reverse so leftmost is processed first
        for &next in graph[node].iter().rev() {
            if !visited[next] {
                stack.push(next);
            }
        }
    }
    order
}
```

**Mental model:** The call stack in recursive DFS becomes an explicit `Vec`. Push the start node. Pop → process → push unvisited neighbors. The `if visited { continue }` guard handles multiple pushes of the same node.

### Pattern 6: Min Stack (O(1) min access)

```rust
struct MinStack {
    stack: Vec<i32>,
    min_stack: Vec<i32>, // parallel stack tracking running minimums
}

impl MinStack {
    fn new() -> Self {
        Self { stack: Vec::new(), min_stack: Vec::new() }
    }

    fn push(&mut self, val: i32) {
        self.stack.push(val);
        let current_min = self.min_stack.last().copied().unwrap_or(i32::MAX);
        self.min_stack.push(val.min(current_min));
    }

    fn pop(&mut self) {
        self.stack.pop();
        self.min_stack.pop();
    }

    fn top(&self) -> i32 { *self.stack.last().unwrap() }
    fn get_min(&self) -> i32 { *self.min_stack.last().unwrap() }
}
```

**Mental model:** Two parallel stacks. `min_stack[i]` = minimum of all elements from bottom up to index `i`. Push/pop both in sync. `get_min` is just a peek at `min_stack` top.

## 4. 📊 Complexity Deep Dive

| Operation | Time | Space |
|---|---|---|
| Push | O(1) amortized | — |
| Pop | O(1) | — |
| Peek / last | O(1) | — |
| Bracket matching | O(n) | O(n) |
| Monotonic stack | O(n) total | O(n) |
| Expression eval | O(n) | O(n) |

### Why Monotonic Stack is O(n), Not O(n²)

The inner while loop looks like it could make the total O(n²). It's not — use amortized analysis:
- Each element is pushed **at most once** and popped **at most once**.
- Total push operations across the entire loop: n.
- Total pop operations across the entire loop: at most n.
- Total work: 2n = **O(n)**.

This is the same argument as for dynamic array amortized push. The inner loop burns through a backlog that was built up, not creating new work per iteration.

### Space

The stack holds at most n elements. In the worst case (strictly decreasing input for "next greater"), every element is pushed and none are popped until the end → O(n) stack space.

### When This Approach Breaks

1. **You need random access** — stacks give O(n) access to non-top elements. Use an array or deque.
2. **You need both ends** — use a `VecDeque` (double-ended queue / deque).
3. **Memory-constrained and n is huge** — an explicit stack of size n may be unacceptable. Recursive DFS uses the call stack (often limited to ~8MB on Linux), so for very deep graphs, iterative + explicit stack is safer.
4. **Parallel processing** — a single stack is inherently sequential. For concurrent event processing, you need a different model.

## 5. ⚠️ Edge Cases & Pitfalls

### Senior-Level Mistakes

**1. Not checking empty stack before pop**
```rust
// Wrong: panics if stack is empty
let top = stack.pop().unwrap();

// Right: handle the empty case
match stack.pop() {
    Some(val) => { /* process */ }
    None => return false, // or handle invalid state
}
```

**2. Bracket matching — forgetting to check stack is empty at end**
```rust
// If you only check mismatches during iteration,
// "((" passes incorrectly — unclosed openers remain
// ALWAYS check: stack.is_empty() as the final return
```

**3. Monotonic stack — storing values vs. indices**

Storing values is simpler but loses positional information. If the problem asks for the **index** of the next greater element, or for the **width** of a span, you need indices. Always default to storing indices unless the problem is trivially value-only.

```rust
// Wrong for span/distance problems:
let mut stack: Vec<i32> = Vec::new(); // values

// Right:
let mut stack: Vec<usize> = Vec::new(); // indices — derive values via nums[idx]
```

**4. RPN evaluation — operand order matters**
```rust
// For non-commutative ops (-, /), pop order is critical
let b = stack.pop(); // second operand (rightmost in expression)
let a = stack.pop(); // first operand (leftmost in expression)
// a - b, NOT b - a
```

**5. Circular array problems — forgetting to run two passes**

For "next greater in circular array", iterate `2n` times with `i % n` indexing. Only push indices from the first pass (`i < n`). The second pass resolves remaining elements.

**6. usize underflow when computing spans/widths**
```rust
// width = right - left - 1
// If left is an index from the stack and right is current index,
// both are usize — if left > right somehow, you get a panic
// Use i32 indices for width calculations to be safe, or check bounds
let width = (right as i32 - left as i32 - 1) as usize;
```

### Debugging Heuristics

- Getting wrong answers on monotonic stack? → Check if you're storing indices vs. values
- Bracket validation returning wrong result? → Check the `stack.is_empty()` final condition
- RPN giving wrong result on subtraction/division? → Verify pop order (`b` then `a`)
- Monotonic stack infinite loop? → The `break` in the else branch is mandatory — don't forget it
- Stack overflow on recursive DFS? → Convert to iterative with explicit stack

## 6. 🔁 Variations (Pattern Stretching)

### Easy
- **Valid Parentheses** — push openers, pop+validate on closers
- **Implement Stack using Queues** — understand the underlying mechanics
- **Baseball Game** — simulate operations on a running score stack

### Medium
- **Min Stack** — parallel min-tracking stack
- **Daily Temperatures** — next greater element returning day distance (indices)
- **Evaluate Reverse Polish Notation** — postfix expression evaluation
- **Decode String** — `3[a2[bc]]` → `abcbcabcbcabcbc`, nested repeat via stack
- **Remove K Digits** — monotonic stack to build smallest number
- **Asteroid Collision** — stack simulation with direction-based collision rules

### Medium-Hard
- **Largest Rectangle in Histogram** — monotonic stack tracking "left boundary" for each bar; pop when a shorter bar invalidates taller ones
- **132 Pattern** — monotonic stack + running minimum; tricky because you're tracking a three-element relationship
- **Online Stock Span** — monotonic stack where each pop accumulates span count

### Hard
- **Trapping Rain Water** — can be solved with monotonic stack (pop when taller bar found, compute trapped water at each pop)
- **Maximal Rectangle** (in binary matrix) — reduce each row to histogram problem, apply largest rectangle in histogram
- **Basic Calculator** (full infix with +, -, *, /, parentheses) — operator precedence stack + operand stack
- **Remove Duplicate Letters** — monotonic stack with character frequency tracking for lexicographically smallest result

### How the Pattern Evolves

```
Base: push/pop for deferred resolution
↓
Bracket matching: push openers, validate against closers
↓
Monotonic stack: maintain ordering invariant during push/pop
↓
Stack + extra state: track counts, spans, min alongside the stack
↓
Two stacks: undo/redo, infix evaluation with precedence
↓
Stack as DFS engine: explicit recursion stack for tree/graph traversal
↓
Monotonic stack + circular array: double-pass, index wrapping
↓
Monotonic stack + other data structures: segment tree for range min queries
```

## 7. 🧪 Interview-Level Problems

### Problem 1 — Medium: Daily Temperatures
**LeetCode 739**

**Summary:** Given array of temperatures, return array where `result[i]` = number of days until a warmer temperature. 0 if no warmer day exists.

**Expected approach:**
- Monotonic stack storing **indices** (not values)
- For each day `i`, pop stack while `temperatures[i] > temperatures[stack.top()]`
- `result[popped_idx] = i - popped_idx` (distance in days)
- Push current index

**Key insight:** Storing indices (not values) is mandatory here — you need the distance, not the temperature itself. This is the canonical "next greater element with distance" variant. The stack holds indices of days still waiting for a warmer day.

---

### Problem 2 — Medium-Hard: Largest Rectangle in Histogram
**LeetCode 84**

**Summary:** Given array of bar heights, find the largest rectangle that fits within the histogram.

**Expected approach:**
- Monotonic increasing stack of indices
- When current bar is shorter than stack top, pop and compute rectangle area:
  - Height = `heights[popped]`
  - Width = `i - stack.top() - 1` (or `i` if stack is now empty)
- After loop, drain remaining stack with `i = n` as right boundary
- Track maximum area seen

**Key insight:** Each popped bar is the **shortest bar in a rectangle that extends from the current left boundary to just before the current bar**. The width calculation `i - stack.top() - 1` uses the new stack top as the left sentinel — the element below the popped index is the last bar shorter than the popped bar (left boundary). Appending a sentinel `0` at both ends simplifies boundary handling.

---

### Problem 3 — Hard: Basic Calculator
**LeetCode 224**

**Summary:** Evaluate a string expression containing `+`, `-`, `(`, `)`, spaces, and non-negative integers. No `*` or `/`.

**Expected approach:**
- Single stack storing `(current_result, sign_before_parenthesis)`
- Maintain running `result` and `sign` (+1 or -1)
- On `(`: push `(result, sign)` onto stack, reset `result = 0`, `sign = 1`
- On `)`: `result = stack.pop_sign * result + stack.pop_result`
- On digit: parse full number (multi-digit), apply `sign * number` to result
- On `+`/`-`: update sign

**Key insight:** Parentheses create a nested scope. The stack saves the "outer" result and the sign that was applied to the parenthesized sub-expression. On `)`, you collapse the inner result back into the outer context. The insight is recognizing that you don't need a separate operand and operator stack — a single stack of `(result, sign)` tuples handles the nesting cleanly.

## 8. 🧠 Staff-Level Thinking

### Trade-offs Between Approaches

| Approach | Use when |
|---|---|
| Recursive DFS | Graph/tree depth < ~10K nodes; cleaner code |
| Iterative stack DFS | Deep graphs, risk of stack overflow; production code |
| Monotonic stack | "Next/previous greater/smaller" — always O(n) |
| Two-pointer | When problem has sorted input and no deferred resolution needed |
| Segment tree | When you need range queries beyond just next/previous element |

**Monotonic stack vs. brute force for "next greater":**
- Brute force: O(n²) — for each element, scan right until greater found
- Monotonic stack: O(n) — each element pushed/popped at most once
- Segment tree / sparse table: O(n log n) build, O(log n) query — overkill unless you need dynamic updates or range queries

### When NOT to Use a Stack

1. **FIFO semantics needed** — if you're processing in arrival order, you need a queue.
2. **Random access needed** — stack only exposes the top. Use an array/deque.
3. **Problem has sorted input and no "deferred" relationship** — two pointers or binary search is likely cleaner.
4. **Parallel/concurrent processing** — `Vec` as a stack is not thread-safe. Use `Arc<Mutex<Vec<T>>>` or a lock-free stack if needed.
5. **The deferred relationship is non-local** — if element at `i` depends on element at `i+100` specifically (not "the next greater"), a stack doesn't model this well. Consider DP or binary search.

### Scalability Considerations

- **Call stack depth**: Rust's default stack size is 8MB. Recursive DFS on a graph with 10^6 nodes will stack overflow. Always use iterative + explicit stack for production graph traversal.
- **Stack as audit log**: In event-driven systems, a stack-like structure (event log with LIFO replay) is the pattern behind database transaction rollback and saga compensation.
- **Memory**: Each stack frame in recursive DFS holds local variables. An explicit stack of indices is far more memory-efficient.

### How to Explain in Interviews

**Frame it as deferred decision:**
> "I'm using a stack because I can't resolve element `i` until I see what comes after it. The stack lets me defer those elements until a future element provides the answer."

**For monotonic stack specifically:**
> "I maintain a monotonically [increasing/decreasing] stack. When I encounter an element that breaks the invariant, I pop — and each pop is an O(1) resolution of a previously deferred answer. Since each element is pushed and popped at most once, the total work is O(n) despite the nested loop."

**Anticipate the follow-up: "Can you do this without extra space?"**
> "For some variants like trapping rain water, yes — two-pointer approach achieves O(1) space at the same O(n) time. The trade-off is code complexity; the stack approach is cleaner and more generalizable. I'd reach for two-pointer only if the interviewer explicitly constrains space."

## 9. 🔗 Real-World Mapping

### Where Stacks Live in Production

**Call Stack / Execution Engine**
The literal stack in every language runtime. Recursive function calls = push frame. Return = pop. Stack overflow = stack too deep. Every developer uses this daily without thinking about it.

**Undo/Redo in Editors and Document Systems**
```
undo_stack: Vec<Operation>
redo_stack: Vec<Operation>
// Ctrl+Z: pop from undo, push to redo
// Ctrl+Y: pop from redo, push to undo
```
Google Docs, VS Code, database transaction rollback — all stacks.

**Browser Back/Forward Navigation**
```
back_stack:    Vec<Url>
forward_stack: Vec<Url>
// Navigate: push to back, clear forward
// Back:     pop back, push to forward
```

**Expression Parsing / Compilers / Calculators**
Compiler's parser maintains a stack for precedence climbing and reduction. Every time you write `3 * (4 + 5)`, a parser stack is resolving your parentheses exactly like the bracket-matching template above.

**Database Transaction Rollback (Savepoints)**
```
// SAVEPOINT A → push state
// DML operations...
// ROLLBACK TO SAVEPOINT A → pop to state A
```
PostgreSQL savepoints are literally stack-based. Nested transactions push/pop savepoint frames.

**Kubernetes Reconciliation Loop / Event Sourcing**
Not a direct stack, but event sourcing's "replay from beginning" is a stack-like audit trail. Saga pattern compensation (rollback sequence is exactly reversed = LIFO) is a production stack pattern.

**Stock Span Problem → Trading Systems**
The "stock span" pattern (how many consecutive days was price ≤ today's price?) is a direct monotonic stack application. Real-time trading dashboards compute rolling span metrics this way.

**HTML/XML/JSON Parsers**
Every streaming parser for nested formats uses a stack. Push on opening tag/brace, pop+validate on closing. The browser's DOM parser, `serde_json`'s streaming mode, XML SAX parsers — all stacks.

## 10. ⚡ 10-Second Recall Cheat Sheet

```
TRIGGER → PATTERN
─────────────────────────────────────────────────────
Brackets / matching pairs           → push openers, pop+validate on closer, check empty at end
Next greater element (right)        → monotonic decreasing stack of indices
Previous greater element (left)     → same stack, check top before pushing
Next smaller element                → monotonic increasing stack (pop when smaller found)
"Stock span" / counting run length  → monotonic stack accumulating counts on pop
Largest rectangle in histogram      → monotonic increasing stack, compute area on pop
Trapping rain water                 → monotonic stack OR two pointers
Expression evaluation (postfix)     → operand stack, pop two on operator
Iterative DFS                       → explicit Vec<node> stack
Undo/Redo                           → two stacks
Min/Max in O(1) alongside push/pop  → parallel min/max stack
─────────────────────────────────────────────────────

CORE RUST IDIOMS
─────────────────────────────────────────────────────
Push:           stack.push(val)
Pop:            stack.pop()           // Option<T>
Peek:           stack.last()          // Option<&T>
Pop+unwrap:     stack.pop().unwrap()  // only when certain non-empty
Check empty:    stack.is_empty()
Monotonic pop:  while let Some(&top) = stack.last() { if cond { stack.pop(); } else { break; } }
─────────────────────────────────────────────────────

MONOTONIC STACK DECISION TABLE
─────────────────────────────────────────────────────
Next GREATER  (right side):  decreasing stack — pop when nums[i] > nums[top]
Next SMALLER  (right side):  increasing stack — pop when nums[i] < nums[top]
Prev GREATER  (left side):   decreasing stack — check top BEFORE pushing
Prev SMALLER  (left side):   increasing stack — check top BEFORE pushing
Circular:                    run 2× passes with i % n
─────────────────────────────────────────────────────

COMPLEXITY
─────────────────────────────────────────────────────
All stack ops:          O(1) per operation
Bracket match:          O(n) time, O(n) space
Monotonic stack:        O(n) time total (amortized), O(n) space
Expression eval:        O(n) time, O(n) space
─────────────────────────────────────────────────────

COMMON MISTAKES
─────────────────────────────────────────────────────
✗ Storing values instead of indices in monotonic stack
✗ Missing stack.is_empty() check at end of bracket validation
✗ Wrong pop order in RPN: pop b first, then a (a op b)
✗ Forgetting the stack drain after the main loop (monotonic)
✗ Using recursion for deep DFS in production (stack overflow risk)
```
