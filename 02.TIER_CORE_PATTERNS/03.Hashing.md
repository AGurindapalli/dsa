# Hashing (Maps, Sets, Frequency Count)

> **Tier:** TIER 1 — CORE PATTERNS
> **Prerequisites:** Arrays, Basic Loops, Time/Space Complexity
> **Interview frequency:** Extremely High — appears in 60–70% of all interview problems either directly or as a supporting pattern

---

## 1. 🧠 Intuition First (No Jargon)

### First Principles

A hash map is a **lookup table that trades memory for speed**. You're pre-computing the answer to "where is this?" so you never search linearly again.

The core idea: instead of scanning an array O(n) to find if a value exists, you **index it** into a bucket by running a hash function over it. Now every lookup, insert, and delete is O(1) average.

### Real-World Analogy

A library with no catalog vs. a library with a card catalog:
- Without hashing: walk every shelf to find a book → O(n)
- With hashing: look up the card → get the shelf number → walk straight to it → O(1)

The catalog (hash map) costs memory to build and maintain. But once built, every lookup is instant.

### Why This Pattern Exists

Arrays give O(1) **positional** access. But most real problems aren't positional — they're **relational**: "have I seen this before?", "how many times did X appear?", "what pairs sum to target?". Hash structures convert relational queries into positional ones by mapping arbitrary keys to integer indices.

## 2. 🔍 Pattern Recognition

### When to Use Hashing

| Situation | Hashing solves it |
|---|---|
| Count frequency of elements | `HashMap<T, usize>` |
| Detect duplicates in O(n) | `HashSet<T>` |
| Find complement pairs (Two Sum) | `HashMap<val, index>` |
| Group elements by a key | `HashMap<key, Vec<T>>` |
| Memoization / caching | `HashMap<state, result>` |
| Sliding window with constraints | `HashMap<T, count>` as window state |
| Check if two strings are anagrams | `HashMap<char, i32>` diff |

### Input/Output Signals

**Problem likely needs hashing when you see:**
- "Find if any two elements sum to X" → complement map
- "Find duplicates" → set
- "Count occurrences" → frequency map
- "Group by / categorize" → map of lists
- "Longest subarray with property" → map stores state + index
- "Check if array A is a permutation of B" → frequency diff
- "Two strings are anagrams" → char frequency map

### Common Keywords

`duplicate`, `frequency`, `count`, `pair`, `sum equals`, `group`, `categorize`, `seen before`, `already exists`, `appeared more than once`, `distinct elements`, `anagram`, `permutation`, `intersection`, `complement`

### 10-Second Identification Rule

> **If the problem asks you to find something that requires knowing what you've already seen — reach for a hash map.**

The mental shortcut: array gives you "what's at position i". Hash map gives you "where is value X". When the question is about values, not positions → hash.

## 3. ⚙️ Core Algorithm / Template

### Pattern 1: Frequency Counter

```rust
use std::collections::HashMap;

fn frequency_count(nums: &[i32]) -> HashMap<i32, usize> {
    let mut freq: HashMap<i32, usize> = HashMap::new();
    for &n in nums {
        *freq.entry(n).or_insert(0) += 1;
    }
    freq
}
```

**Mental model:** `entry().or_insert(0)` is the single most important Rust hashing idiom. It atomically gets-or-creates. No separate `.contains_key()` + `.insert()` dance needed.

---

### Pattern 2: Complement Map (Two Sum archetype)

```rust
fn two_sum(nums: &[i32], target: i32) -> Option<(usize, usize)> {
    let mut seen: HashMap<i32, usize> = HashMap::new(); // value → index
    for (i, &n) in nums.iter().enumerate() {
        let complement = target - n;
        if let Some(&j) = seen.get(&complement) {
            return Some((j, i));
        }
        seen.insert(n, i);
    }
    None
}
```

**Mental model:** For every element, ask "does its complement already exist?" before inserting. The map stores what you've seen so far.

---

### Pattern 3: Existence Check (HashSet)

```rust
use std::collections::HashSet;

fn contains_duplicate(nums: &[i32]) -> bool {
    let mut seen: HashSet<i32> = HashSet::new();
    for &n in nums {
        if !seen.insert(n) { // insert returns false if already present
            return true;
        }
    }
    false
}
```

**Mental model:** `HashSet::insert` returns `false` if the value already existed. Use this — it's a single O(1) operation instead of contains + insert.

---

### Pattern 4: Grouping by Key

```rust
fn group_anagrams(words: Vec<String>) -> Vec<Vec<String>> {
    let mut groups: HashMap<Vec<char>, Vec<String>> = HashMap::new();
    for word in words {
        let mut key: Vec<char> = word.chars().collect();
        key.sort_unstable();                             // canonical form = sorted chars
        groups.entry(key).or_default().push(word);
    }
    groups.into_values().collect()
}
```

**Mental model:** Normalize each element into a canonical key, then group. The key is "what makes two elements equivalent?" — that becomes the hash key.

---

### Pattern 5: Prefix Sum + Hash Map (Subarray with target sum)

```rust
fn subarray_sum_equals_k(nums: &[i32], k: i32) -> i32 {
    let mut prefix_count: HashMap<i32, i32> = HashMap::new();
    prefix_count.insert(0, 1); // empty prefix
    let mut prefix_sum = 0;
    let mut count = 0;
    for &n in nums {
        prefix_sum += n;
        // if (prefix_sum - k) exists as a previous prefix, we found a valid subarray
        count += prefix_count.get(&(prefix_sum - k)).copied().unwrap_or(0);
        *prefix_count.entry(prefix_sum).or_insert(0) += 1;
    }
    count
}
```

**Mental model:** Store "prefix sum → how many times have I seen this sum". For each position, check if `current_prefix - k` was seen before. If yes, those subarrays sum to k.

---

### Pattern 6: Sliding Window + Hash Map

```rust
fn length_of_longest_substring(s: &str) -> usize {
    let mut last_seen: HashMap<char, usize> = HashMap::new();
    let mut max_len = 0;
    let mut left = 0;
    for (right, c) in s.chars().enumerate() {
        if let Some(&prev) = last_seen.get(&c) {
            left = left.max(prev + 1); // shrink window past the duplicate
        }
        last_seen.insert(c, right);
        max_len = max_len.max(right - left + 1);
    }
    max_len
}
```

**Mental model:** Map tracks the last position of each character. On collision, jump `left` to just after the collision. Map acts as window state.

## 4. 📊 Complexity Deep Dive

| Operation | Average | Worst Case |
|---|---|---|
| Insert | O(1) | O(n) — hash collision chain |
| Lookup | O(1) | O(n) — hash collision chain |
| Delete | O(1) | O(n) |
| Space | O(n) | O(n) |

### When Average ≠ Worst

Worst case O(n) happens on **hash collision attacks** — crafted inputs that all hash to the same bucket, degrading to a linked list scan. Rust's `HashMap` uses **SipHash** (cryptographic, DoS-resistant) by default. In competitive programming, people swap to `FxHashMap` for speed, but in interviews Rust's default is fine.

### Trade-offs

| Approach | Time | Space | Notes |
|---|---|---|---|
| Brute force (nested loops) | O(n²) | O(1) | No extra memory, but slow |
| Sorting | O(n log n) | O(1)–O(n) | Good when order matters |
| Hashing | O(n) | O(n) | Best time; extra memory cost |

### When Hashing Breaks

1. **Memory-constrained environments** — O(n) extra space isn't free. For very large n (billions of entries), consider Bloom filters or count-min sketch.
2. **Ordered queries** — "find the k-th largest", "range queries". Use `BTreeMap` (O(log n) per op) instead.
3. **Deterministic hash needed** — HashMap iteration order in Rust is **random** (intentionally). If you need sorted output, collect keys and sort, or use `BTreeMap`.
4. **Floating point keys** — floats don't implement `Eq` + `Hash` in Rust (correctly — NaN ≠ NaN). You can't use `f64` as a HashMap key directly.

## 5. ⚠️ Edge Cases & Pitfalls

### Senior-Level Mistakes

**1. Using sorted map when unsorted suffices**
```rust
// Wrong: O(n log n) for no reason
let map: BTreeMap<i32, usize> = BTreeMap::new();

// Right: O(n) when order doesn't matter
let map: HashMap<i32, usize> = HashMap::new();
```

**2. Forgetting the empty prefix in prefix sum problems**
```rust
// ALWAYS seed the map with (0, 1) before the loop
prefix_count.insert(0, 1);
// Missing this breaks subarrays that start from index 0
```

**3. Off-by-one in sliding window shrink**
```rust
// Wrong: doesn't account for left already being past the collision
left = prev + 1;

// Right: left can only move forward, never backward
left = left.max(prev + 1);
```

**4. Mutating while iterating**
Rust won't let you do this — the borrow checker saves you. But the conceptual mistake is trying to modify a map you're reading from in the same pass. Split into two passes or use indices.

**5. HashMap key type gotchas in Rust**
```rust
// Vec<char> works as a key (implements Hash + Eq)
// [char; N] works as a key
// &str works as a key
// f64 DOES NOT implement Hash — use ordered_float crate or scale to integer
```

**6. Integer overflow in prefix sums**
```rust
// Use i64 for prefix sums when array values are large
let mut prefix_sum: i64 = 0;
```

### Debugging Heuristics

- Off-by-one in count? → Check if the empty prefix `(0, 1)` is seeded.
- Missing some pairs? → Verify you're checking before inserting (Two Sum pattern).
- Wrong group key? → Print the canonical form — is normalization correct?
- Window not shrinking? → Is `left = left.max(prev + 1)` used, not `left = prev + 1`?

## 6. 🔁 Variations (Pattern Stretching)

### Easy
- **Contains Duplicate** — plain `HashSet`, one pass
- **Two Sum** — complement map, single pass
- **Valid Anagram** — char frequency diff, two maps or one with +1/-1
- **Intersection of Two Arrays** — two HashSets, intersection

### Medium
- **Group Anagrams** — normalize to sorted key, `HashMap<Vec<char>, Vec<String>>`
- **Longest Substring Without Repeating Characters** — sliding window + last-seen map
- **Subarray Sum Equals K** — prefix sum + frequency map
- **Top K Frequent Elements** — frequency map + heap or bucket sort
- **4Sum II** — two-pass complement: hash sums of first pair, check against sums of second pair

### Medium-Hard
- **Longest Consecutive Sequence** — `HashSet` membership check, build chains from left endpoints only (O(n))
- **Minimum Window Substring** — sliding window with two frequency maps, shrink from left
- **Subarray with equal 0s and 1s** — remap 0→-1, then prefix sum + hash (becomes subarray sum = 0)
- **Contiguous Array** — prefix sum trick; map `prefix → first_index` for max length

### Hard
- **Subarrays with K Different Integers** — `exactly(k) = atMost(k) - atMost(k-1)`, two sliding windows each with a frequency map
- **Number of Distinct Substrings** — rolling hash (Rabin-Karp) to deduplicate substrings in O(n)
- **All O(1) Data Structure** — `HashMap` + doubly-linked list for O(1) getMaxKey/getMinKey

### How the Pattern Evolves

```
Base: single pass frequency count
↓
Add complement: "does the thing I need already exist?"
↓
Add prefix: "what state existed at an earlier position?"
↓
Add sliding window: "maintain a valid window using map as state"
↓
Add multiple maps: "track two constraints simultaneously"
↓
Rolling hash: "hash substrings for O(n) substring deduplication"
```

## 7. 🧪 Interview-Level Problems

### Problem 1 — Medium: Subarray Sum Equals K
**LeetCode 560**

**Summary:** Given integer array `nums` and integer `k`, return the count of subarrays whose sum equals `k`. Array may contain negatives.

**Expected approach:**
- Prefix sum + frequency map
- For each index, check if `prefix_sum - k` was seen before
- Seed with `(0, 1)` before loop

**Key insight:** You can't use sliding window here because of negatives (window is not monotonic). Prefix sum + hash is the only O(n) solution. The "empty subarray" edge case is why you seed (0,1) — without it you miss subarrays starting at index 0.

---

### Problem 2 — Medium-Hard: Longest Consecutive Sequence
**LeetCode 128**

**Summary:** Given unsorted array, find the length of the longest sequence of consecutive integers. Must be O(n).

**Expected approach:**
- Load all values into a `HashSet`
- For each number, only start a chain if `num - 1` is NOT in the set (left endpoint detection)
- From each left endpoint, walk right counting consecutive members

**Key insight:** The O(n) constraint eliminates sorting. The "only start from left endpoints" observation reduces total inner-loop work to O(n) amortized — each number is touched at most twice. Naive "start chain from every number" is O(n²).

---

### Problem 3 — Hard: Minimum Window Substring
**LeetCode 76**

**Summary:** Given string `s` and pattern `t`, find the minimum window in `s` that contains all characters of `t`.

**Expected approach:**
- Build frequency map of `t` (`need`)
- Sliding window on `s` with a current window frequency map (`have`)
- Track `formed` count (how many unique chars in `t` are satisfied in window)
- Expand right until window is valid, then contract left while still valid
- Track minimum window throughout

**Key insight:** Two pointers + two maps. The `formed` counter avoids re-scanning the entire `need` map on every step — it's an O(1) check for "is window currently valid?". Without this optimization, each step is O(|t|) instead of O(1), blowing up to O(n·|t|) total.

## 8. 🧠 Staff-Level Thinking

### Trade-offs Between Approaches

| Approach | Use when |
|---|---|
| `HashMap` | Most cases — O(n) time, O(n) space |
| `BTreeMap` | Need sorted iteration or range queries |
| `Vec` as frequency array | Keys are bounded integers (e.g., lowercase letters → `[usize; 26]`) |
| `HashSet` | Only care about existence, not count or value |
| Bit manipulation | Keys are small integers, extreme space constraint |
| Two-pass vs one-pass | Two-pass is cleaner code; one-pass is needed when you must process in order |

### When Bounded Key Space Beats HashMap

```rust
// For lowercase-only strings: O(1) space, faster constant
let mut freq = [0i32; 26];
for b in s.bytes() {
    freq[(b - b'a') as usize] += 1;
}
// No heap allocation, better cache locality, no hash function overhead
```

This beats `HashMap<char, i32>` in both space AND time for bounded alphabets. In an interview, mention this optimization — it signals systems awareness.

### Scalability Considerations

- At 10M entries: HashMap memory pressure becomes real. Consider `AHashMap` (faster hash) or capacity pre-allocation: `HashMap::with_capacity(n)`.
- At 1B entries: HashMap doesn't fit in RAM. Use probabilistic structures (Bloom filter for existence, Count-Min Sketch for frequency).
- Distributed system: consistent hashing for partitioning, not standard HashMap.

### When NOT to Use Hashing

1. **You need order** — HashMap gives no ordering guarantee. Use sorted array + binary search or `BTreeMap`.
2. **Space is critical** — if O(n) extra space is unacceptable, look for a sort-based or in-place solution.
3. **Worst-case guarantees matter** — hash tables degrade on adversarial input. For real-time systems, a `BTreeMap` O(log n) worst case may be safer than O(1) average / O(n) worst.
4. **The key is composite and expensive to hash** — hashing a large struct on every operation has real constant-factor cost. Benchmark before defaulting to hash.
5. **The problem is solvable with sorting** — sorting is cache-friendly, predictable, and often simpler. Don't reach for a map when sort + two-pointer works.

### How to Explain in Interviews

**Frame it as a space-time trade-off:**
> "I'm trading O(n) extra memory to eliminate a repeated linear search. By storing what I've seen in a hash map, each lookup becomes O(1) instead of O(n), bringing the total from O(n²) down to O(n)."

**Mention the key design decision:**
> "The critical insight is what to store as the key vs. value. Here I'm storing [value → index / value → count / canonical_form → list], because that's the query I need to answer in O(1)."

**Anticipate the follow-up:**
> "If memory were constrained, I'd consider [sort + two pointer / bit array / sliding window without a map], accepting a higher time complexity in exchange for O(1) or O(log n) space."

## 9. 🔗 Real-World Mapping

### Where This Pattern Lives in Production

**API Rate Limiting**
```
HashMap<user_id, RateState> — O(1) per request check
```
Sliding window with a frequency map is exactly the "sliding window rate limiter" algorithm. Same pattern, production scale.

**Session Store / Auth Cache**
```
HashMap<session_token, UserContext> — O(1) auth validation
```
Every session store is a hash map. Redis is a distributed hash map.

**Deduplication Pipelines**
```
HashSet<event_id> — seen-before check for idempotent event processing
```
Kafka consumer deduplication: maintain a seen-set (or Bloom filter at scale) keyed on idempotency key.

**Database Query Planner — Hash Join**
```
Phase 1: build hash table from smaller table (probe table)
Phase 2: scan larger table, probe hash table for matches
```
Hash join is O(n + m) vs sort-merge join O(n log n + m log m). PostgreSQL chooses hash join for large equijoins without useful indexes.

**Cache / Memoization**
```
HashMap<state, computed_result> — DP memoization, function result cache
```
Every LRU cache is a HashMap + doubly-linked list. Redis is a production hash map with TTL and persistence.

**Inverted Index (Search)**
```
HashMap<word, Vec<doc_id>> — group-by at massive scale
```
Elasticsearch's core data structure. Same pattern as "group anagrams" — normalize term (stemming), map to document list.

─────────────────────────────────────────────────────
