> **Interview frequency:** High — BFS is the dominant use case; monotonic deque appears in hard sliding window problems; queue simulation appears in design problems

---

## 1. 🧠 Intuition First (No Jargon)

### First Principles

A queue is a **fair-ordering machine**. Things are processed in exactly the order they arrived — First In, First Out (FIFO). No element jumps the line.

The deeper insight: a queue models **level-by-level propagation**. When you want to explore or process something "outward" from a source — spreading like a wave — a queue is the natural structure. Each element you process generates new work that goes to the back; current work comes from the front. This naturally processes all "distance-1" items before "distance-2" items, which is exactly BFS.

A deque (double-ended queue) extends this — you can efficiently add/remove from both ends. This unlocks the **monotonic queue** pattern, which is the deque's most powerful interview application.

### Real-World Analogy

**Queue:** A ticket counter. Customers join at the back, are served from the front. The person who arrived first is served first. There's no cutting — FIFO is enforced.

**Deque:** A subway car — passengers board from both ends (front door and back door), exit from both ends. More flexible than a one-ended line.

**Monotonic Deque:** A bouncer at a club who maintains an ordered guest list. When a new, "more important" guest arrives, the bouncer removes all less-important guests at the back before adding the new one. The front always holds the most important guest currently in the club.

### Why This Pattern Exists

BFS is the canonical "shortest path in unweighted graph" algorithm, and it requires FIFO ordering. Why? Because you must finish exploring all nodes at distance `d` before exploring nodes at distance `d+1`. If you used LIFO (stack), you'd go deep down one path — that's DFS. FIFO enforces the "process current level before next level" invariant that makes BFS correct.

The monotonic deque solves a harder problem: "given a sliding window of size k, what's the maximum/minimum in O(1) per step?" A deque with an ordering invariant maintained at both ends achieves this — something neither a plain queue nor a heap can do at O(1).

## 2. 🔍 Pattern Recognition

### When to Use a Queue

| Situation | Queue type |
|---|---|
| BFS (level-order traversal) | Plain `Queue<T>` |
| Shortest path in unweighted graph | BFS with queue |
| Multi-source BFS (start from multiple nodes) | Queue pre-seeded with all sources |
| Level-by-level tree processing | Queue with level-size tracking |
| Sliding window maximum / minimum | Monotonic deque |
| First non-repeating character in stream | Queue + frequency map |
| Task scheduling / round-robin simulation | Queue |
| Rotten oranges / virus spread | Multi-source BFS |

### Input/Output Signals

**Problem likely needs a queue when:**
- "Shortest path", "minimum steps", "minimum distance" in an unweighted graph or grid
- "Level order", "level by level", "print row by row" in a tree
- "Spread from multiple sources simultaneously" → multi-source BFS
- "Sliding window maximum/minimum" → monotonic deque
- "Process in order of arrival" → plain queue
- "Rotting / infection / fire spreading" → multi-source BFS
- Grid problems where you move in 4 directions and want shortest path

### Common Keywords

`level order`, `minimum steps`, `shortest path`, `distance from source`, `spread`, `infect`, `rotten`, `nearest`, `sliding window max`, `sliding window min`, `first unique`, `FIFO`, `round robin`, `BFS`, `all neighbors at once`

### 10-Second Identification Rule

> **If the problem asks for shortest/minimum in an unweighted setting → BFS → queue.**
> **If the problem asks for max/min in a sliding window of fixed size → monotonic deque.**

Stack: "most recent unresolved thing"
Queue: "oldest unresolved thing / level-by-level expansion"

## 3. ⚙️ Core Algorithm / Template

### Pattern 1: Standard BFS (Single Source, Graph)

```csharp
using System.Collections.Generic;

static int[] Bfs(int[][] graph, int start)
{
    int n = graph.Length;
    var dist = new int[n];
    Array.Fill(dist, -1);
    var queue = new Queue<int>();

    dist[start] = 0;
    queue.Enqueue(start);

    while (queue.Count > 0)
    {
        var node = queue.Dequeue();
        foreach (var next in graph[node])
        {
            if (dist[next] == -1)           // unvisited
            {
                dist[next] = dist[node] + 1;
                queue.Enqueue(next);
            }
        }
    }
    return dist;
}
```

**Mental model:** Mark visited **at enqueue time**, not at dequeue time. This is critical — if you mark at dequeue, the same node can be enqueued multiple times, blowing up to O(V + E²) in dense graphs.

### Pattern 2: BFS on a Grid (4-directional)

```csharp
static int[][] BfsGrid(int[][] grid, (int r, int c) start)
{
    int rows = grid.Length;
    int cols = grid[0].Length;
    int[][] dist = new int[rows][];
    for (int r = 0; r < rows; r++)
        dist[r] = Enumerable.Repeat(-1, cols).ToArray();

    var queue = new Queue<(int r, int c)>();

    dist[start.r][start.c] = 0;
    queue.Enqueue(start);

    int[][] dirs = [[-1, 0], [1, 0], [0, -1], [0, 1]];

    while (queue.Count > 0)
    {
        var (r, c) = queue.Dequeue();
        foreach (var dir in dirs)
        {
            int nr = r + dir[0];
            int nc = c + dir[1];
            if (nr >= 0 && nr < rows && nc >= 0 && nc < cols)
            {
                if (grid[nr][nc] != BLOCKED && dist[nr][nc] == -1)
                {
                    dist[nr][nc] = dist[r][c] + 1;
                    queue.Enqueue((nr, nc));
                }
            }
        }
    }
    return dist;
}
```

**Mental model:** The 4-directional array is canonical — define it once, iterate it. All coordinates use `int` so negative direction arithmetic is safe without underflow concerns. Mark visited at enqueue.

### Pattern 3: Multi-Source BFS

```csharp
static int MultiSourceBfs(int[][] grid)
{
    int rows = grid.Length;
    int cols = grid[0].Length;
    var queue = new Queue<(int r, int c, int d)>();

    // Seed all sources simultaneously
    for (int r = 0; r < rows; r++)
    {
        for (int c = 0; c < cols; c++)
        {
            if (grid[r][c] == SOURCE)
            {
                queue.Enqueue((r, c, 0)); // (row, col, distance)
            }
        }
    }

    int maxDist = 0;
    int[][] dirs = [[-1, 0], [1, 0], [0, -1], [0, 1]];

    while (queue.Count > 0)
    {
        var (r, c, d) = queue.Dequeue();
        maxDist = Math.Max(maxDist, d);
        foreach (var dir in dirs)
        {
            int nr = r + dir[0];
            int nc = c + dir[1];
            if (nr >= 0 && nr < rows && nc >= 0 && nc < cols)
            {
                if (grid[nr][nc] == UNVISITED)
                {
                    grid[nr][nc] = VISITED; // mark as visited
                    queue.Enqueue((nr, nc, d + 1));
                }
            }
        }
    }
    return maxDist;
}
```

**Mental model:** Pre-load all sources into the queue before starting. BFS from all sources simultaneously = the wavefront expands from all origins at once. Each cell's distance is to the *nearest* source. This is equivalent to adding a virtual super-source connected to all real sources at distance 0.

### Pattern 4: Level-Order BFS (Tree, tracking level size)

```csharp
class TreeNode
{
    public int Val;
    public TreeNode? Left, Right;
}

static IList<IList<int>> LevelOrder(TreeNode? root)
{
    var result = new List<IList<int>>();
    if (root == null) return result;

    var queue = new Queue<TreeNode>();
    queue.Enqueue(root);

    while (queue.Count > 0)
    {
        int levelSize = queue.Count;       // snapshot current level count
        var level = new List<int>();

        for (int i = 0; i < levelSize; i++)  // process exactly this level
        {
            var node = queue.Dequeue();
            level.Add(node.Val);

            if (node.Left != null)
                queue.Enqueue(node.Left);
            if (node.Right != null)
                queue.Enqueue(node.Right);
        }
        result.Add(level);
    }
    return result;
}
```

**Mental model:** `queue.Count` at the start of each outer iteration = number of nodes in the current level. Process exactly that many nodes before the next outer iteration. Don't re-check `queue.Count` inside the inner loop — it changes as you enqueue children.

### Pattern 5: Monotonic Deque — Sliding Window Maximum

```csharp
static int[] MaxSlidingWindow(int[] nums, int k)
{
    int n = nums.Length;
    var result = new List<int>(n - k + 1);
    var deque = new LinkedList<int>(); // stores indices, decreasing by value

    for (int i = 0; i < n; i++)
    {
        // Remove indices outside the window
        while (deque.Count > 0 && deque.First!.Value + k <= i)
        {
            deque.RemoveFirst();
        }
        // Maintain decreasing invariant: remove smaller elements from back
        while (deque.Count > 0 && nums[deque.Last!.Value] <= nums[i])
        {
            deque.RemoveLast();
        }
        deque.AddLast(i);

        // Window is full — record maximum (always at front)
        if (i + 1 >= k)
        {
            result.Add(nums[deque.First!.Value]);
        }
    }
    return result.ToArray();
}
```

**Mental model:** The deque stores indices. Values in deque are **monotonically decreasing** from front to back. Front = current window maximum. When adding new element:
1. Evict expired indices from front (outside window).
2. Evict smaller indices from back (they can never be the max while current element is in window).
3. Push current index.

The deque is always non-increasing. Each index enters and exits at most once → O(n) total.

### Pattern 6: BFS with State (Shortest Path with Constraints)

```csharp
// BFS where state = (node, extra_dimension)
// Example: shortest path with at most k obstacles removable
static int ShortestPathWithKRemovals(int[][] grid, int k)
{
    int rows = grid.Length;
    int cols = grid[0].Length;
    // visited[r, c, remainingK] = visited?
    bool[,,] visited = new bool[rows, cols, k + 1];
    var queue = new Queue<(int r, int c, int rem, int steps)>();

    queue.Enqueue((0, 0, k, 0)); // (row, col, removals_left, steps)
    visited[0, 0, k] = true;

    int[][] dirs = [[-1, 0], [1, 0], [0, -1], [0, 1]];

    while (queue.Count > 0)
    {
        var (r, c, rem, steps) = queue.Dequeue();
        if (r == rows - 1 && c == cols - 1) return steps;

        foreach (var dir in dirs)
        {
            int nr = r + dir[0];
            int nc = c + dir[1];
            if (nr < 0 || nr >= rows || nc < 0 || nc >= cols) continue;

            int newRem = rem - grid[nr][nc]; // subtract 1 if obstacle
            if (newRem >= 0 && !visited[nr, nc, newRem])
            {
                visited[nr, nc, newRem] = true;
                queue.Enqueue((nr, nc, newRem, steps + 1));
            }
        }
    }
    return -1;
}
```

**Mental model:** State-space BFS. When the problem has constraints beyond position (remaining moves, remaining health, keys collected), encode them into the state. BFS still finds shortest path — just in an expanded state space. Visited set must include the full state, not just position.

## 4. 📊 Complexity Deep Dive

| Operation | Time | Notes |
|---|---|---|
| `Enqueue` | O(1) amortized | |
| `Dequeue` | O(1) amortized | |
| `Peek front/back` | O(1) | |
| BFS on graph | O(V + E) | Each node enqueued once |
| BFS on grid | O(rows × cols) | Each cell enqueued once |
| Monotonic deque | O(n) total | Each element pushed/popped once |
| Multi-source BFS | O(V + E) | Same as single-source |

### BFS Space Complexity

BFS space = O(width of the graph at the widest level). In the worst case (a complete binary tree's bottom level), this is O(n/2) = O(n). In a grid, the BFS frontier can be O(min(rows, cols)) wide at the diagonal.

Compare to DFS space = O(depth) = O(h). For a balanced tree, DFS is O(log n) vs BFS O(n). For a path graph, DFS is O(n) vs BFS O(1). Choose based on graph shape.

### Monotonic Deque — Why O(n)?

Each of the `n` elements:
- Is pushed to the back **exactly once**.
- Is popped (from either front or back) **at most once**.

Total operations: 2n = **O(n)**. The inner `while` loops look like they could make it O(n²) — they don't, because they're burning through a backlog, not creating new work. Same amortized argument as monotonic stack.

### When BFS Breaks

1. **Weighted edges** — BFS finds shortest hop count, not shortest weight. Use Dijkstra (weighted, non-negative) or Bellman-Ford (negative weights).
2. **State space too large** — BFS stores all frontier nodes in memory. If the state space is exponential (e.g., BFS on a chess board with piece positions), it explodes. Use A* or DFS with pruning.
3. **You need DFS-order** — DFS (stack) for topological sort, cycle detection, finding SCCs, recursive tree problems.
4. **Bidirectional BFS not considered** — for single-source single-target in large graphs, bidirectional BFS meets in the middle and reduces explored space from O(b^d) to O(b^(d/2)).

## 5. ⚠️ Edge Cases & Pitfalls

### Senior-Level Mistakes

**1. Marking visited at dequeue instead of enqueue**
```csharp
// WRONG — allows same node to be enqueued multiple times
while (queue.Count > 0)
{
    var node = queue.Dequeue();
    if (visited[node]) continue;
    visited[node] = true;  // too late
    // ...
}

// RIGHT — mark before enqueue
if (!visited[next])
{
    visited[next] = true;   // mark HERE
    queue.Enqueue(next);
}
```
The "mark at dequeue" pattern works but degrades performance from O(V+E) to O(V+E) with larger constant — nodes can be in the queue multiple times. For dense graphs this matters.

**2. Not snapshotting level size for level-order traversal**
```csharp
// WRONG — intent is fragile; snapshot explicitly
while (queue.Count > 0)
{
    for (int i = 0; i < queue.Count; i++)  // DON'T do this — queue.Count changes as you enqueue children
    {
        // ...
    }
}

// RIGHT — snapshot before inner loop
int levelSize = queue.Count;
for (int i = 0; i < levelSize; i++) { ... }
```

**3. Negative direction arithmetic in grid bounds check**
```csharp
// In C# all coordinates are int — no underflow risk like usize in Rust
// Standard pattern: add direction offset and bounds-check before accessing grid
int nr = r + dir[0];
int nc = c + dir[1];
if (nr >= 0 && nr < rows && nc >= 0 && nc < cols)
{
    // safe to use nr, nc
}
```

**4. Monotonic deque — evicting expired front before checking result**
```csharp
// The eviction of out-of-window indices from front MUST happen before reading the max
// Otherwise you might read a stale index
while (deque.Count > 0 && deque.First!.Value + k <= i)
{
    deque.RemoveFirst();
}
// THEN read result
result.Add(nums[deque.First!.Value]);
```

**5. Multi-source BFS — forgetting to mark sources as visited before seeding**
```csharp
// If sources aren't marked visited before the loop starts,
// they can be re-enqueued by neighboring sources
for (int r = 0; r < rows; r++)
{
    for (int c = 0; c < cols; c++)
    {
        if (grid[r][c] == SOURCE)
        {
            visited[r][c] = true;  // MARK HERE before enqueuing
            queue.Enqueue((r, c));
        }
    }
}
```

**6. BFS with state — visited set must include full state**
```csharp
// WRONG — two paths to (r,c) with different remaining_k share one visited entry
visited[r, c] = true;

// RIGHT — state = (position + constraint dimension)
visited[r, c, remainingK] = true;
```

### Debugging Heuristics

- BFS giving wrong shortest distance? → Check "mark at enqueue" vs "mark at dequeue"
- Level order collecting wrong elements per level? → Check level-size snapshot
- Grid BFS accessing wrong indices? → Check bounds before accessing array
- Monotonic deque giving wrong max? → Check front-eviction order; should happen before reading result
- Multi-source BFS giving wrong distances? → Check if all sources are pre-marked visited

## 6. 🔁 Variations (Pattern Stretching)

### Easy
- **Binary Tree Level Order Traversal** — plain BFS with level-size snapshot
- **Flood Fill** — BFS from seed cell, recolor neighbors
- **Number of Islands** — BFS from each unvisited land cell

### Medium
- **Rotting Oranges** — multi-source BFS; answer = max distance, -1 if unreachable fresh oranges remain
- **01 Matrix** — multi-source BFS from all 0s simultaneously; each cell's distance to nearest 0
- **Walls and Gates** — multi-source BFS from all gates; fill cells with distance
- **Sliding Window Maximum** — monotonic deque
- **Jump Game III / IV** — BFS on index graph built from array values
- **Minimum Genetic Mutation** — BFS on string edit graph with valid mutation set

### Medium-Hard
- **Word Ladder** — BFS on word graph (each word = node, edge if one-letter diff); wildcard trick for O(26×L) neighbor generation
- **Shortest Path in Binary Matrix** — BFS with 8-directional movement
- **Nearest Exit from Entrance in Maze** — BFS, first exit reached = minimum steps
- **Pacific Atlantic Water Flow** — reverse multi-source BFS from both oceans inward

### Hard
- **Shortest Path to Get All Keys** — BFS with state `(position, keys_bitmask)`; exponential state space but k ≤ 6 so 2^6 = 64 key states
- **Minimum Jumps to Reach Home** — BFS with forbidden positions, careful state design
- **Sliding Window Maximum (deque) + DP** — deque used to optimize DP transitions in O(n)
- **Constrained Subsequence Sum** — sliding window max with deque to optimize DP

### How the Pattern Evolves

```
Base: FIFO queue for ordered processing
↓
BFS: queue + visited set = shortest path in unweighted graph
↓
Grid BFS: queue + 4-directional movement + bounds check
↓
Multi-source BFS: pre-seed all sources, wavefront expands simultaneously
↓
Level-order BFS: snapshot level size at each outer iteration
↓
State-space BFS: state = (position, constraint) — BFS in expanded state space
↓
Bidirectional BFS: expand from both source and target, meet in middle
↓
Monotonic deque: maintain ordering invariant for O(1) window max/min
↓
Deque-optimized DP: use monotonic deque to pick optimal previous state in O(1)
```

## 7. 🧪 Interview-Level Problems

### Problem 1 — Medium: Rotting Oranges
**LeetCode 994**

**Summary:** Grid contains `0` (empty), `1` (fresh orange), `2` (rotten orange). Each minute, every fresh orange adjacent to a rotten orange becomes rotten. Return minimum minutes until no fresh oranges remain. Return -1 if impossible.

**Expected approach:**
- Multi-source BFS: seed all initially-rotten oranges into queue
- BFS tick = 1 minute; track distance/time per cell
- After BFS, scan grid for any remaining `1` → return -1
- Answer = maximum distance reached (or 0 if no fresh oranges initially)

**Key insight:** This is not single-source BFS. All rotten oranges spread simultaneously — running BFS from each rotten orange independently and taking max would give wrong answers (double-counts time). Multi-source BFS treats all initial rotten oranges as a single wave origin. Count fresh oranges before BFS; decrement as they're processed; if count > 0 after BFS, return -1.

---

### Problem 2 — Medium-Hard: Word Ladder
**LeetCode 127**

**Summary:** Given `beginWord`, `endWord`, and a word list, find the shortest transformation sequence from begin to end where each step changes exactly one letter and the intermediate word must be in the word list. Return 0 if no sequence exists.

**Expected approach:**
- BFS where each word is a node; edges connect words differing by one letter
- Neighbor generation: for each position, try all 26 letters — O(26 × L) per node
- Optimization: build wildcard-to-words map (`h*t → [hot, hit, hat]`) to find neighbors in O(L) instead of O(26 × L × dict_size)
- Mark words as visited by removing from word set (or a separate visited set)
- Return level count when `endWord` is dequeued

**Key insight:** The graph is implicit — you don't build it upfront. You generate neighbors on the fly. The wildcard trick (`hot` → `*ot`, `h*t`, `ho*`) reduces neighbor lookup from O(26 × L × N) to O(26 × L) by inverting the word list into a pattern map. Without this optimization, TLE on large inputs. Bidirectional BFS from both ends cuts the search space from O(b^d) to O(b^(d/2)) — mention this as the optimization path.

---

### Problem 3 — Hard: Shortest Path to Get All Keys
**LeetCode 864**

**Summary:** Grid maze with walls (`#`), open cells (`.`), start (`@`), keys (lowercase `a-f`), and locks (uppercase `A-F`). Find shortest path to collect all keys. Each lock can only be passed with the corresponding key.

**Expected approach:**
- BFS with state `(row, col, keys_bitmask)`
- Keys represented as bitmask: key `a` = bit 0, `b` = bit 1, ..., `f` = bit 5
- On picking up key `c` at position `(r,c)`: new state has `bitmask | (1 << key_index)`
- On encountering lock `C`: can only pass if `bitmask & (1 << lock_index) != 0`
- Visited set: `visited[r][c][bitmask]` — same position with different keys = different state
- Target state: `bitmask == (1 << num_keys) - 1` (all keys collected)

**Key insight:** Position alone is insufficient as visited state — you might revisit the same cell with different keys, and that's valid (leads to shorter path with different capabilities). The bitmask encodes "which keys do I have". With at most 6 keys, the state space is `rows × cols × 2^6 = rows × cols × 64` — manageable. This is the canonical "BFS with bitmask state" pattern used in competitive programming.

## 8. 🧠 Staff-Level Thinking

### Trade-offs Between Approaches

| Approach | Use when |
|---|---|
| BFS (queue) | Unweighted graph, minimum hops |
| Dijkstra (priority queue) | Weighted graph, non-negative weights |
| A* | Weighted graph, heuristic available (grid with Manhattan distance) |
| Bidirectional BFS | Single source to single target, large diameter graph |
| DFS (stack) | Topological sort, cycle detection, connected components |
| DP | Shortest path in DAG (use DP instead of BFS) |

**BFS vs Dijkstra:** BFS is O(V + E). Dijkstra is O((V + E) log V) with a binary heap. For unweighted graphs, BFS wins. The moment edge weights appear (even all-equal non-unit weights), Dijkstra is correct; BFS gives wrong results. A common interview trap: grid where moves cost 0 or 1 — use 0-1 BFS (deque, not priority queue) for O(V + E).

**0-1 BFS:** When edges have weight 0 or 1, use a deque:
- Weight 0 edge → `AddFirst` (don't increment cost)
- Weight 1 edge → `AddLast` (increment cost)
This gives correct shortest path in O(V + E) — faster than Dijkstra's O((V+E) log V).

```csharp
// 0-1 BFS pattern using LinkedList<T> as a deque
var deque = new LinkedList<(int node, int cost)>();
deque.AddFirst((start, 0));
while (deque.Count > 0)
{
    var (node, cost) = deque.First!.Value;
    deque.RemoveFirst();
    foreach (var (next, weight) in graph[node])
    {
        int newCost = cost + weight;
        if (newCost < dist[next])
        {
            dist[next] = newCost;
            if (weight == 0) deque.AddFirst((next, newCost));
            else             deque.AddLast((next, newCost));
        }
    }
}
```

### Scalability Considerations

- **Memory**: BFS frontier can be O(V) in worst case. For billion-node graphs (social networks), BFS isn't viable in-memory. Use distributed BFS (GraphX, Pregel model).
- **BFS parallelism**: BFS is naturally level-parallelizable. All nodes at distance `d` can be processed in parallel before advancing to `d+1`. This is the basis of parallel BFS in GPU graph analytics.
- **Kafka as a distributed queue**: Production systems use message queues (Kafka, SQS) for exactly the same FIFO ordering guarantee — ensuring events are processed in arrival order, one-at-a-time or with controlled parallelism.
- **Monotonic deque in streaming**: The sliding window maximum pattern appears directly in streaming analytics — e.g., "maximum request rate in any 5-second window" over a real-time event stream.

### When NOT to Use BFS / Queue

1. **Weighted shortest path** — use Dijkstra or Bellman-Ford.
2. **You need all paths, not just shortest** — use DFS with backtracking.
3. **You need a specific path structure** (e.g., Hamiltonian path) — BFS explores too broadly; use DFS + pruning.
4. **The graph is a DAG and you want SSSP** — topological order + DP is O(V+E) and simpler.
5. **Memory is critical** — BFS stores the entire frontier. DFS (iterative, O(depth) space) is more memory-efficient for deep graphs.
6. **Monotonic deque where k changes** — the fixed-window assumption breaks. Use a segment tree or sparse table for variable-window range min/max.

### How to Explain in Interviews

**BFS:**
> "I'm using BFS because I need the shortest path in an unweighted setting. BFS processes all nodes at distance `d` before any node at distance `d+1`, so the first time we reach the target is guaranteed to be via the shortest path. I mark nodes visited at enqueue time to prevent re-processing."

**Multi-source BFS:**
> "All sources spread simultaneously, so I pre-load all source nodes into the queue before the BFS starts. This is equivalent to adding a virtual super-source connected to all real sources with zero-cost edges — but without actually adding that node."

**Monotonic Deque:**
> "I maintain a deque of indices with monotonically decreasing values. When the new element is larger than the back, I pop — those elements can never be the maximum while this element is in the window. The front is always the current window maximum. Each element enters and leaves the deque at most once, so the total time is O(n)."

**State-space BFS:**
> "Position alone isn't sufficient to define state here, because the same position reachable with different keys leads to different future capabilities. I encode the constraint into the state, expanding the state space to `position × key_bitmask`. BFS on this expanded space still finds shortest path."

## 9. 🔗 Real-World Mapping

### Where Queue Patterns Live in Production

**Message Queues (Kafka, SQS, RabbitMQ)**
The literal FIFO queue pattern. Messages enqueued by producers, dequeued by consumers in order. Kafka adds partitioning and consumer groups — but within a partition, strict FIFO is guaranteed. Every microservice event pipeline is a queue.

**Task / Job Queues (Celery, Sidekiq, BullMQ)**
Background job processing. Tasks pushed to queue by API handlers, consumed by worker processes. Priority queues used for job prioritization (heap-based queue = Dijkstra's priority queue at the infrastructure level).

**Rate Limiter — Sliding Window**
The monotonic deque / sliding window pattern implements real sliding-window rate limiters:
```
window = deque of (timestamp, count)
On request: evict entries older than window_size from front
Check: sum of counts in window <= limit
```

**BFS in Social Networks**
"Find all friends within 2 degrees of separation" — BFS with depth limit. LinkedIn's "2nd connection" feature. Facebook's friend recommendation. The BFS state-space explodes at 3+ degrees on billion-node graphs → approximate with sampling.

**Network Routing — BFS / Dijkstra**
OSPF (Open Shortest Path First) routing protocol runs Dijkstra on the network topology graph. Network discovery and spanning tree computation (STP in switches) uses BFS. Every packet forwarded on the internet is a result of pre-computed shortest paths via BFS/Dijkstra.

**Crawler / Indexer (Google, web spiders)**
Web crawling is multi-source BFS over the hyperlink graph. Start with seed URLs (sources), BFS outward, enqueue new URLs as pages are scraped. URL deduplication = visited set. Crawl depth limit = BFS level limit.

**Game AI — BFS Pathfinding**
BFS for shortest path in game grids (no weights). A* (BFS + heuristic) for efficient navigation in large maps. Flood fill for paint bucket tools, game board coverage detection, territory claims in strategy games.

**Build Systems (Bazel, Make) — BFS/Topological Order**
Dependency resolution uses topological sort (Kahn's algorithm = BFS variant). To find "which targets need rebuilding", a reverse BFS from changed files propagates through the dependency graph.

## 10. ⚡ 10-Second Recall Cheat Sheet

```
TRIGGER → PATTERN
─────────────────────────────────────────────────────
"Shortest path / min steps (unweighted)"    → BFS + queue
"Level order / row by row"                  → BFS + level-size snapshot
"Spread from all sources simultaneously"    → multi-source BFS (pre-seed all sources)
"Sliding window maximum / minimum"          → monotonic deque (LinkedList<int> as deque)
"BFS but with constraints (keys, health)"   → state-space BFS: state = (pos, constraint)
"0/1 edge weights"                          → 0-1 BFS with deque (AddFirst/AddLast)
─────────────────────────────────────────────────────

CORE C# IDIOMS
─────────────────────────────────────────────────────
Queue:            new Queue<T>()
Enqueue:          queue.Enqueue(val)
Dequeue:          queue.Dequeue()              // throws if empty
Peek front:       queue.Peek()                 // throws if empty
BFS template:     mark visited AT enqueue, not dequeue
Level size:       int levelSize = queue.Count; (snapshot before inner loop)
Grid bounds:      use int arithmetic throughout — no underflow risk
4-directional:    [[-1,0],[1,0],[0,-1],[0,1]]
Monotonic pop:    while Last.Value < current { RemoveLast() }; then AddLast(i)
Expired evict:    while First.Value + k <= i { RemoveFirst() }
─────────────────────────────────────────────────────

BFS DECISION TABLE
─────────────────────────────────────────────────────
Unweighted:       BFS (queue)          O(V+E)
Weighted ≥ 0:     Dijkstra (heap)      O((V+E) log V)
0 or 1 weights:   0-1 BFS (deque)     O(V+E)
Negative weights: Bellman-Ford         O(VE)
Single→Single, large graph: Bidirectional BFS
─────────────────────────────────────────────────────

MONOTONIC DEQUE
─────────────────────────────────────────────────────
Window MAX: deque holds indices in DECREASING value order
Window MIN: deque holds indices in INCREASING value order
On new element i:
  1. RemoveFirst while front index is out of window (First.Value + k <= i)
  2. RemoveLast while back value is weaker than nums[i]
  3. AddLast(i)
  4. Front = answer for current window (when i >= k-1)
─────────────────────────────────────────────────────

COMPLEXITY
─────────────────────────────────────────────────────
BFS:              O(V+E) time, O(V) space (frontier)
Grid BFS:         O(rows×cols) time, O(rows×cols) space
Multi-source:     O(V+E) — same as single source
Monotonic deque:  O(n) time, O(k) space
State BFS:        O(V × |states|) time and space
─────────────────────────────────────────────────────

COMMON MISTAKES
─────────────────────────────────────────────────────
✗ Marking visited at dequeue (allows duplicates in queue)
✗ Re-evaluating queue.Count inside inner level loop
✗ Using wrong index type for grid directions (int is safe in C#)
✗ Not pre-marking sources as visited in multi-source BFS
✗ Using BFS for weighted graphs (wrong shortest path)
✗ State visited = position only (misses constraint dimension)
✗ Reading monotonic deque front before evicting expired indices
```

---

> **One-liner to burn in:**
> *A queue enforces fair, level-by-level expansion — use it whenever the first time you reach something must be via the shortest path (BFS), or whenever you need the maximum/minimum of a sliding window in O(1) per step (monotonic deque).*
