## 1. 🧠 Intuition First (No Jargon)

### First Principles

"Top K" is not a single algorithm — it's a **problem class** with multiple optimal solutions depending on constraints. The unifying question: *how do you select k elements from n without doing more work than necessary?*

Full sorting gives you complete order — but if you only need k elements, you've done O(n log n) work to answer an O(n log k) or even O(n) question. The art of Top K is choosing the right algorithm for the specific access pattern:

- **Static array, any k**: QuickSelect → O(n) average
- **Streaming / online, fixed k**: Size-k heap → O(n log k)
- **Bounded integer values**: Bucket/counting sort → O(n)
- **k = 1 (maximum/minimum)**: Single linear scan → O(n)
- **k = n (sorted order needed)**: Just sort → O(n log n)
- **Multiple queries on same data**: Preprocess → sort or wavelet tree

The skill interviewers test: can you identify *which* Top K variant applies and articulate *why* the chosen approach is optimal under the given constraints?

### Real-World Analogy

**Search engine results page.** When you search Google, you don't see all 10 billion matching documents — you see the top 10. Google doesn't sort all 10 billion results by relevance and hand you the first page. It maintains pre-computed top-k indices per query term, merges k-sorted lists from shards, and stops as soon as it has enough candidates.

At every scale — trending hashtags on Twitter, recommended videos on YouTube, nearest restaurants on Yelp — the underlying question is "top k by some score from a massive dataset". The solution is never "sort everything."

### Why This Pattern Exists

Full sort is wasteful when:
1. You only need rank 1 through k (not the full ranking)
2. Data arrives in a stream (can't sort upfront)
3. n is enormous (can't hold all data in memory)
4. The "score" to rank by changes dynamically

Top K algorithms exploit the fact that you can determine membership in the top-k set without knowing the relative order within that set (except for the exact k-th element problem).

## 2. 🔍 Pattern Recognition

### When to Use Top K

| Trigger | Variant | Algorithm |
|---|---|---|
| k largest/smallest values | Value-based Top K | Min/max heap of size k |
| k most frequent elements | Frequency Top K | Freq map + heap or bucket sort |
| k closest points | Distance Top K | Max-heap of size k on distance |
| k-th largest/smallest exactly | Order statistics | QuickSelect or heap |
| Top k in a stream | Streaming Top K | Size-k heap |
| Top k frequent words | String Top K | Freq map + heap with custom sort |
| Top k in sorted matrix | Matrix Top K | Min-heap with index navigation |
| Top k with grouping | Grouped Top K | Freq map + heap per group |

### Input/Output Signals

**Immediately recognize Top K when you see:**
- Return `k` elements (not a single value)
- "Most frequent", "most common", "highest count"
- "Largest", "smallest", "closest", "nearest" with a specific count k
- "Top", "bottom", "best", "worst" + a number
- "k-th" anything — k-th largest, k-th smallest, k-th most frequent
- Stream of data + maintain running top-k
- Merge multiple sorted sources and take top k combined

### Common Keywords

`top k`, `k largest`, `k smallest`, `k most frequent`, `k closest`, `k nearest`, `k-th element`, `most common`, `highest frequency`, `trending`, `leaderboard`, `ranking`, `best k`, `return k elements`

### 10-Second Identification Rule

> **See "k" + "largest/smallest/frequent/closest" → Top K. Determine if data is static or streaming, if values are bounded integers, then pick algorithm.**

## 3. ⚙️ Core Algorithm / Template

### Algorithm 1: Min-Heap of Size K (Universal, Streaming-Safe)

```csharp
/// Returns k largest elements. Works on streams — O(n log k).
static List<int> TopKLargestHeap(int[] nums, int k) {
    // PriorityQueue<TElement, TPriority> is a min-priority-queue in C#
    var minHeap = new PriorityQueue<int, int>();

    foreach (var n in nums) {
        minHeap.Enqueue(n, n);
        if (minHeap.Count > k) {
            minHeap.Dequeue(); // evict the weakest — only k "VIPs" allowed
        }
    }

    var result = new List<int>();
    while (minHeap.Count > 0) result.Add(minHeap.Dequeue());
    result.Sort((a, b) => b.CompareTo(a)); // sort result descending if order needed
    return result;
}

/// K-th largest only — no need to collect all k.
static int KthLargestHeap(int[] nums, int k) {
    var minHeap = new PriorityQueue<int, int>();
    foreach (var n in nums) {
        minHeap.Enqueue(n, n);
        if (minHeap.Count > k) minHeap.Dequeue();
    }
    return minHeap.Peek();
}
```

**Heap type rule:**
- k *largest* → **min**-heap (top = current threshold = weakest in top-k set)
- k *smallest* → **max**-heap (top = current threshold = strongest in bottom-k set)

**Why this works:** The heap holds the current "qualifying set" of size k. Its top is the boundary — the weakest member that made the cut. Any new element that beats the boundary displaces it.

### Algorithm 2: QuickSelect (Optimal for Static Arrays, k-th Element)

```csharp
/// Finds k-th largest element in O(n) average, O(n²) worst.
/// Mutates the input array (partition in place).
static int KthLargestQuickselect(int[] nums, int k) {
    int n = nums.Length;
    return Quickselect(nums, 0, n - 1, n - k); // k-th largest = (n-k)-th smallest (0-indexed)
}

static int Quickselect(int[] nums, int lo, int hi, int targetRank) {
    if (lo == hi) return nums[lo];

    int pivotIdx = Partition(nums, lo, hi);

    if (pivotIdx == targetRank) {
        return nums[pivotIdx];
    } else if (pivotIdx < targetRank) {
        return Quickselect(nums, pivotIdx + 1, hi, targetRank);
    } else {
        // pivotIdx > targetRank, guard against going below lo
        return Quickselect(nums, lo, pivotIdx - 1, targetRank);
    }
}

static int Partition(int[] nums, int lo, int hi) {
    // Median-of-three pivot for better average performance
    int mid = lo + (hi - lo) / 2;
    if (nums[mid] < nums[lo]) (nums[lo], nums[mid]) = (nums[mid], nums[lo]);
    if (nums[hi] < nums[lo]) (nums[lo], nums[hi]) = (nums[hi], nums[lo]);
    if (nums[mid] < nums[hi]) (nums[mid], nums[hi]) = (nums[hi], nums[mid]);
    // nums[hi] is now the median of three — use as pivot

    int pivot = nums[hi];
    int i = lo;

    for (int j = lo; j < hi; j++) {
        if (nums[j] <= pivot) {
            (nums[i], nums[j]) = (nums[j], nums[i]);
            i++;
        }
    }
    (nums[i], nums[hi]) = (nums[hi], nums[i]);
    return i;
}
```

**Mental model:** Partition the array around a pivot. If the pivot lands at rank `targetRank`, done. If pivot rank > target, recurse left. If pivot rank < target, recurse right. Average case: each partition halves the search space → T(n) = T(n/2) + O(n) → O(n).

**Median-of-three pivot:** Prevents O(n²) worst case on sorted/reverse-sorted inputs (random pivot has same issue). For production, use introselect (QuickSelect with fallback to median-of-medians if recursion depth exceeds threshold).

### Algorithm 3: Bucket Sort / Counting Sort (Bounded Integer Values, O(n))

```csharp
/// Top k most frequent elements — O(n) using bucket sort.
/// Key insight: frequency is bounded by n, so bucket indices = [0..n].
static List<int> TopKFrequentBucket(int[] nums, int k) {
    int n = nums.Length;

    // Step 1: frequency map — O(n)
    var freq = new Dictionary<int, int>();
    foreach (var x in nums) freq[x] = freq.GetValueOrDefault(x) + 1;

    // Step 2: bucket by frequency — O(n)
    // bucket[f] = list of values with frequency f
    var buckets = new List<int>[n + 1];
    for (int i = 0; i <= n; i++) buckets[i] = new List<int>();
    foreach (var kvp in freq) { buckets[kvp.Value].Add(kvp.Key); }

    // Step 3: collect top k from highest-frequency bucket down — O(n)
    var result = new List<int>(k);
    for (int i = n; i >= 0; i--) {
        foreach (var val in buckets[i]) {
            result.Add(val);
            if (result.Count == k) return result;
        }
    }
    return result;
}
```

**Mental model:** Frequencies are bounded by n. Build a bucket array where index = frequency. Sweep from highest bucket downward, collecting until you have k elements. Zero sorting involved — O(n) total.

**When this applies:** The "score" to rank by is a bounded integer (frequency, count, index). Won't work for arbitrary floating-point distances or unbound scores.

### Algorithm 4: Top K with Custom Score (K Closest Points)

```csharp
static int[][] KClosest(int[][] points, int k) {
    // Max-heap of size k on squared distance (no sqrt needed — monotonic)
    // Top = farthest point among current k closest → eviction candidate
    // Negate distance for max-heap behavior with C#'s min-priority-queue
    var maxHeap = new PriorityQueue<int[], int>();

    foreach (var point in points) {
        int distSq = point[0] * point[0] + point[1] * point[1];
        maxHeap.Enqueue(point, -distSq); // negate for max behavior
        if (maxHeap.Count > k) {
            maxHeap.Dequeue(); // evict the farthest
        }
    }

    var result = new List<int[]>();
    while (maxHeap.Count > 0) result.Add(maxHeap.Dequeue());
    return result.ToArray();
}
```

**Mental model:** For k *closest* (minimum distance), use a **max**-heap of size k on distance. Top = current farthest in the "k closest" set. Any new point closer than the current farthest displaces it.

**Optimization:** Use squared distance. `sqrt` is monotonic — comparing `x² + y²` preserves ranking. Avoids expensive floating-point sqrt in the hot loop.

### Algorithm 5: Top K Frequent Words (Heap with Lexicographic Tiebreak)

```csharp
// Custom ordering: higher frequency first; ties broken alphabetically (lower = "better")
record Entry(int Freq, string Word) : IComparable<Entry> {
    public int CompareTo(Entry other) {
        int cmp = Freq.CompareTo(other.Freq);
        if (cmp != 0) return cmp;
        // reverse alpha: "earlier" word is "larger" (better) in min-heap eviction sense
        return string.Compare(other.Word, Word, StringComparison.Ordinal);
    }
}

static List<string> TopKFrequentWords(string[] words, int k) {
    var freq = new Dictionary<string, int>();
    foreach (var w in words) freq[w] = freq.GetValueOrDefault(w) + 1;

    // Min-heap of size k: weakest (lowest freq, or latest alpha for ties) at top
    // Use Entry as both element and priority for custom comparison
    var heap = new PriorityQueue<Entry, Entry>(Comparer<Entry>.Create((a, b) => a.CompareTo(b)));
    foreach (var kvp in freq) {
        var entry = new Entry(kvp.Value, kvp.Key);
        heap.Enqueue(entry, entry);
        if (heap.Count > k) heap.Dequeue();
    }

    // Collect and sort: descending freq, ascending alpha
    var result = new List<Entry>();
    while (heap.Count > 0) result.Add(heap.Dequeue());
    result.Sort((a, b) => {
        var c = b.Freq.CompareTo(a.Freq);
        return c != 0 ? c : string.Compare(a.Word, b.Word, StringComparison.Ordinal);
    });
    return result.Select(e => e.Word).ToList();
}
```

**Mental model:** Multi-criteria ranking requires a custom comparator. The key insight: define "worse" clearly (what gets evicted from the top-k set first?) and invert it for the min-heap's eviction logic.

### Algorithm 6: Top K in a Sorted Matrix

```csharp
/// K-th smallest element in an n×n sorted matrix.
/// Each row and column is sorted ascending.
static int KthSmallestMatrix(int[][] matrix, int k) {
    int n = matrix.Length;
    // Min-heap: (value, row, col) — PriorityQueue<TElement, TPriority> is a min-heap
    var heap = new PriorityQueue<(int val, int r, int c), int>();

    // Seed with first column (smallest in each row)
    for (int r = 0; r < Math.Min(n, k); r++) { // no need to seed more than k rows
        heap.Enqueue((matrix[r][0], r, 0), matrix[r][0]);
    }

    int count = 0;
    while (heap.Count > 0) {
        var (val, r, c) = heap.Dequeue();
        count++;
        if (count == k) return val;
        if (c + 1 < n) {
            heap.Enqueue((matrix[r][c + 1], r, c + 1), matrix[r][c + 1]);
        }
    }
    return -1; // unreachable if k is valid
}
```

**Mental model:** Same structure as "merge k sorted lists" — each row is one sorted list. Heap holds one candidate per active row. Pop minimum → push its right neighbor in same row.

## 4. 📊 Complexity Deep Dive

### Full Comparison Table

| Algorithm | Time | Space | Constraints | Use Case |
|---|---|---|---|---|
| Sort + slice | O(n log n) | O(n) | None | k ≈ n, simplest code |
| Min/max heap size k | O(n log k) | O(k) | None, streaming OK | k << n, streaming |
| QuickSelect | O(n) avg, O(n²) worst | O(log n) | Static array, k-th only | Best avg for k-th |
| Introselect | O(n) guaranteed | O(log n) | Static array | Worst-case O(n) needed |
| Bucket sort (freq) | O(n) | O(n) | Bounded integer score | Top k frequent |
| Counting sort | O(n + range) | O(range) | Bounded integer values | Very small range |
| nth_element (std) | O(n) avg | O(1) | Static, partition only | C++ only; C#: QuickSelect |
| Binary search on answer | O(n log(max-min)) | O(1) | Monotonic check exists | K-th in matrix, etc. |

### When O(n log k) Beats O(n log n) in Practice

```
n = 1,000,000   k = 100
O(n log k):  1,000,000 × log₂(100)  ≈ 1,000,000 × 6.6  ≈ 6.6M ops
O(n log n):  1,000,000 × log₂(1M)  ≈ 1,000,000 × 20   ≈ 20M ops
Speedup: ~3×

n = 1,000,000   k = 10
O(n log k):  1,000,000 × 3.3  ≈ 3.3M ops
O(n log n):  20M ops
Speedup: ~6×
```

For k = 10 from 1M elements, the heap approach is 6x faster than sort. This compounds at larger n.

### Why QuickSelect Isn't Always the Answer

| Factor | Heap | QuickSelect |
|---|---|---|
| Average time | O(n log k) | O(n) |
| Worst case | O(n log k) | O(n²) without median-of-medians |
| Streaming | ✓ | ✗ (needs full array) |
| Returns all k elements | ✓ | ✗ (returns k-th; must partition to collect all k) |
| Mutates input | ✗ | ✓ (partitions in place) |
| Code complexity | Low | Moderate |
| Cache performance | Poor (heap jumps memory) | Good (partitions contiguous range) |

**Rule:** In interviews, default to the heap unless:
- The problem says "static array" + "find k-th exactly" → mention QuickSelect
- The problem says "space O(1)" or "don't use extra memory" → QuickSelect

### Space Analysis

- Heap approach: O(k) extra — only the top-k set in memory. Perfect for streaming over n = 10^9.
- QuickSelect: O(log n) stack space (recursion depth). O(1) extra if iterative.
- Bucket sort: O(n) for the bucket array. Non-negotiable — can't do better for this approach.

## 5. ⚠️ Edge Cases & Pitfalls

### Senior-Level Mistakes

**1. Using max-heap for k largest (most common confusion)**
```csharp
// WRONG: max-heap gives you the global maximum at top, not the k-th largest
// In C#: PriorityQueue with negated priority acts as max-heap
// After processing: top is largest, not k-th largest

// RIGHT: min-heap of size k — top = k-th largest = eviction threshold
var minHeap = new PriorityQueue<int, int>();
// Enqueue as (value, value) — smaller priority = dequeued first
```

The confusion: "I want the largest, so use max-heap." Wrong frame. The heap holds the *answer set*. Its top is the *weakest member of the answer set* — the element most at risk of being displaced. For "largest k", the weakest is the smallest of the large → min-heap.

**2. QuickSelect off-by-one in rank mapping**
```csharp
// Problem: "find k-th largest" (1-indexed, k=1 means the largest)
// In 0-indexed array of n elements: k-th largest = element at sorted index (n - k)
int targetRank = nums.Length - k;  // 0-indexed rank from smallest end

// k=1 (largest) → targetRank = n-1 ✓
// k=n (smallest) → targetRank = 0 ✓
```

**3. QuickSelect recursion on index underflow**
```csharp
// When pivotIdx == lo and targetRank < pivotIdx, recursing on (lo, pivotIdx - 1)
// could pass an invalid range. Guard by checking before recursing:
if (pivotIdx > lo) {
    return Quickselect(nums, lo, pivotIdx - 1, targetRank);
}
// Or use int indices (no unsigned underflow risk in C# — ints are signed)
```

**4. Bucket sort index out of bounds**
```csharp
// Bucket index = frequency, bounded by nums.Length
// But if you build buckets with size nums.Length and frequency CAN equal nums.Length,
// you need buckets of size n + 1 (indices 0..n)
var buckets = new List<int>[nums.Length + 1]; // NOT nums.Length
```

**5. Tiebreaking in top k frequent words**
```csharp
// Problem: "if two words have same frequency, return alphabetically earlier one"
// Min-heap evicts the "worse" element — worse = lower freq OR (same freq AND later alpha)
// Comparator must reflect this precisely
// Common mistake: only compare by frequency, ignoring alphabetical tiebreak
```

**6. k > n (more elements requested than exist)**
```csharp
k = Math.Min(k, nums.Length); // guard at entry — never exceed array size
if (k == 0) return new List<int>();
```

**7. Squared distance overflow for k closest points**
```csharp
// x and y can be up to 10^4 in LeetCode
// x² + y² can be up to 2 × 10^8 — fits in int.MaxValue (2.1 × 10^9) ✓
// But if coordinates are larger (up to 10^5 or 10^6), use long for distance
long distSq = (long)x * x + (long)y * y;
```

### Debugging Heuristics

- Wrong k-th element? → Check rank mapping: `n - k` for 0-indexed k-th largest
- Heap returning wrong elements? → Verify heap type (min vs max) against the goal
- QuickSelect giving wrong results? → Check index bounds on recursive calls
- Bucket sort missing elements? → Verify bucket array size is `n + 1`
- Top k words wrong order? → Verify tiebreaking in comparator is exactly as problem states
- K closest returning wrong points? → Check heap type — max-heap for closest, min-heap for farthest

## 6. 🔁 Variations (Pattern Stretching)

### Easy
- **Kth Largest Element in a Stream** — min-heap of size k; `add()` maintains heap, `top()` returns k-th largest
- **Last Stone Weight** — max-heap; pop two, push difference if non-zero
- **Relative Ranks** — sort with original index tracking

### Medium
- **Kth Largest Element in an Array** — min-heap or QuickSelect
- **Top K Frequent Elements** — freq map + min-heap size k OR bucket sort
- **Top K Frequent Words** — freq map + min-heap with custom comparator (alpha tiebreak)
- **K Closest Points to Origin** — max-heap size k on squared distance
- **Sort Characters By Frequency** — freq map + max-heap → rebuild string

### Medium-Hard
- **K-th Smallest Element in Sorted Matrix** — min-heap navigating sorted rows OR binary search on answer
- **K Pairs with Smallest Sums** — min-heap seeded with (nums1[i] + nums2[0], i, 0); advance j-pointer on pop
- **Find K-th Largest XOR Coordinate Value** — prefix XOR 2D + sort/heap on XOR values
- **Reorganize String** — max-heap on char freq; greedily place most frequent without adjacency violation

### Hard
- **Smallest Range Covering Elements from K Lists** — min-heap + tracked max; invariant: one element per list in heap
- **Maximum Performance of a Team** — sort by speed desc; min-heap of efficiency size k; track sum
- **Find K-th Smallest Pair Distance** — binary search on answer + two-pointer counting
- **K-th Smallest Prime Fraction** — min-heap on fractions OR binary search on answer
- **Sliding Window Median (with K)** — two heaps + lazy deletion map

### How the Pattern Evolves

```
Base: global top-1 (linear scan)
↓
Top K from static array: sort or min-heap size k
↓
Top K with score function: custom comparator on heap
↓
Top K frequent: freq map → score is frequency
↓
Streaming Top K: heap handles online insertions
↓
K-th from structured data (matrix): heap navigation
↓
Top K with two constraints: multi-field comparator or combined score
↓
Approximate Top K at scale: Count-Min Sketch + heap
↓
Distributed Top K: per-shard heaps + coordinator merge
```

## 7. 🧪 Interview-Level Problems

### Problem 1 — Medium: Top K Frequent Elements
**LeetCode 347**

**Summary:** Given integer array, return the k most frequent elements. The answer is guaranteed to be unique. Follow-up: must be better than O(n log n).

**Expected approach (two solutions — present both):**

*Heap solution O(n log k):*
- Build frequency map O(n)
- Min-heap of size k on (freq, value) — O(m log k) where m = distinct elements
- Return heap contents

*Bucket sort O(n):*
- Build frequency map O(n)
- Bucket array of size n+1: `bucket[freq]` = list of values with that frequency
- Sweep from bucket[n] down, collect until k elements gathered

**Key insight:** The follow-up constraint ("better than O(n log n)") is the signal to pivot from heap to bucket sort. Frequency is bounded by n, which is the insight that enables bucket sort. Mentioning both solutions and the constraint that determines which to use is the staff-level answer — not just "use a heap."

---

### Problem 2 — Medium-Hard: K-th Smallest Element in Sorted Matrix
**LeetCode 378**

**Summary:** Given an n×n matrix where each row and column is sorted in ascending order, find the k-th smallest element.

**Expected approach (two solutions):**

*Heap solution O(k log n):*
- Seed min-heap with first element of each row (n entries)
- Pop minimum k times; each pop pushes the right neighbor in same row
- k-th pop = answer

*Binary search on answer O(n log(max-min)):*
- Binary search on the value space [matrix[0][0], matrix[n-1][n-1]]
- For mid, count elements ≤ mid using staircase traversal (start top-right, walk down-left): O(n) per check
- Find smallest value where count ≥ k

**Key insight:** The binary search approach is non-obvious — you're not binary searching an array, you're binary searching the *value space*. The counting function (staircase: start at top-right corner, go left if current > mid, go down if current ≤ mid) runs in O(n) and is the key insight. Binary search on answer recurs throughout Top K problems when you can cheaply count "how many elements are ≤ x?" — flag this as a reusable pattern.

---

### Problem 3 — Hard: Find K-th Smallest Pair Distance
**LeetCode 719**

**Summary:** Given integer array, find the k-th smallest distance among all pairs `(nums[i], nums[j])` where `i < j`.

**Expected approach:**
- Sort the array O(n log n)
- Binary search on the answer (distance value, range [0, nums[max] - nums[min]])
- For a candidate distance `mid`, count pairs with distance ≤ mid using two pointers: O(n)
- Find smallest `mid` where count ≥ k (lower-bound binary search)

**Key insight:** Naively generating all O(n²) pairs and sorting is O(n² log n) — too slow. The binary search on answer reduces this to O(n log W) where W = max value range. The counting step (two pointers on sorted array: for each right pointer, advance left pointer while `nums[right] - nums[left] > mid`) is O(n). Sorting first enables both the binary search bounds and the two-pointer counting. This is the canonical "binary search on answer + fast counting" Top K pattern — appears in 5+ distinct hard problems.

## 8. 🧠 Staff-Level Thinking

### Complete Algorithm Selection Framework

```
Given: Find top k elements from n by some score

Step 1: Is k = 1?
  → Single linear scan O(n), O(1) space. Done.

Step 2: Is score a bounded integer (e.g., frequency ≤ n)?
  → Bucket/counting sort O(n). Best possible.

Step 3: Is data a stream (elements arrive one at a time)?
  → Min/max heap of size k. Only viable O(n log k) option.

Step 4: Need only the k-th element, not all k? Array is mutable?
  → QuickSelect O(n) avg. Mention introselect for worst-case O(n) guarantee.

Step 5: Need all k elements, static array?
  → Both heap O(n log k) and QuickSelect+partition work.
  → k << n: heap or QuickSelect.
  → k ≈ n: just sort.

Step 6: Can you count "how many elements have score ≤ x" cheaply?
  → Binary search on answer. Especially useful for matrix/pair problems.
```

### Trade-offs Deep Dive

**Heap vs QuickSelect — the real difference:**

Heap is *online* and *non-destructive*. QuickSelect is *offline* (needs full array) and *destructive* (partitions in place). In a system design context, if the data source is a database query, a Kafka stream, or a file that you read once, QuickSelect is impossible — you need the heap. This is why in production, top-k is almost always heap-based, not QuickSelect-based, even though QuickSelect is asymptotically faster.

**Exact vs approximate at scale:**

For n = 1B elements (too large for memory), exact top-k requires distributed algorithms. The standard production approach:
- **Count-Min Sketch**: probabilistic frequency estimation, O(1) per update, sub-linear space
- **Space-Saving algorithm** (Misra-Gries): maintains top-k candidates with error bounds, O(k) space
- **HeavyHitters**: identifies elements appearing > n/k times with guaranteed recall

In an interview, knowing these exist and their trade-offs (approximate but O(1) memory vs exact but O(n) memory) is a staff-level signal.

**Distributed Top K (system design adjacent):**

```
1. Shard data across m machines
2. Each shard computes its local top-k (heap, O(shard_size × log k))
3. Coordinator collects m heaps of size k each
4. Merges m×k candidates → final top-k (merge k sorted lists pattern)

Total: O(n log k) + O(mk log k) coordinator step
For m << n: coordinator step is negligible
```

This is how MapReduce top-k works. Mentioning this in an interview when n is described as "very large" differentiates you.

### When NOT to Use Heap for Top K

1. **k = 1**: Just do a linear scan. Heap overhead for k=1 is wasteful.
2. **Data is sorted or nearly sorted**: Top k is trivially the first/last k elements.
3. **Score is a bounded integer**: Bucket sort is O(n) — strictly better than O(n log k).
4. **Multiple repeated Top K queries on the same data**: Sort once (O(n log n)), answer each query in O(1). Heap re-runs O(n log k) per query.
5. **Range queries**: "Top k elements in subarray [l, r]" — wavelet tree or persistent segment tree, not a heap.
6. **Need to update scores dynamically**: Heap has no efficient decrease-key. Use a `SortedSet` or indexed heap.

### How to Explain in Interviews

**Opening framing:**
> "Top K is a problem class, not a single algorithm. Before coding, I want to confirm: is this data static or streaming? Are the values bounded integers? Do I need all k elements or just the k-th? That determines whether I use a heap, QuickSelect, or bucket sort."

**On the heap approach:**
> "I maintain a min-heap of exactly k elements. The heap top is always the current k-th largest — the weakest element in my qualifying set. For each new element, if it beats the threshold, I evict the weakest and insert the new one. O(n log k) time, O(k) space. This works on streams."

**On QuickSelect:**
> "QuickSelect partitions the array around a pivot — if the pivot lands at rank n-k (0-indexed), that's our k-th largest. We recurse on only one side, so average work is O(n) + O(n/2) + O(n/4) + ... = O(2n) = O(n). I'd use this for a static array where I only need the k-th element and can mutate the input. In production I'd add median-of-three pivot selection to avoid O(n²) on sorted inputs."

**On bucket sort:**
> "Frequency is bounded by n, so I can use the frequency as a bucket index directly. Build a bucket array of size n+1 where bucket[f] holds all values with frequency f. Sweep from the highest bucket down until I have k elements. Entirely O(n) — no sorting, no heap."

## 9. 🔗 Real-World Mapping

### Top K in Production Systems

**Search Engine Ranking (Google, Elasticsearch)**
Every search query computes a relevance score for potentially millions of documents. The retrieval layer uses inverted indices to get candidates, then a scoring function ranks them. Top-k extraction using a bounded-size heap is the standard pattern. Elasticsearch's `top_hits` aggregation uses an exact heap internally.

**Social Feed / Trending (Twitter, Instagram)**
"Top k trending hashtags in the last hour" — streaming Top K over a sliding window. Each shard counts hashtag frequencies using Count-Min Sketch (approximate) or exact counters. A coordinator collects local top-k candidates from each shard and merges (merge-k-sorted pattern). Approximate algorithms (HeavyHitters, Space-Saving) handle the memory constraint at Twitter-scale (billions of tweets/day).

**Leaderboards (Gaming, eSports — directly relevant)**
Real-time leaderboard = top-k by score. For real-time updates: Redis Sorted Sets are a skip list internally, providing O(log n) rank queries and updates. For batch leaderboards: heap over player scores. For "top 100 in your region" = partitioned top-k: each region maintains its own heap.

**Database Query Execution — TOP N / LIMIT k**
PostgreSQL, MySQL, SQL Server all implement `SELECT ... ORDER BY score DESC LIMIT k` with a heap of size k — not a full sort. This is called a "top-N sort" in query planners. For large tables with no useful index on the sort column, the planner uses a heap scan rather than sort + slice.

**Recommendation Systems (YouTube, Netflix)**
Candidate generation phase: ANN (approximate nearest neighbor) retrieves top-k similar items from millions by embedding distance. Exact k-closest with a heap is O(n log k) per query — fine for n = 10K, too slow for n = 100M. At scale: HNSW / IVF indices (graph-based ANN) provide approximate top-k in O(log n) with tunable recall.

**Log Analysis / Observability (ELK, Grafana Loki)**
"Top 10 slowest endpoints in the last 5 minutes" = top-k by max/p99 latency over a time-windowed stream. Streaming top-k with a heap per window. Aggregation pipelines in Kibana and Grafana's query engine use this pattern internally.

**FinTech — Portfolio / Risk Analytics**
"Top k positions by P&L loss" for real-time risk management. A min-heap of size k on loss magnitude, updated on each tick. For regulatory reporting: exact top-k required (approximate insufficient). This is a streaming top-k problem with strict correctness requirements — heap, not sketch.

## 10. ⚡ 10-Second Recall Cheat Sheet

```
ALGORITHM SELECTION
─────────────────────────────────────────────────────
k = 1                         → linear scan O(n), O(1) space
Streaming / online            → min/max heap size k, O(n log k)
Score is bounded integer      → bucket sort, O(n)
Static array, k-th only       → QuickSelect O(n) avg
Multiple queries, same data   → sort once O(n log n), query O(1)
Structured data (matrix/pairs)→ binary search on answer
k ≈ n                         → just sort O(n log n)
─────────────────────────────────────────────────────

HEAP TYPE SELECTION
─────────────────────────────────────────────────────
k LARGEST   → min-heap size k  (top = weakest of the strong = eviction threshold)
k SMALLEST  → max-heap size k  (top = strongest of the weak = eviction threshold)
k CLOSEST   → max-heap size k on distance (evict farthest)
k FARTHEST  → min-heap size k on distance (evict closest)
─────────────────────────────────────────────────────

QUICKSELECT RANK MAPPING
─────────────────────────────────────────────────────
k-th largest  (1-indexed) → 0-indexed rank = n - k
k-th smallest (1-indexed) → 0-indexed rank = k - 1
─────────────────────────────────────────────────────

CORE C# IDIOMS
─────────────────────────────────────────────────────
Min-heap:   new PriorityQueue<int, int>()  (enqueue as (val, val))
Max-heap:   new PriorityQueue<int, int>()  (enqueue as (val, -val))
Evict:      heap.Enqueue(x, x); if (heap.Count > k) heap.Dequeue();
Peek top:   heap.Peek()
No sqrt:    compare x²+y² for distance (monotonic, avoids double)
Freq count: freq[x] = freq.GetValueOrDefault(x) + 1;
Bucket:     new List<int>[n + 1]   — size n+1, index by freq
─────────────────────────────────────────────────────

COMPLEXITY QUICK REFERENCE
─────────────────────────────────────────────────────
Sort + slice:   O(n log n) / O(n)
Heap size k:    O(n log k) / O(k)    ← default choice
QuickSelect:    O(n) avg  / O(log n) ← static, k-th only
Bucket sort:    O(n)      / O(n)     ← bounded integer scores
Binary search:  O(n log W)/ O(1)     ← counting function exists
─────────────────────────────────────────────────────

COMMON MISTAKES
─────────────────────────────────────────────────────
✗ Max-heap for k largest  (need min-heap — top is threshold, not answer)
✗ QuickSelect rank off-by-one: k-th largest = index n-k (0-indexed from small end)
✗ Index underflow: pivotIdx - 1 when pivotIdx == lo in QuickSelect
✗ Bucket array size n instead of n+1 (freq can equal n)
✗ Forgetting tiebreak in "top k frequent words" (alphabetical order)
✗ Using heap when k = 1 (just do max/min scan)
✗ Integer overflow: x²+y² can exceed int for large coordinates — use long
─────────────────────────────────────────────────────

STAFF-LEVEL SIGNALS TO DROP IN INTERVIEWS
─────────────────────────────────────────────────────
"Bucket sort is O(n) here because frequency is bounded by n"
"QuickSelect is O(n) avg but I'd add median-of-3 to avoid sorted-input worst case"
"For streaming data QuickSelect isn't viable — only the heap works online"
"At billion-scale: Count-Min Sketch for approximate, per-shard heaps + merge for exact"
"Binary search on answer works when we can count 'elements ≤ x' cheaply"
```
