# Time & Space Complexity Analysis

> **Tier 0 — Foundations** | No prerequisites | Required by: Every tier, every topic

---

## 1. Intuition First (No Jargon)

**The core idea:** Complexity analysis is your ability to predict how an algorithm's resource consumption (time and memory) scales as input grows. It's not about counting exact operations — it's about understanding the *growth curve*.

**Real-world analogy:** You're sizing EKS nodes for a tournament platform. You don't care that a single match takes 3.2ms to process. You care: if we go from 1K to 20K concurrent users, does processing time grow linearly (we need 20x nodes), quadratically (we need 400x — game over), or logarithmically (we barely notice)? That's complexity analysis applied to infrastructure.

**WHY this exists:** Without complexity analysis, you can't distinguish between an O(n) solution and an O(n²) solution until production melts. At 100 elements they both feel fast. At 10⁶ elements, one takes 1 second, the other takes 11.5 *days*. This is the single most important tool for eliminating bad approaches in the first 30 seconds of an interview.

**The mental shift:** Stop counting operations. Start classifying growth rates. "This loop runs n times, each iteration does O(1) work → O(n) total." That's the thinking pattern.

## 2. Pattern Recognition

### When to use

| Signal | Context |
|--------|---------|
| "What's the time complexity?" | Every interview problem |
| "Can you optimize this?" | You need to identify the current complexity first |
| "Will this scale?" | System design — same analysis, different units |
| "Why is this TLE?" | Competitive programming — your complexity exceeds the constraint |
| Choosing between two approaches | Compare growth rates, pick the better one |

### Common keywords in problems

- "Optimize", "efficient", "faster"
- "n can be up to 10⁵" (signals O(n log n) or better needed)
- "n can be up to 10³" (O(n²) is fine, O(n³) borderline)
- "TLE" (Time Limit Exceeded — your complexity is wrong)

### 10-Second Identification Rule

> **Look at the constraint on n. That tells you the maximum complexity you can afford.**
>
> - n ≤ 10 → O(n!) or O(2ⁿ) — brute force / backtracking
> - n ≤ 20 → O(2ⁿ) — bitmask DP, meet in the middle
> - n ≤ 500 → O(n³) — triple nested loop, Floyd-Warshall
> - n ≤ 5,000 → O(n²) — double nested loop, 2D DP
> - n ≤ 10⁵ → O(n log n) — sorting, binary search, segment tree
> - n ≤ 10⁶ → O(n) — single pass, sliding window, two pointers
> - n ≤ 10⁹ → O(log n) or O(√n) — binary search on answer, math

## 3. Core Algorithm / Template

### The Analysis Framework

There is no "algorithm" here — this is an analytical skill. But there IS a repeatable mental template.

### Step 1: Identify the dominant term

```csharp
// Example: What's the complexity?
public int[] TwoSum(int[] nums, int target)
{
    var map = new Dictionary<int, int>();           // O(1) init
    for (int i = 0; i < nums.Length; i++)            // O(n) iterations
    {
        int complement = target - nums[i];           // O(1) per iteration
        if (map.ContainsKey(complement))             // O(1) average (hash lookup)
            return new[] { map[complement], i };     // O(1)
        map[nums[i]] = i;                            // O(1) average (hash insert)
    }
    return Array.Empty<int>();
}
// Verdict: O(n) time, O(n) space
```

**Template for analysis:**

```
1. Count loops:
   - Single loop over n → O(n)
   - Nested loops over n → O(n²)
   - Loop where index doubles/halves → O(log n)

2. Identify work per iteration:
   - Constant operations → multiply by O(1)
   - Hash lookup/insert → O(1) amortized
   - Sorting inside a loop → multiply by O(n log n)
   - Recursive call → set up recurrence relation

3. Combine: Total = (iterations) × (work per iteration)

4. Drop constants and lower-order terms:
   - O(3n + 5) → O(n)
   - O(n² + n) → O(n²)
   - O(n log n + n) → O(n log n)
```

### Step 2: Amortized analysis (the interview differentiator)

```csharp
// This LOOKS like O(n²) but is actually O(n)
public int MinSubarrayLen(int target, int[] nums)
{
    int left = 0, sum = 0, minLen = int.MaxValue;

    for (int right = 0; right < nums.Length; right++)  // outer: O(n)
    {
        sum += nums[right];
        while (sum >= target)                           // inner: looks like O(n) per iteration
        {
            minLen = Math.Min(minLen, right - left + 1);
            sum -= nums[left];
            left++;                                     // BUT: left only moves forward, total moves ≤ n
        }
    }
    return minLen == int.MaxValue ? 0 : minLen;
}
// Key insight: left pointer moves at most n times TOTAL across all outer iterations
// Total work: O(n) for right + O(n) for left = O(n), not O(n²)
```

**The amortized argument:** When a pointer/index moves monotonically (only increases, never resets), the total work across all iterations is bounded by the total distance it can travel, not per-iteration work × iterations.

### Step 3: Recurrence relations (for recursive solutions)

```csharp
// Binary search: T(n) = T(n/2) + O(1)
// Each call halves the problem, does O(1) work
// → T(n) = O(log n)

// Merge sort: T(n) = 2T(n/2) + O(n)
// Two subproblems of half size, O(n) merge
// → T(n) = O(n log n)  (Master Theorem Case 2)

// Naive Fibonacci: T(n) = T(n-1) + T(n-2) + O(1)
// → T(n) = O(2^n)  (exponential — this is why DP exists)
```

### Master Theorem Quick Reference

For recurrences of the form `T(n) = aT(n/b) + O(nᵈ)`:

```
Let c = log_b(a)
- If d < c → T(n) = O(n^c)           [recursion dominates]
- If d = c → T(n) = O(n^d × log n)   [balanced]
- If d > c → T(n) = O(n^d)           [work-per-level dominates]

Examples:
  Binary Search:    a=1, b=2, d=0 → c=0, d=c → O(log n)
  Merge Sort:       a=2, b=2, d=1 → c=1, d=c → O(n log n)
  Strassen:         a=7, b=2, d=2 → c=2.807, d<c → O(n^2.807)
```

### Space Complexity Mental Model

```csharp
// What consumes space?

// 1. Auxiliary data structures
var set = new HashSet<int>(n);           // O(n)
var matrix = new int[n, n];              // O(n²)

// 2. Recursion call stack
void DFS(TreeNode node)                  // O(h) where h = tree height
{                                        // O(log n) balanced, O(n) skewed
    if (node == null) return;
    DFS(node.Left);                      // each call adds a stack frame
    DFS(node.Right);
}

// 3. Output (sometimes counted, sometimes not)
// Convention: if the problem asks you to RETURN a result,
// the output space is typically not counted as "extra" space
```

## 4. Complexity Deep Dive

### The hierarchy (memorize this cold)

```
O(1) < O(log n) < O(√n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!)
```

### Concrete numbers (for gut-checking in interviews)

| Complexity | n = 10 | n = 100 | n = 1,000 | n = 10⁶ | n = 10⁹ |
|-----------|--------|---------|-----------|---------|---------|
| O(1) | 1 | 1 | 1 | 1 | 1 |
| O(log n) | 3 | 7 | 10 | 20 | 30 |
| O(√n) | 3 | 10 | 32 | 1,000 | 31,623 |
| O(n) | 10 | 100 | 1,000 | 10⁶ | 10⁹ |
| O(n log n) | 33 | 664 | 10,000 | 2×10⁷ | 3×10¹⁰ |
| O(n²) | 100 | 10,000 | 10⁶ | 10¹² | 10¹⁸ |
| O(2ⁿ) | 1,024 | 10³⁰ | ∞ | ∞ | ∞ |

**The practical constraint:** Modern machines execute ~10⁸ simple operations per second. If n = 10⁶ and your solution is O(n²), that's 10¹² operations → ~2.7 hours. You need O(n log n) or better.

### Trade-offs

| Scenario | Time | Space | Why you'd pick this |
|----------|------|-------|-------------------|
| Hash map for lookups | O(1) amortized | O(n) | Speed matters, memory is cheap |
| Binary search (sorted array) | O(log n) | O(1) | Memory-constrained, data pre-sorted |
| Sort + two pointers | O(n log n) | O(1) or O(n) | Need ordering, space is tight |
| Brute force | O(n²) or worse | O(1) | n is tiny (≤ 100), correctness > speed |

### When this analysis breaks

- **Hash map is O(1) *amortized*, O(n) worst case.** In interviews this rarely matters, but in production (hash collision attacks, poor hash functions), it can.
- **Constant factors matter at scale.** O(n) with a constant of 100 is slower than O(n log n) with a constant of 1 for n < ~10⁵. Cache locality, branch prediction, and SIMD can make "slower" algorithms faster in practice.
- **Space complexity ignores memory hierarchy.** O(n) in L1 cache is radically different from O(n) hitting main memory. This is why array-based solutions often beat tree-based solutions despite identical Big-O.

## 5. Edge Cases & Pitfalls

### Common senior-level mistakes

1. **Ignoring hidden complexity in library calls.** `Array.Sort()` is O(n log n), `list.Contains()` is O(n), `hashSet.Contains()` is O(1). If you call `list.Contains()` inside a loop, that's O(n²), not O(n).

2. **Forgetting recursion stack space.** A recursive DFS on a tree uses O(h) stack space. On a skewed tree, h = n. "O(1) extra space" is wrong if you're recursing.

3. **Amortized confusion.** The nested while loop in sliding window is O(n) total, not O(n²). The key indicator: does the inner loop's pointer reset? If no → amortized. If yes → multiply.

4. **Confusing average case with worst case.** QuickSort is O(n log n) average, O(n²) worst. In interviews, state both. In production, use randomized pivot or IntroSort to guarantee O(n log n).

5. **Mixing up log bases.** O(log₂ n) = O(log₁₀ n) = O(ln n) in Big-O (they differ by a constant). But when asked "how many levels in a binary tree of n nodes" — that's specifically log₂ n.

6. **String operations are NOT O(1).** String concatenation in a loop is O(n) per concatenation in languages with immutable strings (C#, Java, Python). Use `StringBuilder` — this is an O(n) vs O(n²) difference that interviewers watch for.

### Debugging heuristics

If your solution gets TLE:
1. What's the constraint on n? Derive the maximum acceptable complexity.
2. What's your current complexity? Count your nested loops.
3. Is there a hidden O(n) inside a loop? (string comparison, list search, array copy)
4. Can you trade space for time? (hash map to eliminate a loop)

## 6. Variations (Pattern Stretching)

### Easy: Direct counting

Count loops, multiply by work per iteration. No subtlety.

```csharp
// O(n²) — obvious
for (int i = 0; i < n; i++)
    for (int j = 0; j < n; j++)
        DoSomething(); // O(1)
```

### Medium: Non-obvious loop bounds

```csharp
// What's the complexity?
for (int i = 0; i < n; i++)
    for (int j = i; j < n; j++)  // j starts at i, not 0
        DoSomething();

// Iterations: n + (n-1) + (n-2) + ... + 1 = n(n+1)/2 = O(n²)
// Still quadratic — the half doesn't change the growth rate
```

### Medium-Hard: Logarithmic patterns

```csharp
// O(n log n) — log factor from halving
for (int i = 0; i < n; i++)          // O(n)
{
    int j = n;
    while (j > 0)                     // O(log n) — halving
    {
        DoSomething();
        j /= 2;
    }
}
```

### Hard: Amortized and recurrence

```csharp
// Dynamic array (List<T>) — append is O(1) amortized
// Individual append: O(1) usually, O(n) when resize triggers
// n appends total: O(n) — because resizes are geometric (1, 2, 4, 8, ...)
// Total copy work: 1 + 2 + 4 + ... + n = 2n - 1 = O(n)
// Per operation amortized: O(n) / n = O(1)
```

### Evolution map

```
Counting loops → Non-obvious bounds → Logarithmic patterns
      ↓                                       ↓
  Amortized analysis              Recurrence relations
  (Sliding Window, Dynamic Array)  (Divide & Conquer, DP)
                        ↓
              Master Theorem application
```

## 7. Interview-Level Problems

### Problem 1 (Medium): Analyze the Complexity of Two Sum Approaches

**Summary:** Given an array and a target, find two numbers that sum to the target. Analyze the complexity of: (a) brute force nested loop, (b) sort + two pointers, (c) hash map.

**Expected approach:**
- (a) O(n²) time, O(1) space — two nested loops
- (b) O(n log n) time, O(1) space (or O(n) if sort is not in-place) — sort dominates
- (c) O(n) time, O(n) space — single pass with hash map

**Key insight:** This is THE canonical example of time-space trade-off. Each approach occupies a different point on the Pareto frontier. In interviews, explicitly state the trade-off: "I can trade O(n) space for an O(n) time improvement by using a hash map."

---

### Problem 2 (Medium-Hard): Complexity of Recursive Fibonacci vs DP

**Summary:** Analyze why naive recursive Fibonacci is O(2ⁿ) and how memoization brings it to O(n).

**Expected approach:**
- Draw the recursion tree: each call branches into 2 subproblems, depth is n → O(2ⁿ) nodes
- With memoization: each unique subproblem (0 to n) computed once → O(n) time, O(n) space
- Bottom-up DP: O(n) time, O(1) space (only need last two values)

**Key insight:** The recursion tree has massive overlap — `fib(3)` is computed exponentially many times. Memoization collapses the tree into a DAG. This is the foundational argument for ALL dynamic programming — if you can't explain this complexity transition fluently, DP problems will feel like magic.

---

### Problem 3 (Hard): Is This O(n) or O(n²)?

**Summary:** Given a sliding window solution with a nested while loop (e.g., minimum window substring), prove the time complexity is O(n).

**Expected approach:**
- Identify that the left pointer moves monotonically (never resets)
- Total left pointer movements across ALL outer loop iterations ≤ n
- Total right pointer movements = n
- Total work = O(n) + O(n) = O(n)

**Key insight:** The ability to perform amortized analysis on nested loops is a staff-level differentiator. The key question: "Does the inner loop's variable reset each iteration of the outer loop?" If no → total work is bounded by the variable's range, not per-iteration. This exact argument applies to sliding window, two pointers, and monotonic stack.

## 8. Staff-Level Thinking

### Trade-offs between approaches

The real interview skill isn't computing complexity — it's *choosing* between approaches with different complexity profiles:

| Decision | When O(n²) wins | When O(n log n) wins | When O(n) wins |
|----------|-----------------|---------------------|----------------|
| Implementation complexity | Simplest, lowest bug risk | Moderate | May require non-obvious insight |
| n ≤ 1000 | Always fine | Unnecessary optimization | Over-engineering |
| n ≤ 10⁵ | TLE | Sweet spot | Ideal if achievable |
| n ≤ 10⁶+ | Not viable | Borderline | Required |

### Scalability considerations

In production, Big-O is necessary but not sufficient:
- **Constant factors matter.** An O(n) algorithm with 100x overhead per element loses to O(n log n) with 1x overhead for n < 10⁵. This is why production sort algorithms (TimSort, IntroSort) hybridize.
- **Memory access patterns matter more than Big-O.** Array traversal (cache-friendly) is 10-100x faster than linked list traversal (cache-hostile) at the same Big-O.
- **Worst case vs. expected case.** Hash maps are O(1) expected, O(n) worst. In adversarial contexts (public-facing APIs), worst case matters. In internal tools, expected case is fine.

### When NOT to obsess over complexity

- **n is small and fixed.** If the input is always 50 states or 12 months, O(n³) is fine. Don't optimize what doesn't need optimizing.
- **I/O dominates.** If your algorithm processes data from a network call that takes 100ms, shaving 1ms off the algorithm is meaningless.
- **Correctness first.** In interviews, get the brute force working, state its complexity, THEN optimize. A correct O(n²) beats a buggy O(n).

### How to explain this in interviews

**Opening (5 seconds):** "The brute force is O(n²). Given n can be 10⁵, I need O(n log n) or better."

**Trade-off articulation (10 seconds):** "I can use a hash map to reduce to O(n) time at the cost of O(n) space. Given the constraints, that's a good trade-off."

**Amortized defense (when asked about nested loops):** "The inner pointer moves monotonically — it never resets. Total inner movements across all outer iterations is at most n. So the total is O(n), not O(n × n)."

## 9. Real-World Mapping

| Production scenario | Complexity relevance |
|---|---|
| **Database query optimization** | N+1 query in a loop = O(n) DB calls. Batch fetch = O(1) DB call. This is the #1 backend performance bug. |
| **API rate limiting** | Token bucket refill is O(1) per request. Sliding window log is O(n) for n requests in the window. At 10K RPS, this matters. |
| **Kafka consumer throughput** | O(1) per message processing → linear in throughput. O(n) per message (scanning all previous) → quadratic → consumer lag spiral. |
| **Cache sizing** | O(n) space cache for O(1) lookup. The trade-off is exactly time-space complexity. LRU eviction is O(1) with a hash map + doubly-linked list. |
| **Search infrastructure** | Inverted index lookup is O(1) per term. Full table scan is O(n). This is why you build indexes — trading O(n) space for O(1) time. |
| **EKS pod autoscaling** | If your algorithm is O(n²) and traffic doubles, you need 4x pods, not 2x. Complexity directly drives FinOps. |

**System design relevance:** In system design interviews, complexity analysis shows up as throughput estimation. "This service handles 10K RPS. Each request does O(log n) work on an index of 10M entries. That's ~23 operations per request × 10K = 230K operations/second — easily within a single-core budget." That's complexity analysis applied to capacity planning.

## 10. 10-Second Recall Cheat Sheet

```
CONSTRAINT → MAX COMPLEXITY:
  n ≤ 10      → O(n!) / O(2ⁿ)     backtracking, brute force
  n ≤ 20      → O(2ⁿ)             bitmask DP
  n ≤ 500     → O(n³)             Floyd-Warshall, triple loop
  n ≤ 5,000   → O(n²)             double loop, 2D DP
  n ≤ 10⁵     → O(n log n)        sort, binary search, segment tree
  n ≤ 10⁶     → O(n)              sliding window, two pointers, hashing
  n ≤ 10⁹     → O(log n) / O(√n)  binary search on answer, math

KEY PATTERNS:
  - Single loop → O(n)
  - Nested loop, inner resets → O(n²)
  - Nested loop, inner monotonic → O(n) amortized
  - Loop halving (i *= 2 or i /= 2) → O(log n)
  - Recursion splitting in half → Master Theorem

SPACE TRAPS:
  - Recursion stack = O(depth)
  - String concatenation in loop = O(n²) without StringBuilder
  - Hash map = O(n), array = O(n) — but array is cache-friendly

INTERVIEW MOVES:
  - State brute force complexity → derive target → optimize
  - "I'm trading O(n) space for O(n) time improvement"
  - Nested while loop? Check if inner pointer resets → amortized argument
```
