# Add vs Multiply in Time & Space Complexity

> **Tier 0 — Foundations** | Sub-topic of Time & Space Complexity Analysis
> Prerequisite for: Amortized analysis, Sliding Window (Tier 1), DP (Tier 4), all recursive problems

---

## 1. Intuition First (No Jargon)

**The single rule that explains everything:**

> **Sequential → ADD. Nested → MULTIPLY.**

That's it. Every complexity calculation in DSA reduces to knowing which one applies.

**Real-world analogy:**

You're processing 1,000 tournament match replays.

- **ADD scenario:** First you compress all replays (1s each = 1,000s), *then* you upload all replays (2s each = 2,000s). Total: 1,000 + 2,000 = 3,000s. The phases are *sequential* — they don't interact.

- **MULTIPLY scenario:** For *each* of the 1,000 replays, you compress it (1s), *and while compressing*, you check it against every previous replay for duplicates. Now for replay #500, you're doing 500 duplicate checks. Total work compounds. That's nested → multiply.

**WHY this matters for interviews:**

The #1 source of wrong complexity answers is misidentifying sequential code as nested or vice versa. Two `for` loops in a function are NOT always O(n²). Two `for` loops where one is *inside* the other ARE.

## 2. Pattern Recognition

### Decision Tree (10-second version)

```
Does block B start AFTER block A finishes?
  YES → ADD their complexities

Does block B execute INSIDE block A (per iteration)?
  YES → MULTIPLY their complexities

Is it recursion?
  Count subproblems × work per call level → often MULTIPLY
  (use recurrence relation to be sure)
```

### Input/output signals

| Code shape | Rule | Example |
|---|---|---|
| Two separate loops, same scope | ADD | Sort + scan |
| Inner loop inside outer loop | MULTIPLY | Nested iteration |
| Function call inside a loop | MULTIPLY | Binary search per element |
| Two recursive calls, each at same level | ADD the branches | Merge sort subproblems |
| Work done at each recursion level | MULTIPLY levels × per-level work | Merge sort merge step |
| Multiple independent allocations | ADD (space) | HashMap + array |
| Recursion depth × frame size | MULTIPLY (space) | DFS stack frames |
| 2D DP table | MULTIPLY (space) | rows × cols |

### Common keywords that signal ADD

- "after", "then", "followed by", "first... then"
- Sort the array, *then* process it
- Build the map, *then* query it
- Preprocess, *then* answer queries

### Common keywords that signal MULTIPLY

- "for each", "per element", "every time"
- "inside the loop", nested structure
- Recursive branching with work at each level

## 3. Core Algorithm / Template

### The ADD Template

```rust
fn solve(nums: &[i32]) -> i32 {
    // Phase 1: O(n log n) — runs completely before Phase 2
    let mut sorted = nums.to_vec();
    sorted.sort();                          // O(n log n)

    // Phase 2: O(n) — starts after Phase 1 finishes
    let mut result = 0;
    for &x in &sorted {                     // O(n)
        result += x;
    }

    result
}
// Total: O(n log n) + O(n) = O(n log n)
// The O(n) is dominated — drop it in Big-O
```

**Mental model for ADD:** Draw a timeline. The operations occupy *different time slots*. They don't overlap or nest. Sum them, then drop dominated terms.

```
Timeline: [---- Phase 1: O(n log n) ----][-- Phase 2: O(n) --]
Total cost = n log n + n = O(n log n)
```

### The MULTIPLY Template

```rust
fn solve(nums: &[i32]) -> Vec<i32> {
    let n = nums.len();
    let mut result = Vec::new();

    for i in 0..n {                         // outer: O(n) iterations
        // This block executes FOR EACH i → multiply
        let mut j = n;
        while j > 0 {                       // inner: O(log n) per outer iteration
            result.push(nums[i] + j as i32);
            j /= 2;
        }
    }
    result
}
// Total: O(n) × O(log n) = O(n log n)
```

**Mental model for MULTIPLY:** Inner work is *repeated* for every outer iteration. If outer runs X times and inner runs Y times per outer iteration, total = X × Y.

```
Outer iteration 1: [inner: Y operations]
Outer iteration 2: [inner: Y operations]
...
Outer iteration X: [inner: Y operations]
Total = X × Y
```

### The Recursion Template

```rust
fn merge_sort(arr: &mut Vec<i32>, lo: usize, hi: usize) {
    if lo >= hi { return; }
    let mid = lo + (hi - lo) / 2;

    merge_sort(arr, lo, mid);       // T(n/2) — left half
    merge_sort(arr, mid + 1, hi);   // T(n/2) — right half
    merge(arr, lo, mid, hi);        // O(n) — merge work at this level
}
// Recurrence: T(n) = 2·T(n/2) + O(n)
// Per level total work: O(n)  [at every depth level, merging = O(n) total]
// Number of levels: O(log n)
// Total: O(n) × O(log n) = O(n log n)  ← MULTIPLY here
```

**Recursion rule:** Work per level of the recursion tree × number of levels.

- If work per level is constant across all levels → just multiply by number of levels
- If work per level varies → sum across levels (this becomes Master Theorem territory)

## 4. Complexity Deep Dive

### ADD: When to drop terms

When you add two complexities, the result is dominated by the larger one *only if they share the same variable*:

```
O(n²) + O(n)     = O(n²)          ← n² dominates, safe to drop n
O(n log n) + O(n) = O(n log n)    ← n log n dominates
O(n) + O(n)      = O(n)           ← 2n = O(n), constant factor drops
O(n) + O(m)      = O(n + m)       ← KEEP BOTH if n and m are independent vars
```

**Critical trap:** If `n` and `m` are *different independent variables*, you cannot drop either. `O(n + m)` stays as is.

```rust
// O(n + m) — both terms must stay
fn two_arrays(a: &[i32], b: &[i32]) {
    for x in a { println!("{x}"); }     // O(n)
    for x in b { println!("{x}"); }     // O(m)
}
// n = a.len(), m = b.len() — independent → O(n + m)
// If you know m ≤ n always, then O(n + m) = O(n). But don't assume.
```

### MULTIPLY: When inner work varies

The classic "nested but inner is shorter" trap:

```rust
// What's the complexity of THIS?
for i in 0..n {
    for j in i..n {    // j starts at i, not 0
        // O(1) work
    }
}
// Iterations: n + (n-1) + (n-2) + ... + 1 = n(n+1)/2 = O(n²)
// Still O(n²) — the half is a constant factor, ignored in Big-O
```

And the case where inner work is genuinely bounded:

```rust
// SLIDING WINDOW — looks O(n²), actually O(n)
fn max_subarray_of_size_k(nums: &[i32], k: usize) -> i32 {
    let (mut left, mut sum, mut max_sum) = (0, 0, i32::MIN);

    for right in 0..nums.len() {         // outer: O(n) iterations
        sum += nums[right];
        while right - left + 1 > k {     // inner: looks like O(n) per iteration
            sum -= nums[left];
            left += 1;                   // BUT: left only ever moves forward
        }
        max_sum = max_sum.max(sum);
    }
    max_sum
}
// left moves at most n times TOTAL across ALL outer iterations
// right moves exactly n times
// Total: O(n) + O(n) = O(n)  ← ADD, not multiply
// Key: the inner loop's variable (left) does NOT reset each outer iteration
```

**The reset test:** Does the inner loop variable reset to 0 (or its starting value) at the beginning of each outer iteration?
- **YES → MULTIPLY** (inner runs fully each time)
- **NO → ADD the total inner work** (amortized)

### Space: ADD vs MULTIPLY

```rust
fn example(n: usize, m: usize) {
    let map: std::collections::HashMap<i32, i32> = HashMap::new(); // O(n) space
    let arr: Vec<i32> = vec![0; m];                                // O(m) space
    // Both exist simultaneously → O(n + m) space  [ADD]
}
```

```rust
fn dfs(node: usize, graph: &Vec<Vec<usize>>, visited: &mut Vec<bool>, depth: usize) {
    // Each stack frame holds: node (O(1)), local vars (O(1))
    // Max recursion depth: O(n) in worst case (linear graph)
    // Space per frame: O(1)
    // Total stack space: O(n) × O(1) = O(n)  [MULTIPLY depth × frame size]
    visited[node] = true;
    for &next in &graph[node] {
        if !visited[next] {
            dfs(next, graph, visited, depth + 1);
        }
    }
}
```

```rust
fn dfs_with_path(
    node: usize,
    graph: &Vec<Vec<usize>>,
    visited: &mut Vec<bool>,
    path: &mut Vec<usize>,   // ← each frame passes a growing path
) {
    // Stack depth: O(n), each frame references path of size O(n)
    // Space: O(n) depth × O(n) per frame = O(n²) in pathological case
    // (if you're CLONING path per frame — this is the gotcha)
    path.push(node);
    visited[node] = true;
    for &next in &graph[node] {
        if !visited[next] {
            dfs_with_path(next, graph, visited, path);
        }
    }
    path.pop();
    // Using a shared mutable ref (not cloning) → O(n) stack + O(n) path = O(n) [ADD]
}
```

### The BFS O(V + E) explanation — a common confusion point

```rust
fn bfs(graph: &Vec<Vec<usize>>, start: usize) {
    let n = graph.len();
    let mut visited = vec![false; n];
    let mut queue = std::collections::VecDeque::new();
    queue.push_back(start);
    visited[start] = true;

    while let Some(node) = queue.pop_front() {   // each node dequeued once: O(V) total
        for &next in &graph[node] {              // edges processed per node
            if !visited[next] {
                visited[next] = true;
                queue.push_back(next);
            }
        }
    }
}
// Total edge processing across ALL node dequeues:
//   Node 0's neighbors: degree(0) iterations
//   Node 1's neighbors: degree(1) iterations
//   ...
//   Sum = degree(0) + degree(1) + ... + degree(n-1) = 2E (each edge counted twice)
// Total: O(V) for dequeues + O(E) for edge processing = O(V + E)  [ADD]
//
// WHY NOT MULTIPLY? Because each edge is processed exactly once total,
// not once per vertex. It's O(V) + O(E), not O(V × avg_degree).
// avg_degree × V = E — so both formulations give the same answer,
// but the ADD framing is clearer.
```

## 5. Edge Cases & Pitfalls

### Pitfall 1: Two loops = O(n²)? NO.

```rust
// WRONG mental model:
// "There are two loops, so it's O(n²)"

fn process(nums: &[i32]) {
    for x in nums { /* O(n) */ }     // loop 1
    for x in nums { /* O(n) */ }     // loop 2 — NOT inside loop 1
}
// Correct: O(n) + O(n) = O(n), not O(n²)
// O(n²) requires one loop INSIDE the other
```

### Pitfall 2: Function call inside loop — must check its cost

```rust
fn bad_contains(nums: &[i32], queries: &[i32]) -> Vec<bool> {
    let mut results = Vec::new();
    for &q in queries {                      // O(n) iterations
        results.push(nums.contains(&q));     // O(n) each — Vec::contains is linear scan
    }
    results
}
// Total: O(n) × O(n) = O(n²)  [MULTIPLY — function runs inside loop]
// Fix: build a HashSet first (O(n)), then query in O(1) each → O(n) total
```

### Pitfall 3: Forgetting that "sequential recursive calls" ADD, not multiply

```rust
fn fibonacci_naive(n: u64) -> u64 {
    if n <= 1 { return n; }
    fibonacci_naive(n - 1) + fibonacci_naive(n - 2)
}
// T(n) = T(n-1) + T(n-2) + O(1)
// Both branches happen FOR EACH call → you'd think ADD
// BUT: the branching causes exponential blowup
// The ADD is at the recurrence level, not the overall growth
// Each call spawns 2 more → tree has ~2^n nodes total → O(2^n)
//
// Rule: At the recurrence level you ADD the subproblem costs
// but the TREE STRUCTURE (how many nodes) drives the actual complexity
```

### Pitfall 4: Space in recursion — are you allocating per frame?

```rust
// O(log n) space — each frame O(1)
fn binary_search(arr: &[i32], target: i32, lo: usize, hi: usize) -> Option<usize> {
    if lo > hi { return None; }
    let mid = lo + (hi - lo) / 2;
    match arr[mid].cmp(&target) {
        std::cmp::Ordering::Equal => Some(mid),
        std::cmp::Ordering::Less => binary_search(arr, target, mid + 1, hi),
        std::cmp::Ordering::Greater => binary_search(arr, target, lo, mid - 1),
    }
}
// Call depth: O(log n), frame size: O(1) → O(log n) × O(1) = O(log n) space
```

### Pitfall 5: The "halving" confusion — log n × n

```rust
// For each element (O(n)), do binary search (O(log n))
fn pair_exists(nums: &mut Vec<i32>, target: i32) -> bool {
    nums.sort();                                // O(n log n)
    for &x in nums.iter() {                    // O(n) iterations
        let complement = target - x;
        if nums.binary_search(&complement).is_ok() {  // O(log n) per iteration
            return true;
        }
    }
    false
}
// Sort: O(n log n)
// Loop + binary search: O(n) × O(log n) = O(n log n)  [MULTIPLY]
// Total: O(n log n) + O(n log n) = O(n log n)  [ADD — same term, factor of 2 drops]
```

## 6. Variations (Pattern Stretching)

### Easy → Medium → Hard progression

**Easy: Direct sequential blocks**

```rust
fn easy(arr: &mut Vec<i32>) -> i32 {
    arr.sort();                     // O(n log n)
    arr.iter().sum()                // O(n)
}
// O(n log n) + O(n) = O(n log n) ← trivial ADD
```

**Medium: Identifying nested vs sequential in disguise**

```rust
fn medium(matrix: &Vec<Vec<i32>>) -> i32 {
    let (rows, cols) = (matrix.len(), matrix[0].len());
    let mut sum = 0;

    for row in matrix {              // O(rows)
        for &val in row {            // O(cols) PER row → MULTIPLY
            sum += val;
        }
    }
    sum
}
// O(rows × cols) — MULTIPLY
// NOT O(rows + cols) — the inner loop runs cols times FOR EACH of the rows iterations
```

```rust
fn medium_tricky(matrix: &Vec<Vec<i32>>) -> i32 {
    let rows = matrix.len();
    let mut row_sums = vec![0i32; rows];

    for (i, row) in matrix.iter().enumerate() {
        row_sums[i] = row.iter().sum();          // O(cols) per row
    }
    // phase 1: O(rows × cols)

    row_sums.sort();                             // O(rows log rows)
    // phase 2: O(rows log rows)

    row_sums.iter().sum()                        // O(rows)
    // phase 3: O(rows)
}
// Total: O(rows × cols) + O(rows log rows) + O(rows)
// = O(rows × cols) + O(rows log rows)
// If cols >> log(rows): O(rows × cols)
// If cols << log(rows): O(rows log rows)
// → Leave as O(rows × cols + rows log rows) when variables are independent
```

**Hard: Recursion with per-level work accumulation**

```rust
fn quick_sort(arr: &mut Vec<i32>, lo: usize, hi: usize) {
    if lo >= hi { return; }
    let pivot = partition(arr, lo, hi);         // O(hi - lo) work
    if pivot > 0 {
        quick_sort(arr, lo, pivot - 1);         // T(pivot - lo)
    }
    quick_sort(arr, pivot + 1, hi);             // T(hi - pivot)
}
// Best/avg case: pivot splits evenly
//   T(n) = 2T(n/2) + O(n)
//   Per level work: O(n) (spread across all subproblems at that level)
//   Levels: O(log n)
//   Total: O(n) × O(log n) = O(n log n)  ← MULTIPLY levels × per-level work
//
// Worst case (already sorted, pivot always min/max):
//   T(n) = T(n-1) + O(n)
//   Level 1: O(n) work, Level 2: O(n-1) work, ... → sum = n(n+1)/2 = O(n²)
//   Here the per-level work DECREASES — can't simply multiply
//   Must SUM the series: O(n) + O(n-1) + ... + O(1) = O(n²)
```

**The recursion decision:**

```
Uniform work per level, log n levels → MULTIPLY: O(n log n)
Decreasing work per level, n levels → SUM the series: typically O(n²) or O(n log n)
Set up recurrence → apply Master Theorem or expansion
```

## 7. Interview-Level Problems

### Problem 1 (Medium): Matrix Diagonal Traversal

**Summary:** Given an n×m matrix, traverse it in diagonal order. What's the time and space complexity?

**Expected approach:**
- Time: O(n×m) — every element visited exactly once [MULTIPLY for total elements, but ADD for traversal cost since each element processed once]
- Space: O(min(n,m)) for the diagonal buffer, OR O(1) if done in-place
- The traversal iterates over (n+m-1) diagonals; elements per diagonal form a triangle — SUM the series = O(n×m)

**Key insight:** The complexity is O(n×m), not O((n+m)²). Total elements = n×m regardless of traversal order. "Diagonal" doesn't change total work, only order. Interviewers probe whether you correctly apply MULTIPLY for total matrix elements vs misapply ADD.

---

### Problem 2 (Medium-Hard): K Closest Points using Sorting vs Heap

**Summary:** Given n points and integer k, find k closest to origin. Compare three approaches and derive complexities.

**Expected approach:**
- **Sort all:** O(n log n) sort + O(k) extract = O(n log n + k) = O(n log n) [ADD]
- **Max-heap of size k:** For each of n elements, heap push/pop = O(log k). Total: O(n) × O(log k) = O(n log k) [MULTIPLY]
- **QuickSelect:** O(n) average [amortized — partition doesn't nest uniformly]

**Key insight:** The heap approach is O(n log k) not O(n log n) because the heap is bounded at size k. The variable inside the log matters — it's the heap size, not the input size. This is a MULTIPLY where the second factor depends on k, not n.

---

### Problem 3 (Hard): Complexity of Backtracking — Subsets with Constraint

**Summary:** Generate all subsets of an array of size n where sum ≤ target. What's the time complexity?

**Expected approach:**
- Worst case (target = ∞, all elements valid): 2^n subsets, each takes O(n) to construct → O(n × 2^n) [MULTIPLY: for each of 2^n subsets, O(n) work to build it]
- With constraint: harder to bound exactly, but still O(n × 2^n) worst case
- Space: O(n) for recursion stack + O(n) for current subset being built = O(n) [ADD for the two allocations, they're at the same depth]

**Key insight:** In backtracking, time = (number of leaf nodes in recursion tree) × (work per leaf) + (work at internal nodes). For subsets, 2^n leaves × O(n) work per leaf = O(n × 2^n). The MULTIPLY is between the branching factor raised to depth and the work done.

## 8. Staff-Level Thinking

### Trade-offs: ADD vs MULTIPLY in algorithm design

The entire field of algorithm optimization is about **converting MULTIPLY to ADD**:

| Original (MULTIPLY → expensive) | Optimized (ADD → cheaper) | Technique |
|---|---|---|
| For each element, scan all others: O(n²) | Build hash set O(n), then query O(n): O(n + n) = O(n) | Preprocessing |
| For each element, sort remaining: O(n² log n) | Sort once O(n log n), then scan O(n): O(n log n + n) = O(n log n) | Sort once, reuse |
| Recompute prefix sum each query: O(n) per query × Q queries = O(nQ) | Build prefix sum O(n), answer each in O(1): O(n + Q) | Precomputation |
| DFS that clones path at every level: O(n × 2^n) space | Pass path by mut ref + push/pop: O(n) stack + O(n) path = O(n) | Reference sharing |

**The algorithmic insight:** Whenever you see MULTIPLY with a large first term (like O(n)), ask: "Can I compute this first term once (O(n)) and reuse the result for all iterations of the second term?" That's the entire logic behind hashing, prefix sums, memoization, and precomputation.

### Scalability: How ADD/MULTIPLY drives system design

```
At 20K CCU, a service handling match events:

Bad design:  For each of N events, scan all M active sessions → O(N × M)
             N = 50K events/sec, M = 20K sessions
             = 1 billion operations/sec — 10 cores won't save you

Good design: Build a session map O(M) at startup, then lookup per event O(1)
             N events × O(1) lookup = O(N) + O(M) total → ADD
             = 50K operations/sec — easily handled

This is the MULTIPLY→ADD optimization in production.
Every N+1 query problem in backend is this pattern.
```

### When the ADD/MULTIPLY rule gets subtle

**1. Amortized operations:** Dynamic array `.push()` is O(1) amortized. N pushes = O(N), not O(N²). The occasional O(N) resize is *spread* across N operations. Treat as O(1) per operation when amortized analysis applies.

**2. Memoized recursion:** Naive Fibonacci is O(2^n) (MULTIPLY effect of branching). Memoized: O(N) unique states × O(1) work per state = O(N) [MULTIPLY becomes small when states are bounded].

**3. Path compression in Union-Find:** `find()` is O(α(n)) amortized — nearly O(1). N operations = O(N × α(n)) ≈ O(N). The MULTIPLY has a nearly-constant second factor.

### How to explain in interviews (the verbal formula)

> "This outer loop runs O(n) times. For each iteration, I'm doing [X]. Does [X] reset its state each iteration? Yes → multiply: O(n × cost(X)). No → amortize: total inner work is bounded by [range], so it's O(n + range) at most."

> "These are two sequential passes over the data. The first costs O(f(n)), the second costs O(g(n)). They don't interact. Total is O(f(n) + g(n)) = O(max(f,g)) if n is the same variable."

> "For the recursion: I make A calls at each level, each doing O(W) work, across O(D) levels. Total = A^D nodes in tree × O(W) work per node — but that overcounts. The correct framing is: total work per level × number of levels. Per level: O(n) spread across all calls. Levels: O(log n). Total: O(n log n)."

## 9. Real-World Mapping

| Production pattern | ADD or MULTIPLY? | Implication |
|---|---|---|
| **N+1 query in ORM** (query per record in loop) | MULTIPLY: O(N × DB_latency) | Kill it. Batch fetch → ADD: O(1 batch) + O(N processing) |
| **DynamoDB scan per user request** | MULTIPLY: O(requests × table_size) | Add GSI, reduce to O(1) per lookup → MULTIPLY by O(1) = O(requests) |
| **Kafka consumer**: process + checkpoint | ADD: O(process) + O(checkpoint) | Sequential phases; checkpoint doesn't nest inside process |
| **JWT validation per request** (parse + verify) | ADD: O(parse) + O(verify) | Both are O(1) ops, adds to O(1). No nesting. |
| **Cache miss → DB fallback** (cache-aside) | ADD: O(cache_check) + O(db_query) | Sequential; the ADD is fine. MULTIPLY would mean DB query per cache item. |
| **Grafana metric aggregation** (sum over N pods × M metrics) | MULTIPLY: O(N × M) | At 1000 pods × 500 metrics = 500K operations per scrape. Cardinality problem. |
| **Search: inverted index vs full scan** | O(1) lookup vs O(n) per query | ADD query_count × O(1) = O(Q) vs MULTIPLY query_count × O(n) = O(Q×n) |

**The FinOps angle:**

```
Lambda cost ∝ execution time × invocations
If your Lambda does O(n) per request and you get R requests/sec:
  Total cost ∝ R × O(n) — MULTIPLY

Optimize the Lambda to O(1) with a cache hit:
  Total cost ∝ R × O(1) + O(n) for cache build — ADD

This is why ElastiCache/Valkey exists as a cost lever, not just a latency lever.
```

## 10. 10-Second Recall Cheat Sheet

```
THE RULE:
  Sequential blocks    → ADD    their complexities
  Nested (one inside)  → MULTIPLY their complexities
  Recursion per-level  → MULTIPLY levels × per-level-work

ADD — common shapes:
  [loop A] then [loop B]              → O(A) + O(B)
  sort() + scan()                     → O(n log n) + O(n) = O(n log n)
  BFS: vertices + edges               → O(V) + O(E) = O(V + E)
  Two separate allocations            → O(n) + O(m) [space]
  DROP dominated terms if same var    → O(n² + n) = O(n²)
  KEEP both if independent vars       → O(n + m) stays

MULTIPLY — common shapes:
  outer O(n), inner O(k)              → O(n × k)
  for each element, binary search     → O(n log n)
  recursion: D levels × W per level   → O(D × W)
  2D grid: rows × cols                → O(rows × cols) [time + space]
  recursion stack: depth × frame      → O(depth × frame_size) [space]

AMORTIZED — the ADD exception inside MULTIPLY shape:
  Inner loop variable doesn't reset?  → ADD total inner work, not multiply
  Test: does inner var reset per outer iter?
    YES → multiply  |  NO → amortize (add)

CONVERT MULTIPLY→ADD (optimization signal):
  For each query, scan all data      → Precompute once + O(1) lookup per query
  Recompute inside loop              → Hoist computation before loop
  Clone data structure per recursion → Pass by mut ref + undo (push/pop)

INTERVIEW PHRASE:
  "These two loops are sequential → ADD → O(n + n) = O(n)"
  "This loop is nested → MULTIPLY → O(n × log n)"
  "Inner pointer doesn't reset → amortized → O(n) total for inner"
  "Two independent allocations → ADD → O(n + m) space"
  "Recursion: depth log n, O(1) per frame → O(log n) stack space"
```
