## 1. Intuition First (No Jargon)

**The core idea:** Binary search doesn't find things — it **eliminates half the search space in every step**. The trick is knowing *which half to eliminate* and *why you can safely discard it*.

**Real-world analogy:**

You're looking for a name in a physical phone book (remember those). You open to the middle. If the name you want comes *after* the middle name alphabetically, you tear out and discard the first half — you've just eliminated 500 pages in one look. Repeat until one page remains. 1,000 pages → 10 looks. 1,000,000 pages → 20 looks. That's O(log n).

**But here's the deeper insight most people miss:**

Binary search works on any *monotonic predicate* — not just sorted arrays. A predicate `P(x)` is monotonic if there exists some threshold `T` such that `P(x) = false` for all `x < T` and `P(x) = true` for all `x ≥ T` (or vice versa). Binary search finds that threshold `T` in O(log n).

```
Classic binary search (sorted array):
  [false, false, false, TRUE, TRUE, TRUE, TRUE]
   P(x) = "arr[x] >= target"
   Find the first TRUE.

Binary search on answer:
  [false, false, false, TRUE, TRUE, TRUE, TRUE]
   P(x) = "can we complete the task with capacity x?"
   Find the minimum x where P(x) = true.
```

Both are the same algorithm. The only difference is what `P(x)` means.

**WHY this pattern exists:**

Linear search is O(n). If data has no structure, that's the best you can do. But sorted order (or any monotonic structure) is an **irreversibility guarantee**: once you cross a threshold, values only go one direction. This irreversibility lets you eliminate half the remaining candidates with every single comparison. That's the information-theoretic argument for O(log n): each comparison gives you 1 bit, and log₂(n) bits distinguishes n possibilities.

## 2. Pattern Recognition

### Two modes of binary search

| Mode | Question | Example |
|---|---|---|
| **Classic** | Where is value X in sorted data? | Find index of target, count occurrences, find insertion point |
| **On Answer** | What is the minimum/maximum X such that condition holds? | Min speed, max capacity, min days, kth smallest |

### When to use

**Classic:**
- Input is **sorted** (or structured with monotonic property)
- Need to find exact position, or first/last occurrence
- Need O(log n) lookup instead of O(n) scan

**On Answer:**
- Problem asks for **minimum** or **maximum** of some value
- There's a clear predicate: "is this value *feasible*?"
- Feasibility is **monotonic**: if X is feasible, X+1 is also feasible (or vice versa)
- Computing feasibility for a given X takes O(n) or O(n log n)

### Input/output signals

```
Classic signals:
  "sorted array" + "find / search / locate"
  "find first occurrence where..."
  "find insertion position"
  "count elements in range [lo, hi]"

On-Answer signals:
  "minimum [speed/days/capacity/size] such that..."
  "maximum [k/units/workers] such that..."
  "what is the minimum X that satisfies condition?"
  "is it possible to..." with a numeric parameter
  answer lies in a range and condition is monotonic
```

### Common keywords

```
Classic:
  "sorted array", "find target", "first/last position",
  "count of elements equal to", "search in rotated array",
  "find peak element", "search insert position"

On Answer:
  "minimum time / speed / days / capacity"
  "maximum k / pages / jobs / chunks"
  "split array into k parts to minimize maximum"
  "allocate books", "ship packages in D days"
  "Koko eating bananas"
  "kth smallest in matrix / sorted arrays"
```

### 10-second identification

> **Classic:** "Is the input sorted? Can I express 'find X' as finding a boundary between false/true regions?"
>
> **On Answer:** "Is the problem asking for a minimum or maximum numeric answer? If I guess the answer X, can I check feasibility in O(n)? Does feasibility flip exactly once as X increases?"
>
> YES to both → binary search on answer. Every greedy feasibility check + monotonic answer range = binary search.

## 3. Core Algorithm / Template

### The Universal Framework

Every binary search finds the **first position where a predicate flips from false to true**.

```
Search space:  [lo .......................... hi]
Predicate P:   [false, false, ..., false, TRUE, TRUE, ..., TRUE]
Goal:          find the index of the first TRUE
```

This single framework unifies all variants.

### Template 1 — Classic Exact Match

```rust
fn binary_search_exact(arr: &[i32], target: i32) -> Option<usize> {
    let (mut lo, mut hi) = (0usize, arr.len()); // half-open: [lo, hi)

    while lo < hi {
        let mid = lo + (hi - lo) / 2;           // prevent overflow
        match arr[mid].cmp(&target) {
            std::cmp::Ordering::Equal => return Some(mid),
            std::cmp::Ordering::Less => lo = mid + 1,  // target is right of mid
            std::cmp::Ordering::Greater => hi = mid,   // target is left of mid (or at mid)
        }
    }
    None  // not found
}
// Use Rust built-in instead:
// arr.binary_search(&target) → Ok(idx) if found, Err(insertion_point) if not
```

### Template 2 — Lower Bound (First position where arr[i] >= target)

```rust
// "Find first index where condition is true"
// Returns arr.len() if all elements < target
fn lower_bound(arr: &[i32], target: i32) -> usize {
    let (mut lo, mut hi) = (0usize, arr.len()); // [lo, hi) half-open

    while lo < hi {
        let mid = lo + (hi - lo) / 2;
        if arr[mid] < target {
            lo = mid + 1;  // mid is definitely NOT the answer, exclude it
        } else {
            hi = mid;      // mid COULD be the answer, keep it in range
        }
    }
    lo  // first index where arr[i] >= target
}

// Upper bound: first position where arr[i] > target
fn upper_bound(arr: &[i32], target: i32) -> usize {
    let (mut lo, mut hi) = (0usize, arr.len());
    while lo < hi {
        let mid = lo + (hi - lo) / 2;
        if arr[mid] <= target {
            lo = mid + 1;
        } else {
            hi = mid;
        }
    }
    lo  // first index where arr[i] > target
}

// Count of target in sorted array
fn count_occurrences(arr: &[i32], target: i32) -> usize {
    upper_bound(arr, target) - lower_bound(arr, target)
}

// Rust idiomatic equivalents:
// lower_bound = arr.partition_point(|&x| x < target)
// upper_bound = arr.partition_point(|&x| x <= target)
```

**Mental model for lower/upper bound:**

```
Invariant:
  Everything in [0, lo) has predicate = false
  Everything in [hi, n) has predicate = true
  [lo, hi) is the unresolved range

Termination: lo == hi → lo is the first TRUE position
```

### Template 3 — Binary Search on Answer

```rust
// Generic template: find MINIMUM x in [lo, hi] where predicate(x) is true
fn binary_search_on_answer<F>(mut lo: i64, mut hi: i64, predicate: F) -> i64
where
    F: Fn(i64) -> bool,
{
    while lo < hi {
        let mid = lo + (hi - lo) / 2;
        if predicate(mid) {
            hi = mid;       // mid works, but maybe something smaller also works
        } else {
            lo = mid + 1;   // mid doesn't work, need at least mid+1
        }
    }
    lo  // minimum value where predicate is true
}

// For MAXIMUM x where predicate is true (flip the predicate):
// predicate: "is x feasible?" → find max feasible
fn binary_search_max<F>(mut lo: i64, mut hi: i64, predicate: F) -> i64
where
    F: Fn(i64) -> bool,
{
    while lo < hi {
        let mid = lo + (hi - lo + 1) / 2; // ceiling division — prevents infinite loop
        if predicate(mid) {
            lo = mid;       // mid works, maybe something larger also works
        } else {
            hi = mid - 1;   // mid doesn't work
        }
    }
    lo  // maximum value where predicate is true
}
```

**The ceiling division trap:**

When searching for the maximum (lo side moves), if `lo = mid` and `hi - lo = 1`, then `mid = lo + (hi-lo)/2 = lo` → `lo = mid = lo` → infinite loop. Fix: use ceiling division `mid = lo + (hi - lo + 1) / 2`.

```
Minimum (hi moves):  mid = lo + (hi - lo) / 2     — floor, OK
Maximum (lo moves):  mid = lo + (hi - lo + 1) / 2  — ceiling, required
```

### Template 4 — Concrete Example: Koko Eating Bananas

```rust
fn min_eating_speed(piles: &[i32], h: i32) -> i32 {
    // Predicate: can Koko eat all bananas in h hours at speed k?
    let can_finish = |k: i64| -> bool {
        let hours: i64 = piles.iter()
            .map(|&p| (p as i64 + k - 1) / k)  // ceiling division: hours for pile p
            .sum();
        hours <= h as i64
    };

    // Binary search on answer: minimum k where can_finish(k) is true
    let (mut lo, mut hi) = (1i64, *piles.iter().max().unwrap() as i64);
    while lo < hi {
        let mid = lo + (hi - lo) / 2;
        if can_finish(mid) {
            hi = mid;       // mid works, try smaller
        } else {
            lo = mid + 1;   // mid too slow, need faster
        }
    }
    lo as i32
}
// Time: O(n log(max_pile)) — O(n) feasibility check × O(log max) search space
```

### Template 5 — Binary Search on Floats

```rust
// Binary search on real-valued answer
fn binary_search_float<F>(mut lo: f64, mut hi: f64, predicate: F) -> f64
where
    F: Fn(f64) -> bool,
{
    for _ in 0..100 {  // 100 iterations gives ~10^-30 precision
        let mid = (lo + hi) / 2.0;
        if predicate(mid) {
            hi = mid;
        } else {
            lo = mid;
        }
    }
    lo  // converged value
}

// Alternative: iterate until precision is sufficient
fn binary_search_float_eps<F>(mut lo: f64, mut hi: f64, predicate: F, eps: f64) -> f64
where
    F: Fn(f64) -> bool,
{
    while hi - lo > eps {
        let mid = (lo + hi) / 2.0;
        if predicate(mid) { hi = mid; }
        else { lo = mid; }
    }
    lo
}
// Use 100 iterations over eps — avoids infinite loops from floating point imprecision
```

### Rust Built-in Reference

```rust
// Exact search
v.binary_search(&target)                    // Ok(idx) or Err(insertion_point)
v.binary_search_by(|x| x.cmp(&target))     // custom comparator
v.binary_search_by_key(&key, |x| x.field)  // by key

// Partition point (lower/upper bound)
v.partition_point(|&x| x < target)         // lower bound: first idx where x >= target
v.partition_point(|&x| x <= target)        // upper bound: first idx where x > target

// Decode binary_search Result
match v.binary_search(&target) {
    Ok(idx) => { /* found at idx */ }
    Err(idx) => { /* not found; idx = insertion point */ }
}

// Count occurrences in sorted vec
let lo = v.partition_point(|&x| x < target);
let hi = v.partition_point(|&x| x <= target);
let count = hi - lo;
```

## 4. Complexity Deep Dive

### Classic Binary Search

| | Time | Space |
|---|---|---|
| Exact search | O(log n) | O(1) |
| Lower / upper bound | O(log n) | O(1) |
| Count occurrences | O(log n) | O(1) |
| Search in 2D sorted matrix | O(log(nm)) | O(1) |

**Why exactly O(log n):** Each iteration, the search space `[lo, hi)` is halved. Starting from size n: after k iterations, size = n / 2^k. Terminates when size = 1: n / 2^k = 1 → k = log₂(n).

### Binary Search on Answer

| Problem | Feasibility check | Total time |
|---|---|---|
| Koko bananas | O(n) | O(n log(max_val)) |
| Ship packages in D days | O(n) | O(n log(sum)) |
| Split array largest sum | O(n) | O(n log(sum)) |
| Kth smallest in sorted matrix | O(n) | O(n log(max_val)) |
| Median of two sorted arrays | O(log n) | O(log(min(m,n))) |

**General formula:** O(feasibility_cost × log(answer_range))

Answer range analysis:
- Speed/capacity problems: range is usually `[1, max_element]` or `[1, sum]`
- `log(10⁹)` ≈ 30 iterations — even O(n log n) feasibility × 30 = very fast

### When binary search breaks

1. **Data is not sorted and has no monotonic property.** Hash map for O(1), or sort first.
2. **Predicate is not monotonic.** If feasibility doesn't have a clean false→true (or true→false) transition, binary search will find a wrong answer. Verify monotonicity explicitly.
3. **Floating point precision issues.** Using `eps` termination can cause infinite loops. Use fixed iteration count (100) instead.
4. **Integer overflow in `mid = (lo + hi) / 2`.** Classic bug: `lo + hi` overflows when both are near `i32::MAX`. Always use `lo + (hi - lo) / 2`.
5. **Off-by-one in boundary setting.** Mixing open/closed interval conventions mid-solution causes missed elements or infinite loops.

## 5. Edge Cases & Pitfalls

### Pitfall 1: The overflow trap (universal)

```rust
// WRONG — overflows when lo = i32::MAX/2, hi = i32::MAX
let mid = (lo + hi) / 2;

// CORRECT — always use this
let mid = lo + (hi - lo) / 2;

// For usize, even safer since no negatives, but still use subtraction form:
let mid = lo + (hi - lo) / 2;
```

### Pitfall 2: Infinite loop from wrong pointer movement

```rust
// Searching for MINIMUM (hi moves):
if predicate(mid) { hi = mid; }       // OK: hi shrinks
else { lo = mid + 1; }                // OK: lo advances past mid

// Searching for MAXIMUM (lo moves):
if predicate(mid) { lo = mid; }       // DANGER if mid = lo → infinite loop
// FIX: use ceiling mid
let mid = lo + (hi - lo + 1) / 2;    // ceiling, ensures mid > lo when hi > lo
```

**How to never get this wrong:**

When `lo = 5, hi = 6`:
- Floor mid: `5 + (6-5)/2 = 5` → if `lo = mid`, then `lo` doesn't move → **infinite loop**
- Ceiling mid: `5 + (6-5+1)/2 = 6` → if `lo = mid = 6 = hi` → **terminates**

Rule: **If your assignment is `lo = mid`, use ceiling mid. If `hi = mid`, use floor mid.**

### Pitfall 3: Wrong answer range for binary search on answer

```rust
// Koko bananas: minimum speed
// WRONG lower bound: lo = 0 (speed 0 = never finishes)
// CORRECT: lo = 1

// WRONG upper bound: hi = i32::MAX (wastes iterations)
// CORRECT: hi = max_pile (at this speed, each pile takes 1 hour — always possible)

// Ship packages: minimum capacity
// lo = max_weight (must fit heaviest package)
// hi = sum of all weights (ship everything in one day)
// Getting lo wrong causes predicate to return false for valid range edge
```

### Pitfall 4: Predicate that's not truly monotonic

```rust
// "Find peak element" — the array is NOT sorted, no single monotonic predicate
// BUT: there IS a local monotonic property at the boundary
// arr[mid] > arr[mid+1] → peak is in [lo, mid] (includes mid)
// arr[mid] < arr[mid+1] → peak is in [mid+1, hi]
// This works because peak MUST exist (array has finite length, edges treated as -inf)
// Pitfall: trying to binary search for a value in an unsorted array — won't work

fn find_peak(arr: &[i32]) -> usize {
    let (mut lo, mut hi) = (0usize, arr.len() - 1);
    while lo < hi {
        let mid = lo + (hi - lo) / 2;
        if arr[mid] > arr[mid + 1] {
            hi = mid;           // peak is on the left (mid could be it)
        } else {
            lo = mid + 1;       // peak must be on the right
        }
    }
    lo
}
```

### Pitfall 5: Rotated sorted array — mid comparison order

```rust
// Search in rotated sorted array [4,5,6,7,0,1,2]
// You MUST determine which half is sorted before making a decision
fn search_rotated(nums: &[i32], target: i32) -> i32 {
    let (mut lo, mut hi) = (0i32, nums.len() as i32 - 1);
    while lo <= hi {
        let mid = lo + (hi - lo) / 2;
        if nums[mid as usize] == target { return mid; }

        // Determine which half is definitively sorted
        if nums[lo as usize] <= nums[mid as usize] {
            // Left half is sorted
            if nums[lo as usize] <= target && target < nums[mid as usize] {
                hi = mid - 1;   // target in sorted left half
            } else {
                lo = mid + 1;   // target in right half
            }
        } else {
            // Right half is sorted
            if nums[mid as usize] < target && target <= nums[hi as usize] {
                lo = mid + 1;   // target in sorted right half
            } else {
                hi = mid - 1;   // target in left half
            }
        }
    }
    -1
}
// PITFALL: Checking if target is in "current left half" without confirming left half is sorted
// leads to wrong answers. Always identify sorted half FIRST.
```

### Pitfall 6: Integer vs float feasibility comparison

```rust
// When checking feasibility, use integer arithmetic where possible
// Ceiling division for integer problems:
let hours = (pile + speed - 1) / speed;  // ceiling(pile / speed)
// BETTER THAN: (pile as f64 / speed as f64).ceil() as i64 — floating point imprecision
```

### Debugging heuristics

1. **Verify loop terminates:** Does `hi - lo` strictly decrease each iteration? If `hi = mid` path is taken, mid < hi (floor guarantees this). If `lo = mid + 1`, lo > mid. Check both.
2. **Trace on n=1, n=2:** Single element and two-element arrays expose off-by-one bugs immediately.
3. **Verify predicate:** Run `predicate(lo_initial)` (should be false) and `predicate(hi_initial)` (should be true) before the loop. If not, your search bounds are wrong.
4. **Off by one in result:** After loop, answer is at `lo`. If you return `lo - 1` or `lo + 1` and get wrong answers, re-examine what `lo` represents at termination.

## 6. Variations (Pattern Stretching)

### Easy: Search Insert Position

```rust
// Where would target be inserted to keep array sorted?
fn search_insert(nums: &[i32], target: i32) -> usize {
    nums.partition_point(|&x| x < target)
    // = lower bound: first index where nums[i] >= target
    // If target exists, returns its index. If not, returns insertion point.
}
```

### Medium: Find First and Last Position

```rust
fn search_range(nums: &[i32], target: i32) -> [i32; 2] {
    let first = nums.partition_point(|&x| x < target);
    if first == nums.len() || nums[first] != target {
        return [-1, -1];
    }
    let last = nums.partition_point(|&x| x <= target) - 1;
    [first as i32, last as i32]
}
```

### Medium: Search in Rotated Sorted Array

Already shown in pitfalls section. The key insight: one half is always sorted. Identify the sorted half, check if target is in it, binary search there.

### Medium: Kth Smallest in Sorted Matrix

```rust
// Matrix where each row and column is sorted
// Binary search on answer: what value is the kth smallest?
fn kth_smallest(matrix: &Vec<Vec<i32>>, k: i32) -> i32 {
    let n = matrix.len();
    let (mut lo, mut hi) = (matrix[0][0], matrix[n-1][n-1]);

    while lo < hi {
        let mid = lo + (hi - lo) / 2;

        // Count elements <= mid
        let count = count_le(matrix, mid, n);

        if count < k {
            lo = mid + 1;   // need a larger value
        } else {
            hi = mid;       // mid is a candidate for kth smallest
        }
    }
    lo
}

fn count_le(matrix: &Vec<Vec<i32>>, target: i32, n: usize) -> i32 {
    let (mut row, mut col) = (n - 1, 0usize);
    let mut count = 0i32;
    while col < n {
        while row < n && matrix[row][col] > target {
            if row == 0 { break; }
            row -= 1;
        }
        // All elements in column col from row 0 to row are <= target
        if row < n && matrix[row][col] <= target {
            count += row as i32 + 1;
            col += 1;
        } else {
            break;
        }
    }
    count
}
// Time: O(n log(max-min)) — O(n) count × O(log(range)) binary search
```

### Hard: Median of Two Sorted Arrays

```rust
// Binary search on the partition of the smaller array
fn find_median_sorted_arrays(nums1: Vec<i32>, nums2: Vec<i32>) -> f64 {
    // Ensure nums1 is the smaller array
    let (a, b) = if nums1.len() <= nums2.len() {
        (&nums1, &nums2)
    } else {
        (&nums2, &nums1)
    };

    let (m, n) = (a.len(), b.len());
    let half = (m + n + 1) / 2;
    let (mut lo, mut hi) = (0usize, m);

    while lo <= hi {
        let i = lo + (hi - lo) / 2;  // partition in a: a[0..i] in left half
        let j = half - i;            // partition in b: b[0..j] in left half

        let a_left  = if i == 0 { i32::MIN } else { a[i-1] };
        let a_right = if i == m { i32::MAX } else { a[i] };
        let b_left  = if j == 0 { i32::MIN } else { b[j-1] };
        let b_right = if j == n { i32::MAX } else { b[j] };

        if a_left <= b_right && b_left <= a_right {
            // Correct partition found
            let max_left = a_left.max(b_left);
            if (m + n) % 2 == 1 {
                return max_left as f64;
            }
            let min_right = a_right.min(b_right);
            return (max_left + min_right) as f64 / 2.0;
        } else if a_left > b_right {
            hi = i - 1;  // too many elements from a in left half
        } else {
            lo = i + 1;  // too few elements from a in left half
        }
    }
    0.0  // unreachable
}
// Time: O(log(min(m, n))) — binary search on the smaller array's partition
// This is the most elegant O(log n) solution to a conceptually hard problem
```

### Evolution map

```
Search Insert Position (basic lower bound)             → O(log n)
       ↓
First/Last Position (two lower bounds)                 → O(log n)
       ↓
Search in Rotated Array (conditional on sorted half)   → O(log n)
       ↓
Koko Bananas / Ship Packages (binary search on answer) → O(n log max)
       ↓
Kth Smallest in Matrix (count function + BSoA)         → O(n log range)
       ↓
Median of Two Sorted Arrays (partition binary search)  → O(log min(m,n))
       ↓
Minimize Max / Maximize Min (optimization + BSoA)      → O(n log n)
```

## 7. Interview-Level Problems

### Problem 1 (Medium): Find Minimum in Rotated Sorted Array II

**Summary:** Given a rotated sorted array that may contain duplicates, find the minimum element. Example: `[2,2,2,0,1]` → `0`.

**Expected approach:**
Binary search with a special case for duplicates. When `arr[mid] == arr[hi]`, you cannot determine which half to eliminate — both could contain the minimum. In this case, shrink `hi` by one (`hi -= 1`) and try again. When `arr[mid] < arr[hi]`, min is in `[lo, mid]`. When `arr[mid] > arr[hi]`, min is in `[mid+1, hi]`.

**Key insight:** The duplicate case is the crux. Without duplicates, you have a clean binary decision. With duplicates, `arr[mid] == arr[hi]` is ambiguous: `[3,3,1,3]` and `[3,1,3,3]` both have `mid == hi`, but minima are in different halves. The safe move — decrement `hi` by 1 — doesn't miss the minimum because `arr[hi] == arr[mid]`, so `arr[mid]` itself is a backup if `arr[hi]` was the minimum. Worst case degrades to O(n) (all duplicates), average O(log n).

---

### Problem 2 (Medium-Hard): Split Array Largest Sum

**Summary:** Given array `nums` and integer `k`, split `nums` into `k` non-empty contiguous subarrays to minimize the largest subarray sum. Return that minimum largest sum.

**Expected approach:**
Binary search on answer. The answer lies in `[max(nums), sum(nums)]`. For a given target `mid`, check if you can split into ≤ k parts where each part has sum ≤ `mid` (greedy: pack as many elements as possible into each part until adding the next would exceed `mid`). The feasibility check is O(n). Binary search the answer space.

**Key insight:** The answer is monotonic: if you CAN split with max-sum = mid, you can certainly split with max-sum = mid+1. The feasibility function is greedy — greedily maximize each chunk without exceeding `mid`. Counter-intuitively, the greedy DOESN'T minimize parts count in general — but it correctly ANSWERS whether ≤ k parts are achievable with max-sum ≤ mid. This is the jump from "hard DP problem" (O(n²k)) to "binary search on answer" (O(n log(sum))). Recognizing that minimizing the maximum is a binary-search-on-answer trigger is the key pattern to internalize.

---

### Problem 3 (Hard): Find in Mountain Array

**Summary:** Given a "mountain array" (strictly increases then strictly decreases), find the index of a target value. You can only call `mountainArr.get(index)` (API with call limit). The array length is unknown but can be queried. Must use ≤ 100 calls.

**Expected approach:**
Three-phase binary search:
1. Find the **peak index**: binary search where `arr[mid] < arr[mid+1]` → go right, else go left. O(log n) calls.
2. Binary search on the **ascending half** `[0, peak]` for target. O(log n) calls.
3. If not found, binary search on the **descending half** `[peak+1, n-1]` for target (reverse comparator). O(log n) calls.

Total calls: ~3 × log₂(10⁴) ≈ 42 calls — well within the limit.

**Key insight:** A non-monotonic array is not directly binary-searchable. But the mountain has TWO monotonic regions. Identifying the peak boundary enables binary search on each region independently. The problem tests whether you can decompose "binary search on a complex structure" into "binary search on a simple monotonic substructure". Finding the peak is itself a binary search (compare adjacent elements, not element to target).

## 8. Staff-Level Thinking

### Trade-offs: Binary Search vs Hash Map vs Linear Scan

| Criterion | Linear Scan | Binary Search | Hash Map |
|---|---|---|---|
| Time (one query) | O(n) | O(log n) | O(1) avg |
| Space | O(1) | O(1) | O(n) |
| Requires sorted | No | Yes | No |
| Build cost | O(1) | O(n log n) | O(n) |
| Range queries | O(n) | O(log n) | O(n) or not possible |
| Floor/ceiling queries | O(n) | O(log n) | Not possible (HashMap) |
| Predictable worst case | O(n) | O(log n) | O(n) worst (hash collision) |
| Cache performance | Excellent | Good | Poor (hash indirection) |

**The decision matrix:**
- One-time lookup on unsorted data → linear scan (no build cost)
- Repeated lookups, data can be sorted → binary search (sort once, log n per query)
- Repeated exact lookups, O(1) required → hash map (O(n) space trade)
- Range queries (all elements in [lo, hi]) → binary search (hash map can't do this)
- Membership only → hash set (O(1), but no ordered queries)

### Binary Search on Answer — the power pattern

This is the pattern that separates senior from staff in interviews. The recognition that:

```
"Minimize the maximum X such that condition Y holds"
OR
"Maximize the minimum X such that condition Y holds"

→ ALWAYS binary search on answer + greedy feasibility check
```

Examples you'll see:
- Koko Eating Bananas, Capacity to Ship, Minimum Days to Make M Bouquets → all O(n log max)
- Aggressive Cows, Split Array Largest Sum, Allocate Books → O(n log sum)
- Painter's Partition Problem → same pattern

**The framing test:**
1. Can you check "is answer ≤ X feasible?" in O(n) or O(n log n)?
2. Is feasibility monotonic? (If feasible at X, feasible at X+1 for min problems)
3. Can you bound the answer range?

All three YES → binary search on answer. Replace O(n²) or O(n³) DP with O(n log n).

### Scalability considerations

**Why binary search matters in distributed systems:**

At 20K CCU, if you're searching through sorted data (user IDs, timestamps, leaderboard scores):
- Linear scan on 10⁶ sorted records per request = 10⁶ × 20K = 2×10¹⁰ ops/sec — not feasible
- Binary search = 20 × 20K = 4×10⁵ ops/sec — trivial

**Database B-tree index = binary search on disk blocks:**
- B-tree height ≈ log_B(n) where B = branching factor (~100-1000 for disk blocks)
- 1 billion records: tree height ≈ log₁₀₀₀(10⁹) = 3 levels → 3 disk reads
- This is literally binary (multi-way) search on sorted, indexed data

**Binary search on answer in infrastructure:**
- Auto-scaling: "what's the minimum number of pods to keep p99 latency ≤ 100ms?" — can be framed as binary search on answer with load simulation as feasibility check
- Capacity planning: "minimum DB connection pool size to handle 10K RPS with max 50ms wait?" — same pattern

### When NOT to use binary search

1. **Unsorted data with no exploitable structure.** Must sort first (O(n log n)) which may negate the benefit.
2. **When you need ALL occurrences.** Binary search finds one index; finding all occurrences in a large range is still O(n) after finding the boundaries.
3. **When the data changes frequently.** Binary search requires sorted data. Resorting on every update is O(n log n). Use a balanced BST (BTreeMap) for O(log n) insert + O(log n) search on dynamic sorted data.
4. **When feasibility is expensive.** If checking `predicate(mid)` costs O(n²), binary search gives O(n² log n) which may be worse than O(n²) direct search.
5. **Non-monotonic answer space.** Binary search requires a clean false→true threshold. Unimodal functions (answer goes down then up) require ternary search, not binary.

### How to explain in interviews

**Classic:**
> "The array is sorted. I'll binary search for the target in O(log n). The invariant is: the target, if it exists, is always in `[lo, hi)`. I shrink the range by half each step."

**On Answer:**
> "The problem asks for the minimum X that satisfies condition Y. I'll binary search on X in range `[lo, hi]`. For a guessed X, I can check feasibility in O(n) using a greedy pass. The feasibility is monotonic — if X works, X+1 also works — so there's a clean threshold, and binary search finds it in O(log(range)) guesses. Total: O(n log(range))."

**When asked "how do you know it's monotonic":**
> "Because the condition is 'can we achieve the goal with capacity ≤ X'. More capacity only helps — you can always do with more what you can do with less. So if X is feasible, X+1 is trivially feasible."

## 9. Real-World Mapping

| Production scenario | Binary search role |
|---|---|
| **B-tree / database index** | Every B-tree lookup is multi-way binary search. `WHERE id = 12345` on indexed column = O(log n) disk reads instead of O(n) full scan |
| **DynamoDB range queries** | Sort key enables binary search within a partition. `BETWEEN 100 AND 200` on sort key = O(log n + k) not O(n) |
| **git bisect** | Binary search on commit history to find which commit introduced a bug. Predicate: "does this commit have the bug?" Exactly binary search on answer. |
| **DNS TTL optimization** | Finding minimum TTL where cache hit rate meets SLA — binary search on TTL value with simulation as feasibility check |
| **Load balancer threshold** | "Minimum server count where p99 latency ≤ threshold" — binary search on server count, load simulation as feasibility |
| **Karpenter node bin packing** | Finding minimum node size that fits all pending pod requests — variant of search on answer |
| **A/B test power analysis** | "Minimum sample size to detect X% lift at 95% confidence" — binary search on sample size with statistical test as predicate |
| **API rate limiting (token bucket)** | Binary search for the correct refill rate that sustains target throughput without exceeding burst limit |

**System design scenario — leaderboard at 20K CCU:**

Tournament platform needs to answer "what rank is player X?" and "who is in ranks 100-200?" in real time.

Options:
1. Store unsorted, rank on read: O(n) per read — terrible
2. Sorted array + binary search: O(log n) per read, O(log n) per update (insert in sorted position — actually O(n) shift) — read-heavy workload
3. **Redis sorted sets (skip list)**: O(log n) insert, O(log n) rank query, O(log n + k) range query — the production answer

But the interview answer is: "If reads dominate, sort once, binary search for rank. `partition_point(|&score| score < player_score)` gives the rank in O(log n). For writes, maintain sorted order with a BTreeSet (O(log n) insert) and binary search for range queries."

## 10. 10-Second Recall Cheat Sheet

```
TWO MODES:
  Classic     → "find X in sorted structure" → O(log n)
  On Answer   → "find minimum X where predicate is true" → O(log(range) × feasibility_cost)

CLASSIC TEMPLATES:
  Exact match         → binary_search() → Ok(idx) or Err(insertion_point)
  Lower bound         → partition_point(|&x| x < target) → first idx where x >= target
  Upper bound         → partition_point(|&x| x <= target) → first idx where x > target
  Count occurrences   → upper_bound - lower_bound
  Rotated array       → find sorted half first, then check if target is in it

ON ANSWER TEMPLATES:
  Find minimum X:     lo moves up, hi moves down → use FLOOR mid
  Find maximum X:     lo moves up, hi stays     → use CEILING mid (lo + (hi-lo+1)/2)
  Trigger: "minimize the maximum" / "maximize the minimum" → always this pattern

BOUND SETTING FOR ON ANSWER:
  lo = smallest possible valid answer (not 0 unless 0 is valid)
  hi = largest possible valid answer (not INT_MAX — use problem constraints)
  Examples:
    Koko bananas:    [1, max_pile]
    Ship packages:   [max_weight, total_weight]
    Split array:     [max_element, sum]

POINTER MOVEMENT RULES:
  predicate(mid) = true  → hi = mid      (mid might be the answer)
  predicate(mid) = false → lo = mid + 1  (mid definitely not the answer)
  NEVER: lo = mid (use lo = mid+1) when searching for minimum
  NEVER: floor mid when lo = mid path (causes infinite loop)

OVERFLOW RULE (ALWAYS):
  mid = lo + (hi - lo) / 2      — for minimum search (floor)
  mid = lo + (hi - lo + 1) / 2  — for maximum search (ceiling)
  NEVER: (lo + hi) / 2

MONOTONICITY CHECK (before writing code):
  "If X is feasible, is X+1 also feasible?" → YES for minimize problems
  "If X works, does X-1 also work?" → YES for maximize problems
  If neither → not binary search on answer, rethink

COMMON TRAPS:
  Wrong lo/hi bounds → predicate always true or always false at initial bounds
  Floor mid when lo = mid → infinite loop
  Not checking mid == target on rotated array before checking halves
  Float binary search: use 100 iterations, NOT eps-based (infinite loop risk)
  Forgetting: result is lo after loop, not mid

COMPLEXITY:
  Classic:     O(log n) time, O(1) space
  On Answer:   O(feasibility × log(range)) time, O(1) space
  log(10^9) = 30 iterations — always fast

VS ALTERNATIVES:
  Hash map: O(1) lookup but no range queries, no floor/ceiling
  Binary search: O(log n) lookup WITH range queries, floor, ceiling
  Linear scan: O(n) — only when data is unsorted AND single query
```
