## 1. Intuition First (No Jargon)

**The core idea:** Sorting imposes order on chaos so that other algorithms — binary search, two pointers, deduplication, greedy — can exploit that order to run in O(n) or O(log n) instead of O(n²). Sorting is the preprocessing step that unlocks 80% of DSA optimizations.

**Real-world analogy:**

You have 1,000 unsorted customer invoices. Finding a specific one = scan all 1,000. Finding the three largest amounts = scan all 1,000 three times. Checking for duplicates = scan every pair. **Everything is O(n) or worse.**

Sort them once by invoice number — O(n log n) upfront cost — and now every subsequent operation is cheap: find by number = O(log n), get top 3 = O(1), dedup = one pass O(n). The O(n log n) investment pays dividends on every future query.

**WHY multiple sorting algorithms exist:**

There is no universally optimal sort. The right choice depends on:
- **Data distribution**: random, nearly-sorted, reverse-sorted, many duplicates
- **Stability requirement**: do equal elements preserve their original relative order?
- **Memory**: in-place (O(1) extra) vs external (O(n) extra)
- **Data type**: integers in bounded range → linear time possible; arbitrary comparisons → O(n log n) is provably optimal

**The information-theoretic lower bound for comparison sorts:**

To sort n distinct elements, you need to determine which of n! possible orderings is correct. Each comparison gives 1 bit of information. To distinguish n! orderings: at minimum ⌈log₂(n!)⌉ comparisons. By Stirling's approximation: log₂(n!) ≈ n log₂(n). Therefore **O(n log n) is a provable lower bound for any comparison-based sort**. You cannot do better with comparisons alone.

Non-comparison sorts (Counting, Radix, Bucket) break this bound by exploiting the numeric structure of keys — they don't use comparisons between elements.

## 2. Pattern Recognition

### When to use sorting as a DSA tool

Sorting is rarely the end goal in interviews — it's the **enabling step**. Recognize it when you think:

| Problem structure | Sorting unlocks |
|---|---|
| "Find pair/triplet with sum X" | Two pointers on sorted array |
| "Interval problems: merge, overlap, scheduling" | Sort by start time |
| "Anagram detection" | Sort characters of each string |
| "Find k-th largest/smallest" | Sort + index, or QuickSelect |
| "Remove duplicates" | Sort first, adjacent scan |
| "Greedy scheduling (jobs, tasks)" | Sort by deadline/profit/duration |
| "Minimum number of operations to make array non-decreasing" | Sort reveals structure |
| "Custom ordering (build largest number, etc.)" | Custom comparator sort |

### Input/output signals

- Array with no inherent order + pair/group questions → **sort first**
- "Largest / smallest" combined with "k-th" → **sort or partial sort (heap)**
- Intervals or ranges → **sort by start**
- Strings that should be treated as sets of characters → **sort characters**
- "Without extra space" + any grouping → **sort in-place, then scan**

## 3. Core Algorithm / Template

### Algorithm 1: QuickSort

**Idea:** Pick a pivot. Partition so all elements ≤ pivot are left, all ≥ pivot are right. Recurse on both halves. In-place, cache-friendly, fast in practice.

```csharp
static void Quicksort(int[] arr, int lo, int hi)
{
    if (hi - lo <= 0) return;

    int pivotIdx = Partition(arr, lo, hi);
    Quicksort(arr, lo, pivotIdx - 1);
    Quicksort(arr, pivotIdx + 1, hi);
}

static int Partition(int[] arr, int lo, int hi)
{
    int pivot = arr[(lo + hi) / 2]; // middle pivot (better than last for nearly-sorted)
    int left = lo, right = hi;

    while (true)
    {
        while (arr[left] < pivot) left++;
        while (arr[right] > pivot) right--;
        if (left >= right) return right;
        (arr[left], arr[right]) = (arr[right], arr[left]);
        left++;
        right--;
    }
}
// Time: O(n log n) avg, O(n²) worst (already sorted + bad pivot)
// Space: O(log n) avg stack, O(n) worst
// Stable: NO
// In-place: YES
```

**Randomized QuickSort** — prevents O(n²) worst case:

```csharp
static readonly Random _random = new Random();

static void QuicksortRandomized(int[] arr, int lo, int hi)
{
    if (hi - lo <= 0) return;

    // Swap random element to end to use as pivot
    int pivotPos = lo + _random.Next(hi - lo + 1);
    (arr[pivotPos], arr[hi]) = (arr[hi], arr[pivotPos]);

    // Lomuto partition: pivot at end
    int store = lo;
    for (int i = lo; i < hi; i++)
    {
        if (arr[i] <= arr[hi])
        {
            (arr[i], arr[store]) = (arr[store], arr[i]);
            store++;
        }
    }
    (arr[store], arr[hi]) = (arr[hi], arr[store]);

    QuicksortRandomized(arr, lo, store - 1);
    QuicksortRandomized(arr, store + 1, hi);
}
// Note: In interviews, just say "randomize pivot to prevent O(n²) worst case"
// C#'s Array.Sort() uses introsort — handles this automatically
```

### Algorithm 2: MergeSort

**Idea:** Divide in half. Recursively sort each half. Merge two sorted halves. Stable, guaranteed O(n log n), but requires O(n) extra space.

```csharp
static void MergeSort(int[] arr, int lo, int hi)
{
    if (hi - lo <= 0) return;

    int mid = lo + (hi - lo) / 2;
    MergeSort(arr, lo, mid);
    MergeSort(arr, mid + 1, hi);
    Merge(arr, lo, mid, hi);
}

static void Merge(int[] arr, int lo, int mid, int hi)
{
    int[] left = arr[lo..(mid + 1)];
    int[] right = arr[(mid + 1)..(hi + 1)];

    int i = 0, j = 0, k = lo;
    while (i < left.Length && j < right.Length)
    {
        if (left[i] <= right[j]) // <= for stability
        {
            arr[k] = left[i]; i++;
        }
        else
        {
            arr[k] = right[j]; j++;
        }
        k++;
    }
    while (i < left.Length) { arr[k] = left[i]; i++; k++; }
    while (j < right.Length) { arr[k] = right[j]; j++; k++; }
}
// Time: O(n log n) ALWAYS — best/avg/worst
// Space: O(n) auxiliary
// Stable: YES
// In-place: NO (without O(n log n) complexity tricks)
```

**Why MergeSort over QuickSort?**
- Guaranteed O(n log n) — no adversarial worst case
- Stable — equal elements preserve order (critical for multi-key sorts)
- Parallelizable — the two halves can be sorted independently
- Works on linked lists naturally (no random access needed)
- External sort foundation (data too large for RAM: merge sorted chunks from disk)

### Algorithm 3: HeapSort

**Idea:** Build a max-heap. Repeatedly extract max (swap root with last element, heapify). In-place, O(n log n) guaranteed, but poor cache performance.

```csharp
static void HeapSort(int[] arr)
{
    int n = arr.Length;

    // Build max-heap (heapify from bottom up)
    for (int i = n / 2 - 1; i >= 0; i--)
    {
        SiftDown(arr, i, n);
    }

    // Extract elements in sorted order
    for (int end = n - 1; end >= 1; end--)
    {
        (arr[0], arr[end]) = (arr[end], arr[0]); // move max to end
        SiftDown(arr, 0, end);                   // restore heap property for [0..end)
    }
}

static void SiftDown(int[] arr, int root, int end)
{
    while (true)
    {
        int left = 2 * root + 1;
        int right = 2 * root + 2;
        int largest = root;

        if (left < end && arr[left] > arr[largest]) largest = left;
        if (right < end && arr[right] > arr[largest]) largest = right;

        if (largest == root) break;
        (arr[root], arr[largest]) = (arr[largest], arr[root]);
        root = largest;
    }
}
// Time: O(n log n) ALWAYS
// Space: O(1) — in-place
// Stable: NO
// Cache performance: POOR (non-sequential memory access in SiftDown)
```

### Algorithm 4: Counting Sort

**Idea:** Count occurrences of each value. Reconstruct sorted array from counts. Beats O(n log n) because it doesn't compare elements.

```csharp
static int[] CountingSort(int[] arr, int maxVal)
{
    var count = new int[maxVal + 1];

    foreach (var x in arr)
    {
        count[x]++;
    }

    var result = new List<int>(arr.Length);
    for (int val = 0; val <= maxVal; val++)
    {
        for (int c = 0; c < count[val]; c++)
        {
            result.Add(val);
        }
    }
    return result.ToArray();
}

// Stable counting sort (for use in Radix sort)
static int[] CountingSortStable(int[] arr, int maxVal)
{
    var count = new int[maxVal + 1];
    foreach (var x in arr) count[x]++;

    // Prefix sum: count[i] = number of elements <= i
    for (int i = 1; i <= maxVal; i++) count[i] += count[i - 1];

    var result = new int[arr.Length];
    // Traverse in REVERSE for stability
    for (int i = arr.Length - 1; i >= 0; i--)
    {
        count[arr[i]]--;
        result[count[arr[i]]] = arr[i];
    }
    return result;
}
// Time: O(n + k) where k = range of values
// Space: O(k) for count array
// Stable: YES (stable version)
// Only works for: integers (or things mappable to integers) in bounded range
```

### Algorithm 5: Radix Sort

**Idea:** Sort by digits from least significant to most significant, using stable sort at each digit. Each pass is O(n + base). With base 10 and d digits: O(d × n).

```csharp
static void RadixSort(int[] arr)
{
    if (arr.Length == 0) return;
    int maxVal = arr.Max();

    for (int exp = 1; maxVal / exp > 0; exp *= 10)
    {
        CountingSortByDigit(arr, exp);
    }
}

static void CountingSortByDigit(int[] arr, int exp)
{
    int n = arr.Length;
    var output = new int[n];
    var count = new int[10];

    foreach (var x in arr)
    {
        count[(x / exp) % 10]++;
    }
    for (int i = 1; i < 10; i++)
    {
        count[i] += count[i - 1];
    }
    for (int i = n - 1; i >= 0; i--)
    {
        int digit = (arr[i] / exp) % 10;
        count[digit]--;
        output[count[digit]] = arr[i];
    }
    Array.Copy(output, arr, output.Length);
}
// Time: O(d × (n + base)) where d = number of digits
// For 32-bit integers, d = 10 (base 10) or 32 (base 2) → O(n) for fixed-width ints
// Space: O(n + base)
// Stable: YES
```

### C# Built-in Sorts — What to Use

```csharp
// In interviews, ALWAYS use these:

// Stable sort (TimSort — hybrid merge + insertion)
Array.Sort(v);                                       // O(n log n), O(n) space
Array.Sort(v, (a, b) => a.CompareTo(b));            // custom, stable via LINQ OrderBy
v = v.OrderBy(x => x).ToArray();                    // stable LINQ sort
v = v.OrderBy(x => Math.Abs(x)).ToArray();          // by key, stable

// Unstable sort (introsort — introspective quicksort)
Array.Sort(v);                                       // O(n log n), O(log n) space
Array.Sort(v, (a, b) => b.CompareTo(a));            // descending
v = v.OrderByDescending(x => x).ToArray();          // descending via LINQ

// Partial sort: find kth element (QuickSelect, O(n) avg) — no built-in, implement manually
// After QuickSelect: v[0..k] <= v[k] <= v[k+1..], v[k] is correct

// Sort tuples / multi-key
Array.Sort(items, (a, b) =>
{
    int cmp = a.Start.CompareTo(b.Start);
    return cmp != 0 ? cmp : a.End.CompareTo(b.End);  // asc both
});
// start asc, end desc:
Array.Sort(items, (a, b) =>
{
    int cmp = a.Start.CompareTo(b.Start);
    return cmp != 0 ? cmp : b.End.CompareTo(a.End);
});

// Floats — double implements IComparable, sort() works directly
Array.Sort(v);                                       // works for double[]
Array.Sort(v, (a, b) => a.CompareTo(b));            // explicit, handles NaN via CompareTo

// Sort indices (not the values themselves)
var indices = Enumerable.Range(0, n).ToArray();
Array.Sort(indices, (i, j) => arr[i].CompareTo(arr[j]));
```

**C# Array.Sort vs LINQ OrderBy:**
- `Array.Sort()` (introsort): unstable, O(log n) stack space — default choice when stability doesn't matter
- `OrderBy()` (TimSort via LINQ): stable, O(n) space — use when order of equal elements matters

## 4. Complexity Deep Dive

### Algorithm comparison matrix

| Algorithm | Best | Average | Worst | Space | Stable | Notes |
|---|---|---|---|---|---|---|
| QuickSort | O(n log n) | O(n log n) | O(n²) | O(log n) | No | Fast in practice, cache-friendly |
| MergeSort | O(n log n) | O(n log n) | O(n log n) | O(n) | Yes | Guaranteed, parallelizable |
| HeapSort | O(n log n) | O(n log n) | O(n log n) | O(1) | No | Poor cache, rare in practice |
| Insertion Sort | O(n) | O(n²) | O(n²) | O(1) | Yes | Best for small n (< 16) or nearly sorted |
| Bubble Sort | O(n) | O(n²) | O(n²) | O(1) | Yes | Almost never use |
| Counting Sort | O(n+k) | O(n+k) | O(n+k) | O(k) | Yes | Only integers, bounded range |
| Radix Sort | O(dn) | O(dn) | O(dn) | O(n+b) | Yes | Fixed-width integers → O(n) |
| Bucket Sort | O(n+k) | O(n+k) | O(n²) | O(n) | Yes | Uniform distribution assumption |
| TimSort | O(n) | O(n log n) | O(n log n) | O(n) | Yes | Python/Java default, excellent on real data |
| introsort | O(n) | O(n log n) | O(n log n) | O(log n) | No | C#'s `Array.Sort()`, pattern-defeating |

### The O(n log n) barrier in practice

```
n = 10⁶ elements:
  O(n log n) ≈ 20 × 10⁶ operations ≈ 0.02 seconds
  O(n²)      ≈ 10¹² operations ≈ thousands of seconds

n = 10⁵ elements:
  O(n log n) ≈ 1.7 × 10⁶ → fast
  O(n²)      ≈ 10¹⁰ → TLE in competitive programming
```

### When O(n log n) is the bottleneck vs when it's fine

| n constraint | Max affordable | Approach |
|---|---|---|
| n ≤ 10⁶ | O(n log n) | Standard sort fine |
| n ≤ 10⁷ | O(n) | Must use linear sort or avoid sorting |
| n ≤ 10⁸ | O(n) tight | Radix/Counting sort, careful constants |
| Values ≤ 10⁶ (bounded integers) | O(n + k) | Counting sort |

### Trade-offs in real production

**QuickSort wins:** Random data, in-memory, cache performance critical, stability not needed.

**MergeSort wins:** Need guaranteed O(n log n) (no adversarial input), stability required, external sort (too big for RAM), linked lists.

**Counting/Radix wins:** Integers in bounded range (event timestamps, user IDs, ages), sort is the bottleneck.

**HeapSort wins:** Embedded systems with no extra memory, streaming sort where you need the "next minimum" online.

## 5. Edge Cases & Pitfalls

### Pitfall 1: Comparator contract violation

```csharp
// WRONG — overflow trap
Array.Sort(v, (a, b) => a - b);          // WRONG: overflow if a = int.MaxValue, b = -1

// CORRECT
Array.Sort(v, (a, b) => a.CompareTo(b)); // always safe
Array.Sort(v, (a, b) => a.CompareTo(b)); // for doubles, CompareTo handles edge cases
```

A valid comparator must satisfy:
1. **Antisymmetry:** if `cmp(a, b) = Less` then `cmp(b, a) = Greater`
2. **Transitivity:** if `a ≤ b` and `b ≤ c` then `a ≤ c`
3. **Totality:** every pair has a defined ordering

**Violating any of these causes undefined behavior in sort algorithms** — the sort may not terminate, may throw, or produce wrong results. C# may throw an `InvalidOperationException` if the comparator is inconsistent.

### Pitfall 2: Stability when it actually matters

```csharp
// Sort by first element — two elements with same first element:
// [(1, 'b'), (1, 'a'), (2, 'c')]
// OrderBy(x => x.Item1) → stable = [(1,'b'),(1,'a'),(2,'c')]
// Array.Sort with comparator = either order for the (1,_) elements (unstable)

// THIS MATTERS for: multi-key sort, sort then group-by, ranking with tiebreakers
// Always use OrderBy() not Array.Sort() when downstream logic depends on equal-element order
```

### Pitfall 3: Sorting strings numerically

```csharp
// Problem: sort ["10", "9", "1", "100"]
// Lexicographic: ["1", "10", "100", "9"] ← WRONG for numeric intent
// Numeric:       ["1", "9", "10", "100"] ← correct

var v = new[] { "10", "9", "1", "100" };
Array.Sort(v);                                              // ["1", "10", "100", "9"] — lexicographic!
Array.Sort(v, (a, b) => int.Parse(a).CompareTo(int.Parse(b))); // ["1", "9", "10", "100"] — numeric
```

### Pitfall 4: Custom comparator for "Largest Number" problem

```csharp
// Sort numbers so their concatenation forms the largest number
// e.g., [3, 30, 34, 5, 9] → "9534330"
// Comparator: "a before b" if ab > ba (as concatenated strings)

var numsStr = nums.Select(x => x.ToString()).ToArray();
Array.Sort(numsStr, (a, b) =>
{
    string ab = $"{a}{b}";
    string ba = $"{b}{a}";
    return ba.CompareTo(ab); // descending: larger concatenation first
});
// PITFALL: This comparator IS transitive for natural number strings
// (can be proven), but looks non-obvious. Must verify transitivity!
```

### Pitfall 5: Partial sort when full sort is unnecessary

```csharp
// Finding the kth largest — DON'T sort the whole array:
Array.Sort(v);                          // O(n log n) — wasteful
var kth = v[n - k];

// DO use QuickSelect:
// QuickSelect(v, k - 1) partitions around kth largest element — O(n) avg
// v[k-1] is now the kth largest after QuickSelect
```

### Pitfall 6: Counting sort with negative numbers

```csharp
// Counting sort assumes values >= 0
// For negative values, shift by the minimum:
int minVal = arr.Min();
int maxVal = arr.Max();
int range = maxVal - minVal + 1;
var count = new int[range];
foreach (var x in arr)
{
    count[x - minVal]++;
}
// Reconstruct with shift back:
int idx = 0;
for (int i = 0; i < range; i++)
{
    for (int c = 0; c < count[i]; c++)
    {
        arr[idx++] = i + minVal;
    }
}
```

### Debugging heuristics

1. **Comparator wrong?** Run `Debug.Assert(Enumerable.Range(0, v.Length - 1).All(i => v[i] <= v[i + 1]))` after sorting.
2. **Stability issue?** Use a tuple `(key, originalIndex)` and verify secondary sort.
3. **Wrong type of sort?** For doubles, `Array.Sort()` works directly — `double` implements `IComparable`.
4. **Off-by-one in custom partition?** Manually trace on `[5, 3, 1, 4, 2]` — verify both halves are correct relative to pivot.

## 6. Variations (Pattern Stretching)

### Easy: Sort as a prerequisite

```csharp
// Pattern: "does a sorted version of A equal sorted version of B?"
// Anagram check
static bool IsAnagram(string s, string t)
{
    if (s.Length != t.Length) return false;
    var sc = s.ToCharArray();
    var tc = t.ToCharArray();
    Array.Sort(sc);
    Array.Sort(tc);
    return sc.SequenceEqual(tc);
}
// Alternative (O(n) with counting): frequency array of 26 chars
// Sort approach is simpler code; frequency is faster
```

### Medium: Sort with custom comparator

```csharp
// Largest Number: [3, 30, 34, 5, 9] → "9534330"
static string LargestNumber(int[] nums)
{
    var strs = nums.Select(x => x.ToString()).ToArray();
    Array.Sort(strs, (a, b) =>
    {
        string ab = $"{a}{b}";
        string ba = $"{b}{a}";
        return ba.CompareTo(ab);
    });
    if (strs[0] == "0") return "0"; // all zeros edge case
    return string.Concat(strs);
}
```

### Medium: Interval merge (sort + scan)

```csharp
// Merge overlapping intervals
static int[][] MergeIntervals(int[][] intervals)
{
    if (intervals.Length == 0) return Array.Empty<int[]>();
    Array.Sort(intervals, (a, b) => a[0].CompareTo(b[0])); // sort by start

    var result = new List<int[]> { intervals[0] };

    foreach (var curr in intervals[1..])
    {
        var last = result[^1];
        if (curr[0] <= last[1])                   // overlapping
        {
            last[1] = Math.Max(last[1], curr[1]); // extend end
        }
        else
        {
            result.Add(curr);                     // non-overlapping, add new
        }
    }
    return result.ToArray();
}
// Key insight: sorting by start guarantees if curr.start > last.end,
// ALL future intervals also start after last.end (no need to look back)
```

### Medium-Hard: Meeting Rooms II (greedy with sort)

```csharp
// Minimum number of conference rooms required for all meetings
// Sort by start time; use min-heap of end times
static int MinMeetingRooms(int[][] intervals)
{
    Array.Sort(intervals, (a, b) => a[0].CompareTo(b[0]));
    var heap = new PriorityQueue<int, int>(); // min-heap of end times

    foreach (var interval in intervals)
    {
        if (heap.Count > 0 && heap.Peek() <= interval[0])
        {
            heap.Dequeue(); // reuse this room
        }
        heap.Enqueue(interval[1], interval[1]);
    }
    return heap.Count;
}
// Key: sort by start → process meetings in order → heap tracks
// room availability by earliest end time
```

### Hard: Count of Range Sum (sort + merge sort with side effect)

**The merge sort pattern that counts while sorting:**

In problems like "count inversions" or "count range sums", merge sort can count cross-half pairs during the merge step — turning an O(n²) count into O(n log n).

```csharp
// Count inversions: pairs (i,j) where i < j and arr[i] > arr[j]
static long CountInversions(int[] arr, int lo, int hi)
{
    if (hi - lo <= 0) return 0;

    int mid = lo + (hi - lo) / 2;
    int[] left = arr[lo..(mid + 1)];
    int[] right = arr[(mid + 1)..(hi + 1)];

    long count = CountInversions(left, 0, left.Length - 1)
               + CountInversions(right, 0, right.Length - 1);

    // Copy sorted halves back
    Array.Copy(left, 0, arr, lo, left.Length);
    Array.Copy(right, 0, arr, mid + 1, right.Length);

    // Merge and count cross-inversions
    int i = lo, j = mid + 1, k = lo;
    var merged = new int[hi - lo + 1];
    int mi = 0;
    int li = 0, ri = 0;
    while (li < left.Length && ri < right.Length)
    {
        if (left[li] <= right[ri])
        {
            merged[mi++] = left[li++];
        }
        else
        {
            // ALL remaining elements in left[li..] are > right[ri]
            // (left is sorted) → count them all
            count += left.Length - li;
            merged[mi++] = right[ri++];
        }
    }
    while (li < left.Length) merged[mi++] = left[li++];
    while (ri < right.Length) merged[mi++] = right[ri++];
    Array.Copy(merged, 0, arr, lo, merged.Length);

    return count;
}
// Key insight: during merge of two sorted halves, when we take from right,
// ALL remaining left elements form inversions with it — count in O(1) per such choice.
```

### Evolution map

```
Anagram detection (sort chars, compare)           — sort as equality check
       ↓
Merge intervals (sort by start, scan)             — sort enables greedy scan
       ↓
Meeting rooms (sort + heap)                       — sort + data structure
       ↓
Count inversions (merge sort side-effect)         — sort AS the algorithm
       ↓
Count of range sum (merge sort + binary search)   — sort + search embedded
```

## 7. Interview-Level Problems

### Problem 1 (Medium): Sort Characters By Frequency

**Summary:** Given a string `s`, sort it such that characters appear in decreasing order of frequency. `"tree"` → `"eert"` or `"eetr"`.

**Expected approach:**
Build a frequency map (O(n)). Sort the unique characters by their frequency in descending order (O(k log k) where k = distinct chars ≤ 26 for lowercase). Reconstruct by repeating each char `freq` times. Total: O(n) dominated by input scan.

**Key insight:** This is NOT a traditional sort of the input array — it's a sort of the *metadata* (characters by their frequency). Recognizing that you sort the key space, not the input, separates this from naive O(n log n) approaches. For fixed alphabet (a-z), k ≤ 26, so sorting is effectively O(1) — the whole algorithm is O(n).

---

### Problem 2 (Medium-Hard): Wiggle Sort II

**Summary:** Given an unsorted array `nums`, reorder it in-place such that `nums[0] < nums[1] > nums[2] < nums[3]...`. No equal adjacent elements allowed (unlike Wiggle Sort I).

**Expected approach:**
Sort the array (O(n log n)). Split into two halves: smaller half and larger half. Interleave them: place larger half elements at odd indices, smaller half at even indices — both in reverse order (reverse fills prevent equal adjacent elements when duplicates exist). The reverse fill is the non-obvious insight.

**Key insight:** The naive approach (just swap adjacent out-of-order elements) fails for inputs with many duplicates. After sorting, reverse-filling both halves into their respective positions separates equal elements maximally. Specifically: the median element appears only once in the result (it's the split point), so equal elements can't end up adjacent. This is the "virtual indexing" trick — O(n log n) sort + O(n) fill.

### Problem 3 (Hard): Count of Smaller Numbers After Self

**Summary:** Given an array `nums`, return counts array where `counts[i]` = number of elements to the right of `nums[i]` that are smaller than `nums[i]`.

**Expected approach:**
Modified merge sort that counts right-side inversions during merge. Process from right to left; during the merge of two sorted halves, when we take an element from the right half, all remaining elements in the left half that are larger than it get their count incremented by 1.

Alternative: Coordinate compression + Binary Indexed Tree (Fenwick Tree — Tier 6). Process right to left, query BIT for count of elements smaller than current, then update BIT.

**Key insight:** This is the "augmented merge sort" pattern. The merge step's O(n) cross-comparison becomes O(n log n) total because we count *all* left elements ≥ right[j] in O(1) per merge decision (since left is sorted, if `left[i] > right[j]` then all `left[i..] > right[j]`). The count is `left.Length - i`, not computed in a loop. This transforms O(n²) brute force into O(n log n) using sort structure itself.

## 8. Staff-Level Thinking

### Trade-offs matrix for algorithm selection

| Situation | Best choice | Why |
|---|---|---|
| General purpose, stability needed | TimSort (`OrderBy()`) | Adaptive to real data, stable |
| General purpose, speed priority | introsort (`Array.Sort()`) | ~20% faster, handles adversarial inputs |
| Nearly sorted input | Insertion sort or TimSort | O(n) for nearly-sorted runs |
| Integers, bounded range k | Counting sort | O(n + k) beats O(n log n) when k = O(n) |
| Fixed-width integers (IDs, timestamps) | Radix sort | O(n) provably, very cache-friendly |
| External sort (TB of data) | Merge sort | Works without random access, parallelizes |
| Linked list | Merge sort | No random access needed |
| Online sort (next-minimum online) | Heap | O(log n) insert + O(1) peek |
| Kth element only | QuickSelect | O(n) average, no need to sort all |
| Uniform floats in [0, 1) | Bucket sort | O(n) expected |

### Scalability considerations

**Counting sort at scale:**
- Event timestamps (Unix epoch): range ≈ 10¹⁰ → too large for standard counting sort
- But relative timestamps (e.g., last 24 hours in seconds) = range 86,400 → trivial
- In production: for sorting user IDs (0 to 10⁸), counting sort uses 400MB just for the count array. Not worth it. Use radix sort (2 passes base 10⁴ → count array size 10,000 each).

**Parallel merge sort:**
- Each subproblem is independent → embarrassingly parallel
- Database query engines sort in parallel across CPU cores and disk chunks
- The merge step is the bottleneck for parallelism (can be parallelized with p-way merge)

**Sorting in Kafka/stream context:**
- You can't sort an infinite stream
- Time-windowed sort: collect all events in a time window, sort within window
- This is why Kafka Streams uses tumbling windows for ordered processing
- True streaming: use a min-heap (priority queue) as a "sort buffer" with a K-second delay

### When NOT to use sorting

1. **When you need O(n) time and the sort is the bottleneck.** Use hash map, counting sort, or a streaming approach instead.
2. **When original indices matter.** Sort loses positional information. Either sort index arrays or use a hash map.
3. **When data is streamed (unbounded).** Can't sort what you haven't seen. Use heaps for "top K" or sliding window for range queries.
4. **When the comparator is expensive.** String comparison is O(k). Sorting n strings of length k = O(nk log n). For long strings, radix sort or suffix arrays may be better.
5. **When stability causes a different algorithm.** If you need stable sort but also need O(1) space, your options are limited (in-place merge sort is O(n log² n), not O(n log n)). Understand the trade-off.

### How to explain in interviews

**When asked "what's the time complexity":**
> "Sorting dominates at O(n log n), and all subsequent operations are O(n) or O(log n), so the overall is O(n log n)."

**When asked "can you do better":**
> "The comparison-based lower bound is O(n log n). If the problem has additional structure — bounded integers, specific distributions — we can break this with counting sort or radix sort for O(n). But for arbitrary comparisons, O(n log n) is optimal."

**When asked "why not just sort and binary search":**
> "Sort + binary search is O(n log n) preprocessing + O(log n) per query = good when queries are many. But if queries are one-time and the array is already unsorted, a hash set gives O(n) build + O(1) per query with O(n) space — depends on the access pattern."

**Stability in interviews — when to volunteer it:**
> "I'm using stable sort here because after sorting by department, I want employees within each department to remain in their original hire-date order. An unstable sort would shuffle them."

## 9. Real-World Mapping

| Production scenario | Sorting role |
|---|---|
| **Database ORDER BY** | External merge sort on disk when data exceeds RAM; indexed scan (skip sort entirely) when index exists |
| **Kafka offset compaction** | Internal Kafka log segments are sorted by offset; merge sort during compaction merges sorted segments |
| **RDS query planner** | Sort-merge join: sort both tables by join key, then merge-scan. Chosen over hash join when data is already indexed/sorted or memory is constrained |
| **DynamoDB GSI** | Global Secondary Index keeps data sorted by alternate key — enabling O(log n) range queries instead of O(n) scans. "Sort key" is literally the MergeSort invariant maintained at write time |
| **ElasticSearch BM25 ranking** | Documents are sorted by relevance score after scoring phase. TimSort handles nearly-sorted inputs efficiently (documents with high scores cluster) |
| **EKS event sorting** | kubectl get events --sort-by=.lastTimestamp — sorting pod events by timestamp for incident triage. Counting sort if you bucketize to seconds |
| **Grafana time-series** | Metric data points arrive out of order; sorted before rendering. Real-time ingestion: insertion sort on nearly-sorted new data (O(n) for appends) |
| **Git merge/diff** | Comparing sorted lists of files. Same-direction two-pointer on sorted arrays computes added/removed/unchanged in O(n + m) |

**System design: the sort-once, query-many pattern**

In the tournament platform: player rankings are updated after each match. Options:
1. Sort all players on each ranking request: O(n log n) per request × R requests/sec = O(Rn log n) total
2. Maintain a sorted structure (BTreeSet/SortedSet): O(log n) per update, O(1) for top-k queries
3. Sort once offline, serve from cache: O(n log n) every T seconds, O(1) per request

At 20K players and 1K RPS: option 1 = 20K × 14 × 1K = 280M ops/sec — too slow. Option 2 or 3 is correct. The sorting algorithm itself changes based on update frequency vs query frequency. This is the architectural framing of sorting that separates staff from senior.

## 10. 10-Second Recall Cheat Sheet

```
ALGORITHM SELECTION:
  General use (stable needed)    → OrderBy()      TimSort, O(n log n), O(n) space
  General use (speed priority)   → Array.Sort()   introsort, O(n log n), O(log n) space
  Integers, small range k        → Counting Sort  O(n + k), O(k) space
  Fixed-width integers           → Radix Sort     O(dn), O(n) space, d = digits
  Kth element only               → QuickSelect    O(n) avg, no full sort needed
  Linked list                    → Merge Sort     works without random access
  External / parallel            → Merge Sort     divide & parallelizable

SORTING AS PREREQUISITE (most common interview use):
  Pair sum / 3Sum / kSum         → sort first → two pointers O(n) / O(n²)
  Merge overlapping intervals    → sort by start → single scan O(n)
  Meeting rooms / scheduling     → sort by start → greedy with heap
  Anagram grouping               → sort each string → group by sorted key
  Duplicate detection            → sort → adjacent comparison O(n)
  Greedy problems (almost all)   → sort to determine optimal processing order

STABILITY — when it matters:
  Multi-key sort (stable primary, then secondary) → use OrderBy() not Array.Sort()
  Sort then group                → elements within groups must stay ordered
  Ranking with tiebreakers       → secondary criterion = original order → must be stable

COMPLEXITY CHEAT:
  O(n log n) is the wall for comparison sorts
  Break the wall: counting (bounded ints), radix (fixed-width), bucket (uniform float)
  Partial sort (kth): O(n) with QuickSelect, not O(n log n)

CUSTOM COMPARATOR RULES:
  Never use (a - b) — overflow risk
  Always use a.CompareTo(b) or Comparer<T>.Default.Compare(a, b)
  Custom comparator MUST be: antisymmetric + transitive + total
  Violating any → undefined sort behavior (InvalidOperationException or wrong result)

TRAPS:
  Array.Sort() is unstable — use OrderBy() when stability matters
  Lexicographic vs numeric string sort → ["10","9"] sorts wrong lexicographically
  Counting sort with negatives → shift by min value
  "Largest number" comparator → ab vs ba string comparison, verify transitivity
  QuickSelect modifies array in place — elements outside kth are unordered

MERGE SORT FOR COUNTING:
  Count inversions / count range sums → augment merge step
  During merge: if left[i] > right[j], count += left.Length - i (all remaining left)
  Transforms O(n²) counting into O(n log n) using sorted structure
```
